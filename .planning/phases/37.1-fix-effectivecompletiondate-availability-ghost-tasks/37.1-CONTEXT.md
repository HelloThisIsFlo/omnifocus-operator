# Phase 37.1: Fix effectiveCompletionDate availability ghost tasks - Context

**Gathered:** 2026-04-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix tasks with `effectiveCompletionDate` set (inherited from completed parent) but no own `dateCompleted` from appearing as `availability: "available"`. These "ghost tasks" pollute default queries and break hierarchy references. The fix uses effective (inherited) dates as authoritative for availability mapping — both in Python mappers and SQL filter clauses.

</domain>

<decisions>
## Implementation Decisions

### Availability mapping — effective dates are authoritative
- **D-01:** `_map_task_availability()` must check `effectiveDateCompleted` instead of `dateCompleted` for the "completed" check. If `effectiveDateCompleted` is set, availability = "completed" — regardless of whether the task's own `dateCompleted` is set.
- **D-02:** Same treatment for drops: check `effectiveDateHidden` (effective drop date) instead of `dateHidden` for the "dropped" check. Symmetric with completion, consistent with Phase 28's "all or none" principle for effective fields.
- **D-03:** The priority chain in `_map_task_availability` stays the same (dropped > completed > blocked > available), only the columns checked change from own to effective.

### SQL filter clauses — same effective columns
- **D-04:** `_TASK_AVAILABILITY_CLAUSES` in `query_builder.py` must use `effectiveDateCompleted` and `effectiveDateHidden` instead of `dateCompleted` and `dateHidden`. This ensures `list_tasks` and `count_tasks` filters are consistent with the Python mapper used by `get_all`/`get_task`.
- **D-05:** `_PROJECT_AVAILABILITY_CLAUSES` should be reviewed for the same pattern — projects join on `ProjectInfo` and may have different effective field semantics.

### Alignment with v1.3.2 direction
- **D-06:** v1.3.2 spec explicitly states "all date filters use effective (inherited) values" and plans to trim availability to `available`/`blocked` only, with completed/dropped expressed via date filters. This fix pre-aligns the availability mapping with that direction.
