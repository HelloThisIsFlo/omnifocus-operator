<decisions>
## Implementation Decisions

### Diff computation
- New helper method `_compute_tag_diff(tag_actions, current_tags)` on OperatorService
- Returns `(add_ids, remove_ids, warnings)` -- single responsibility
- Computes `final_set` from TagActionSpec input:
  - `replace: [A, B]` â†’ final = {A, B}
  - `add: [X]` â†’ final = current âˆª {X}
  - `remove: [B]` â†’ final = current - {B}
  - `add: [X] + remove: [B]` â†’ final = (current âˆª {X}) - {B}
  - `replace: null` or `replace: []` â†’ final = {} (clear all)
- Diff: `to_remove = current - final`, `to_add = final - current`
- Warnings generated during diff computation (single pass, not a separate step)

### No-op detection
- Tag diff feeds into unified no-op detection: if tag diff empty AND field changes empty AND no move â†’ skip bridge call entirely
- If tag diff is empty, no tag payload sent to bridge at all (skip, not empty arrays)
- Fix existing bug: `len(payload) > 1` â†’ `len(payload) >= 1` so "agent sent only id with no fields/actions" also gets the no-op warning
- Same "No changes detected" message for the empty-edit case

### Warning behavior
- Per-tag warnings preserved: "Tag 'X' (id) is already on this task" / "Tag 'X' (id) is not on this task"
- NEW: replace no-op warning when replace produces empty diff (tags already match)
- Action-specific warnings remain separate from generic field no-op warning
- Warning message format: Claude's discretion (per-tag vs aggregated)

### Clearing semantics
- `replace: null` continues to mean "clear all tags" (same as `replace: []`)
- Consistent with actions convention: null = clear/reset to empty state (same as `move.beginning: null` = inbox)
- UNSET handles "no tag action" (omit actions.tags entirely)

### Bridge payload change (clean break)
- Remove all `tagMode` handling from bridge.js entirely
- Bridge accepts only `addTagIds` and `removeTagIds` -- no backward compat, no dead code
- Bridge still resolves IDs to Tag objects via `Tag.byIdentifier()` (OmniJS API requirement)
- Shared resolve helper preferred (DRY), but Claude's discretion on implementation
- Result: ~4 lines replacing ~45 lines of 4-branch dispatch

### Test strategy
- Bridge tests (test_bridge_edit.test.js): replace 4-mode tests with 2-3 minimal tests (addTagIds works, removeTagIds works, both together)
- Service tests (test_service.py): rewrite tag handling tests around diff computation. Test organization: Claude's discretion
- InMemoryRepository: adapt to new payload shape (addTagIds/removeTagIds), matching bridge interface exactly
- End-to-end behavior unchanged -- UAT skill (test-edit-operations) needs no updates
- No new UAT scenarios needed (new replace no-op warning is additive, not a behavior change)

### Claude's Discretion
- Whether tag resolution happens inside or outside the diff helper
- Test organization (by input mode vs diff outcome)
- Warning message aggregation strategy
- Exact bridge.js resolve helper structure
