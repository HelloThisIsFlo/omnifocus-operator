<objective>
Open the write surface for `completesWithChildren: Patch[bool]` and `type: Patch[TaskType]` on tasks (not projects — PROP-07 defers projects to v1.7, addressed in 56-07). Implement create-default resolution via the preferences plumbing built in 56-01: when the agent omits either field on `add_tasks`, the service writes the user's explicit OF preference value (with factory-default fallback `True` / `"parallel"`). Server never relies on OmniFocus's implicit defaulting.

Purpose: locked CONTEXT.md — task writes for these two fields; `null` rejected on both (no cleared state); `"singleActions"` rejected naturally via `TaskType` enum; `extra="forbid"` continues to reject the six derived read-only flags added in 56-05. Edit semantics are standard `Patch`: omit = no change, value = update.

Output:
- `AddTaskCommand` and `EditTaskCommand` gain the two new Patch fields.
- `AddTaskRepoPayload` carries resolved (never-null) values for both.
- `EditTaskRepoPayload` carries optional values (None = no change, set = update).
- `bridge.js` handleAddTask writes both fields unconditionally (service always resolves for add). handleEditTask writes only when present in params.
- Service's `_AddTaskPipeline` gets a `_resolve_type_defaults` step invoking the two new preferences getters.
- `tests/doubles/bridge.py:InMemoryBridge._handle_add_task` and `_handle_edit_task` store the two new fields so round-trip tests work.
- Two Wave-2 sanity guards (marked `# REMOVE IN 56-06` in 56-05) are deleted.

56-07 handles project-write rejection + round-trip verification + golden master scaffolding.
