# Phase 33.1: Refactor Frequency to Flat Model with Type-Optional Edits - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the 9-subtype discriminated Frequency union with a single flat model using 6 types. Optional specialization fields (`on_days`, `on`, `on_dates`) replace the sub-types (`weekly_on_days`, `monthly_day_of_week`, `monthly_day_in_month`). Write models get proper `CommandModel` inheritance. Type becomes optional on edits. All model validation moves from `validate.py` to models. Breaking change on read output — acceptable (pre-release, single user, from source).

</domain>

<decisions>
## Implementation Decisions

### Type Collapse: 9 Types → 6 Types
- **D-01:** Collapse sub-types into their base types. `weekly_on_days` → `weekly` with optional `on_days`. `monthly_day_of_week` → `monthly` with optional `on`. `monthly_day_in_month` → `monthly` with optional `on_dates`. The sub-types were an artifact of the discriminated union, not a domain concept.
- **D-02:** `type` field uses `Literal["minutely", "hourly", "daily", "weekly", "monthly", "yearly"]` on all three models (read, add spec, edit spec). Strictest option — catches typos at Pydantic level, preserves schema documentation on outputSchema.
- **D-03:** Field names stay: `on_days`, `on`, `on_dates`. They read naturally in context: "on days MO, FR", "on the second tuesday", "on dates 1 and 15."

### Three Flat Models
- **D-04:** `Frequency` (read, `OmniFocusBaseModel`): `type` required (no default → survives `exclude_defaults`), `interval: int = 1`, optional fields `on_days`, `on`, `on_dates` with `None` defaults.
- **D-05:** `FrequencyAddSpec` (write-add, `CommandModel`): Same field shapes as `Frequency`. `extra="forbid"` catches agent typos. Fixes the Phase 33 D-05 gap where write specs embedded a read model.
- **D-06:** `FrequencyEditSpec` (write-edit, `CommandModel`): Uses `Patch`/`PatchOrClear` pattern. `type: Patch[Literal[...]] = UNSET`. `interval: Patch[int] = UNSET`. `on_days: PatchOrClear[list[str]] = UNSET`. `on: PatchOrClear[dict[str, str]] = UNSET`. `on_dates: PatchOrClear[list[int]] = UNSET`. Consistent with how `EditTaskCommand` handles clearable fields.

### Cross-Type Field Validation
- **D-07:** Hard error for cross-type fields. `on_days` sent with type `daily` → educational error: "on_days is not valid for type 'daily'." VALID-02 contract preserved. Contradiction is unrecoverable — agent must fix it.
- **D-07a:** Cross-type validators live on `Frequency` and `FrequencyAddSpec` only — NOT on `FrequencyEditSpec`. The edit spec is a patch container with no business rules. On the edit path, cross-type validation fires when the merged `Frequency` is constructed (step 3 below), catching both explicit-type and inferred-type cases.
- **D-08:** `on` and `on_dates` are mutually exclusive on `monthly`. Setting one in an edit auto-clears the other (no explicit null needed). This includes the merge case: if the edit sets `on` and `on_dates` comes from the existing task, the domain layer auto-clears `on_dates` from the merged result. Error only if both explicitly set in the same edit payload. On the add path, both set = hard error (no existing value to auto-clear). Auto-clear logic lives in the domain layer (same kind of smart normalization as `normalize_empty_on_dates`).

### Type-Optional Edit Semantics
- **D-09:** Type omitted on edit = same-type merge from existing task. Cross-type fields on inferred type = hard error with educational message. Type changes always require explicit `type` + full frequency object (EDIT-13/14 preserved). Edit merge flow: (1) create `FrequencyEditSpec` from agent input (patch container, no validation), (2) service infers type from existing or uses explicit type, merges fields, (3) construct `Frequency` from merged result → `@model_validator` fires here → catches cross-type violations.
- **D-10:** Adding/removing specialization fields on the same base type is NOT a type change. Adding `on_days` to a `weekly` task, clearing `on_dates` from a `monthly` task — these are field-level edits, not type changes. No full object required.
- **D-11:** Merge uses `is_set()` TypeGuard (existing `Patch`/`PatchOrClear` pattern), not `model_fields_set`. Consistent with the rest of the codebase.

### Serialization: interval=1 Suppression
- **D-12:** `@field_serializer('frequency')` on `RepetitionRule` calls `freq.model_dump(exclude_defaults=True, by_alias=True)`. Verified safe via Pydantic Context7 docs: `@field_serializer` does NOT affect JSON Schema generation, and `to_jsonable_python` (FastMCP's serialization) respects field serializers. `interval=1` omitted from read output; `type` (no default) always included.

### Validation Migration: validate.py → Models
- **D-13:** All pure-input validation moves to Pydantic model validators. Validators raise `ValueError` with the educational message strings from `agent_messages/errors.py` — Pydantic wraps these in `ValidationError`. The existing `_format_validation_errors()` in `server.py` already catches `ValidationError` from `model_validate()` and extracts agent-friendly messages. No new error handling plumbing needed.
  - `Frequency` / `FrequencyAddSpec`: `@model_validator` for cross-type fields and mutual exclusion; `@field_validator` for day code normalization; `Field(ge=1)` for interval
  - `EndByOccurrences`: `Field(ge=1)` for occurrences
  - `AddTaskCommand` / `EditTaskCommand`: `@field_validator` for task name validation
  - Cross-model validation (needs task state) stays in service layer
  - `validate.py` shrinks or is removed entirely

### Empty Specialization Field Normalization
- **D-17:** Empty specialization fields (`on_days: []`, `on: {}`, `on_dates: []`) normalize to `None` + warn. All three fields, same pattern as existing `normalize_empty_on_dates`. This stays in the service/domain layer — semantic normalization ("ambiguous input → canonical form") lives in the service alongside similar decisions (e.g., note null → empty string). The domain logic extends to cover all three fields.

### Builder Type Compatibility
- **D-18:** `build_rrule()` needs to accept both `Frequency` (read model, used after edit merge) and `FrequencyAddSpec` (write model, used on add path). Both have identical fields. Claude's discretion: union type annotation, protocol, or convert to `Frequency` before calling.

### Tool Descriptions
- **D-14:** add_tasks repetition section — use this template verbatim:

```
- repetitionRule: Repetition rule (all three root fields required on creation)
  - frequency (required): Object with "type" + optional specialization fields
    - {type: "minutely", interval: N}
    - {type: "hourly", interval: N}
    - {type: "daily", interval: N}
    - {type: "weekly", interval: N}  — optionally add onDays
    - {type: "weekly", interval: N, onDays: ["MO","WE","FR"]}  — specific days (MO-SU)
    - {type: "monthly", interval: N}  — optionally add on or onDates
    - {type: "monthly", interval: N, on: {"second": "tuesday"}}
        ordinals: first/second/third/fourth/fifth/last
        days: monday-sunday, weekday, weekend_day
    - {type: "monthly", interval: N, onDates: [1, 15]}
        valid dates: 1 to 31, use -1 for last day of month
    - {type: "yearly", interval: N}
    - interval defaults to 1, omit or set explicitly
    - on and onDates are mutually exclusive
  - schedule (required): "regularly" / "regularly_with_catch_up" / "from_completion"
  - basedOn (required): "due_date" / "defer_date" / "planned_date"
  - end: {"date": "ISO-8601"} or {"occurrences": N} — omit for no end

  Examples:
    Every 3 days from completion:
      {frequency: {type: "daily", interval: 3}, schedule: "from_completion", basedOn: "defer_date"}

    Every 2 weeks on Mon and Fri, stop after 10:
      {frequency: {type: "weekly", interval: 2, onDays: ["MO", "FR"]}, schedule: "regularly", basedOn: "due_date", end: {occurrences: 10}}

    Last Friday of every month:
      {frequency: {type: "monthly", on: {"last": "friday"}}, schedule: "regularly", basedOn: "due_date"}
```

- **D-15:** edit_tasks repetition section — use this template verbatim:

```
- repetitionRule: Set, update, or clear a repetition rule
  - Full rule: same shape as add_tasks (frequency, schedule, basedOn required)
  - Partial update: send only changed fields, omitted root fields preserved
    - frequency.type optional (inferred from existing task)
    - frequency.type required when changing to a different type
    - Same type: omitted frequency fields preserved from existing rule
    - Different type: full replacement, defaults apply like creation
    - onDays/on/onDates: null to clear, omit to preserve
    - on and onDates are mutually exclusive — setting one auto-clears the other
  - end: null to clear, omit to preserve
  - null to clear the repetition rule

  Examples:
    Change just the interval (type inferred from existing):
      {frequency: {interval: 5}}

    Add specific days to a weekly task (no type change):
      {frequency: {onDays: ["MO", "WE", "FR"]}}

    Remove day constraint from weekly:
      {frequency: {onDays: null}}

    Switch monthly from dates to weekday pattern (onDates auto-cleared):
      {frequency: {on: {"last": "friday"}}}

    Change from daily to weekly (type required):
      {frequency: {type: "weekly", onDays: ["MO", "FR"]}}

    Clear:
      null
```

<specifics>
## Specific Ideas

- The 6-type collapse was discovered during discussion by observing that the normalize+warn pattern (empty onDates → monthly) applied identically to all three sub-type pairs. The sub-types were an artifact of the discriminated union, not a domain concept.
- `@field_serializer` on `RepetitionRule.frequency` (the parent, not Frequency itself) was chosen specifically to avoid the Phase 32.1 `@model_serializer` schema erasure problem. Verified via Pydantic Context7 docs.
- Mutually exclusive monthly fields (`on`/`onDates`) auto-clear on edit because there's zero ambiguity — you can't have both. Requiring explicit null is ceremony.
- The `FrequencyEditSpec` uses `PatchOrClear` on specialization fields because clearing `on_days` on a weekly task is a meaningful operation (removes day constraints), unlike `interval` where clearing makes no sense (it always has a value).
- Tool description templates iterated during Phase 33 discussion remain the starting point — updated for 6 types and type-optional semantics in this phase.
