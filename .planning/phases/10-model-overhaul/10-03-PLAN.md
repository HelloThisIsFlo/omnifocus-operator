**repository.py -- Wire adapter into _refresh:**

The adapter must run for RealBridge and SimulatorBridge (which output bridge-format data) but NOT for InMemoryBridge (which returns new-shape data directly).

Approach: The adapter is idempotent-safe by design -- it pops old fields and adds new ones. However, per user decision, InMemoryBridge fixtures use new-shape directly (no adapter needed). The cleanest approach is to ALWAYS run the adapter when bridge data contains old-format markers (presence of `status` key on tasks), OR check bridge type.

**Recommended approach:** Always call `adapt_snapshot(raw)` in `_refresh`. The adapter should be safe to call on already-adapted data (InMemoryBridge data won't have the old fields to pop, and already has urgency/availability). BUT this means InMemoryBridge test data must NOT have a `status` field -- which is already true after Plan 02 updated the factories.

Update `_refresh`:
```python
from omnifocus_operator.bridge.adapter import adapt_snapshot

async def _refresh(self, current_mtime: int) -> DatabaseSnapshot:
    raw: dict[str, Any] = await self._bridge.send_command("snapshot")
    adapt_snapshot(raw)  # Transform bridge-format -> new model shape (no-op if already new shape)
    snapshot = DatabaseSnapshot.model_validate(raw)
    self._last_mtime_ns = current_mtime
    return snapshot
```

IMPORTANT: The adapter must handle the case where a dict already has `urgency`/`availability` and no `status` field (InMemoryBridge data). If `status` key is not present, skip that entity. Update `_adapt_task` and `_adapt_project` in adapter.py to check `if "status" not in raw: return` at the top. Similarly for tags/folders. This makes the adapter safe to run on any data shape.
