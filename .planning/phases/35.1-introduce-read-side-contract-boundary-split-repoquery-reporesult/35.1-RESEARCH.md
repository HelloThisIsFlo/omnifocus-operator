**Domain:** Pydantic model taxonomy, Python package reorganization, import migration
**Confidence:** HIGH

## Summary

Pure structural refactor. No behavioral changes, no new dependencies, no external tools. The work is:
1. Create 5 new Pydantic models (4 RepoQuery + 1 ListRepoResult generic)
2. Reorganize `contracts/use_cases/` from flat files into per-use-case packages (`list/`, `add/`, `edit/`)
3. Update Repository protocol + HybridRepository + BridgeRepository signatures to use repo-boundary types
4. Migrate all import sites to new paths (clean break, no re-exports from old paths)

All patterns are established in the codebase (write-side split already exists). Risk is low -- mypy + 1229 tests catch every broken import.

**Primary recommendation:** Execute in 3 waves: (1) create new models + package structure, (2) update protocol/repository signatures, (3) migrate all imports. Each wave is independently verifiable.

### Pattern: RepoQuery Model Definition

Mirrors the existing Query models exactly. Base class is `QueryModel` (inherits `StrictModel` with `extra="forbid"`).

```python
# contracts/use_cases/list/tasks.py
from omnifocus_operator.contracts.base import QueryModel
from omnifocus_operator.models.enums import Availability
from pydantic import Field

class ListTasksQuery(QueryModel):
    """Agent-facing query -- validated filter + pagination."""
    # ... (existing fields, unchanged)

class ListTasksRepoQuery(QueryModel):
    """Repo-facing query -- identical fields today, diverges in Phase 35.2."""
    in_inbox: bool | None = None
    flagged: bool | None = None
    project: str | None = None
    tags: list[str] | None = None  # names today, Phase 35.2 adds tag_ids
    estimated_minutes_max: int | None = None
    availability: list[Availability] = Field(
        default_factory=lambda: [Availability.AVAILABLE, Availability.BLOCKED]
    )
    search: str | None = None
    limit: int | None = None
    offset: int | None = None
```

### Pattern: ListRepoResult Generic

Mirrors `ListResult[T]` exactly. Inherits `OmniFocusBaseModel` (outbound, system-constructed).

```python
# contracts/use_cases/list/common.py
from omnifocus_operator.models.base import OmniFocusBaseModel

class ListResult[T](OmniFocusBaseModel):
    """Agent-facing result container."""
    items: list[T]
    total: int
    has_more: bool

class ListRepoResult[T](OmniFocusBaseModel):
    """Repo-facing result container -- no warnings field (added to ListResult in ~~Phase 37~~ Phase 36)."""
    items: list[T]
    total: int
    has_more: bool
```

### Pattern: Protocol Signature Split

Write-side pattern already established. Read-side mirrors it:

```python
# contracts/protocols.py
class Service(Protocol):
    async def list_tasks(self, query: ListTasksQuery) -> ListResult[Task]: ...

class Repository(Protocol):
    async def list_tasks(self, query: ListTasksRepoQuery) -> ListRepoResult[Task]: ...
```

### Anti-Patterns to Avoid
- **Shared type across boundaries:** Don't use `ListTasksQuery` on both Service and Repository protocols -- the entire point of this phase is separate types
- **Re-exports from old paths:** D-04 explicitly forbids `list_entities.py` re-exporting from new locations. Clean break.
- **Forgetting `model_rebuild()`:** New models must be added to `contracts/__init__.py` with `model_rebuild()` calls for forward reference resolution

## Common Pitfalls

### Pitfall 1: Forgetting `contracts/__init__.py` Registration
**What goes wrong:** New RepoQuery/ListRepoResult models are created but not imported + `model_rebuild()`'d in `contracts/__init__.py`. Forward references fail at runtime.
**How to avoid:** After creating each new model, immediately add import + `_ns` dict entry + `model_rebuild()` call in `contracts/__init__.py`.
**Warning signs:** `PydanticUndefinedAnnotation` errors at import time.

### Pitfall 2: HybridRepository Sync Methods
**What goes wrong:** Async wrapper methods updated to new types but internal `_list_*_sync` methods still use old types.
**How to avoid:** Update both `async def list_tasks` AND `def _list_tasks_sync` signatures in `hybrid.py`. There are 10 methods total (5 async + 5 sync).
**Warning signs:** mypy errors on return type mismatches between async wrapper and sync implementation.

### Pitfall 3: BridgeRepository Stub Signatures
**What goes wrong:** `bridge.py` has `NotImplementedError` stubs for all 5 list methods. These must also update signatures to satisfy the Repository protocol.
**How to avoid:** Update all 5 list method signatures in `bridge.py` to use RepoQuery/ListRepoResult.
**Warning signs:** mypy protocol conformance errors.

### Pitfall 4: Service Imports Without Signature Change
**What goes wrong:** Service imports change (D-04) but service list method signatures stay the same (D-06). It's easy to accidentally change service signatures too.
**How to avoid:** Service protocol is unchanged. Service class only needs import path updates, not signature changes.
**Warning signs:** Service tests failing due to unexpected type changes.

### Pitfall 5: Query Builder Receives Wrong Type
**What goes wrong:** `query_builder.py` functions take `ListTasksQuery`/`ListProjectsQuery` parameters. After the split, the repository passes RepoQuery objects. If the query builder signatures aren't updated, mypy catches it but the mental model is confusing.
**How to avoid:** `query_builder.py` functions should accept `ListTasksRepoQuery`/`ListProjectsRepoQuery` since they serve the repository layer.
**Warning signs:** mypy type errors in `hybrid.py` when passing RepoQuery to query builder functions.

### Pitfall 6: Test Files Using Query Models Against Repository
**What goes wrong:** `test_hybrid_repository.py` constructs `ListTasksQuery()` objects to pass to `hybrid_repo.list_tasks()`. After the split, the repository accepts `ListTasksRepoQuery` -- but the test still passes the old Query type.
**How to avoid:** All test files that call repository methods directly must switch to RepoQuery construction. Tests that call service methods keep using Query.
**Warning signs:** mypy errors in test files; runtime: Pydantic might accept it due to structural compatibility, but mypy won't.
