# Phase 35.1: Introduce Read-Side Contract Boundary Split (RepoQuery / RepoResult) - Context

**Gathered:** 2026-03-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Structural refactor — introduce repo-facing query and result types for the read side, mirroring the write side's `Command → RepoPayload` / `Result ← RepoResult` split. Reorganize `contracts/use_cases/` into per-use-case packages. **No behavioral changes.**

This phase applies the [Structure Over Discipline](docs/structure-over-discipline.md) principle to the read-side contracts: the split exists so divergence is "add a field," not a design decision. Fields are identical across boundaries today — the boundary itself is the deliverable.

### Models Created (exhaustive list)

| Model | Location | Base class | Purpose |
|-------|----------|------------|---------|
| `ListTasksRepoQuery` | `contracts/use_cases/list/tasks.py` | `QueryModel` | Repo-facing query — identical to `ListTasksQuery` today |
| `ListProjectsRepoQuery` | `contracts/use_cases/list/projects.py` | `QueryModel` | Repo-facing query — identical to `ListProjectsQuery` today |
| `ListTagsRepoQuery` | `contracts/use_cases/list/tags.py` | `QueryModel` | Repo-facing query — identical to `ListTagsQuery` today |
| `ListFoldersRepoQuery` | `contracts/use_cases/list/folders.py` | `QueryModel` | Repo-facing query — identical to `ListFoldersQuery` today |

No `ListPerspectivesRepoQuery` — perspectives have no query model (D-09 from Phase 34).

<decisions>
## Implementation Decisions

### RepoQuery Field Design (D-01)

- **D-01a:** RepoQuery fields are **identical** to Query fields — name strings, not IDs
- **D-01b:** `ListTasksRepoQuery.tags` is `list[str] | None` (names), same as `ListTasksQuery.tags`
- **D-01c:** `ListProjectsRepoQuery.folder` is `str | None` (name), same as `ListProjectsQuery.folder`
- **D-01d:** Per Structure Over Discipline: "separate types, even when fields are identical today"
- **D-01e:** Phase 35.2 adds `tag_ids` to RepoQuery when behavioral change lands — divergence becomes "add a field"

### ListRepoResult — Defer Warnings (D-02)

- **D-02a:** `ListRepoResult[T]` has same fields as `ListResult[T]`: `items`, `total`, `has_more`
- **D-02b:** No `warnings` field on either type in Phase 35.1
- **D-02c:** Phase 37 adds `warnings: list[str] | None` to `ListResult[T]` when "did you mean?" suggestions are built
- **D-02d:** Both types identical today — same Structure Over Discipline logic as D-01

### Full Package Reorganization (D-03)

- **D-03a:** All three use cases become packages: `list/`, `add/`, `edit/`
- **D-03b:** `list_entities.py` → `list/` package with per-entity modules + `common.py`
- **D-03c:** `add_task.py` → `add/tasks.py`
- **D-03d:** `edit_task.py` → `edit/tasks.py`
- **D-03e:** Structure Over Discipline: pave the path before divergence needs it

### Clean Break Import Migration (D-04)

- **D-04a:** Update all ~17 import sites (9 src + 8 test files) to new paths
- **D-04b:** No re-exports from old paths — clean break
- **D-04c:** mypy + 534 tests catch every broken import
- **D-04d:** Not a published library — no external import compatibility concern

### Protocol Signature Update (D-05)

- **D-05a:** Repository protocol updated: `list_tasks(query: ListTasksRepoQuery) -> ListRepoResult[Task]`
- **D-05b:** Service protocol unchanged: `list_tasks(query: ListTasksQuery) -> ListResult[Task]`
- **D-05c:** All 5 list methods on Repository protocol switch to RepoQuery/ListRepoResult
- **D-05d:** `list_perspectives()` returns `ListRepoResult[Perspective]` on Repository, `ListResult[Perspective]` on Service (no query change — still no query model)
- **D-05e:** HybridRepository implementations update signatures to match

### Service Stubs Unchanged (D-06)

- **D-06a:** Service list method stubs remain `NotImplementedError`
- **D-06b:** Service wiring (Query→RepoQuery conversion, repo call, RepoResult→Result conversion) is Phase 37's scope
- **D-06c:** Phase 35.1 is contracts-only — no service behavior changes

### Claude's Discretion

- Whether `perspectives.py` exists in `list/` as an empty placeholder or is omitted (no query models for perspectives)
- Internal organization of `__init__.py` re-exports within each package
- Test organization for the new contract models (new test file vs extending existing)

## Existing Code Insights

### Reusable Assets
- `QueryModel` (contracts/base.py): Base class for all RepoQuery models — `extra="forbid"` via `StrictModel`
- `OmniFocusBaseModel` (models/base.py): Base class for `ListRepoResult[T]` — camelCase serialization
- `ListResult[T]` (contracts/use_cases/list_entities.py): Template for `ListRepoResult[T]` — identical fields

### Established Patterns
- **Write-side boundary split**: `AddTaskCommand` (agent) → `AddTaskRepoPayload` (repo) with different field types (`tags: list[str]` → `tag_ids: list[str]`). Read side mirrors this but fields are identical today.
- **Protocol boundary types**: Repository protocol uses repo-boundary types (`AddTaskRepoPayload`, `AddTaskRepoResult`), Service protocol uses agent-boundary types (`AddTaskCommand`, `AddTaskResult`). List methods should follow the same pattern.
- **Package structure**: `contracts/use_cases/` currently flat files. Phase 35.1 introduces per-use-case packages.

### Integration Points
- `contracts/protocols.py`: Repository list method signatures update to RepoQuery/ListRepoResult
- `contracts/use_cases/`: Full directory reorganization (flat files → packages)
- `repository/hybrid.py`: Method signatures update from Query → RepoQuery, ListResult → ListRepoResult
- `service/service.py`: Import paths update (stubs stay as NotImplementedError)
- All test files importing from `contracts/use_cases/list_entities.py`: Import paths update

<specifics>
## Specific Ideas

### Structure Over Discipline as Phase Rationale
This phase is a direct application of the project's core architectural principle. The todo captures it well: "Every entity gets a RepoQuery even if identical to Query today (Structure Over Discipline — divergence becomes 'add a field,' not a design decision)." The same applies to `ListRepoResult[T]` vs `ListResult[T]`.

### Phase 34 D-06b Superseded
Phase 34 decision D-06b ("query model is shared across layers — same field carries names at input and IDs after service resolution") is replaced by this phase. The query model is no longer shared — Query stays at the agent boundary, RepoQuery lives at the repo boundary. This was always the intended direction per model-taxonomy.md; Phase 34 deferred the split.
