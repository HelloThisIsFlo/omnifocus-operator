# Phase 36.1: Migrate Write Tools to Typed Params with Validation Middleware - Context

**Gathered:** 2026-03-31
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace `list[dict[str, Any]]` handler signatures on `add_tasks` and `edit_tasks` with typed `list[AddTaskCommand]` / `list[EditTaskCommand]` params. Add `ValidationReformatterMiddleware` to catch Pydantic `ValidationError`, reformat into clean `ToolError`. Remove manual `model_validate()` + try/except from handlers. Agents get rich inputSchema (52-61 fields/enums/refs vs 2 today).

**Requirements:** WRIT-01 through WRIT-11 (see `.planning/REQUIREMENTS.md`)

**NOT in scope:**
- Docstring restructuring (manual step post-phase — see D-09)
- Custom UNSET error type (`omitted_field_sentinel`) — separate enhancement, deferred
- Batch support expansion
- `strict_input_validation` on FastMCP (lax mode coercion like `flagged: "yes"` → `True` is acceptable)

</domain>

<decisions>
## Implementation Decisions

### Middleware Architecture

- **D-01:** `ValidationReformatterMiddleware` goes in existing `src/omnifocus_operator/middleware.py` alongside `ToolLoggingMiddleware`
- **D-02:** `_format_validation_errors` and `_clean_loc` move from `server.py` into `middleware.py` — they're middleware concerns, not server concerns
- **D-03:** Middleware stack ordering: **Validation INSIDE logging** — add `ValidationReformatterMiddleware` first (innermost), `ToolLoggingMiddleware` second (outermost). Logging MW sees the reformatted `ToolError`, logs what agent received, captures timing. Full observability.
- **D-04:** Middleware scope: **All tools (global)** — no tool-name filtering. `ValidationError` won't fire on read tools (`id: str`) under normal operation — zero-cost coverage. Future tools get automatic coverage.

### Handler Migration

- **D-05:** Handler signatures change: `items: list[dict[str, Any]]` → `items: list[AddTaskCommand]` / `items: list[EditTaskCommand]`
- **D-06:** Handler bodies: remove try/except + `model_validate()` block. Use typed `items[0]` directly.
- **D-07:** Batch limit check (`if len(items) != 1` → `ValueError`) stays in handler, NOT as `max_length=1` on the list param. Temporary scaffolding — no need to bake it into the schema where it becomes a contract agents cache. `ValueError` path remains unchanged (FastMCP wraps to `ToolError`).

### Error Formatting

- **D-08:** Error location prefix: `"Task {idx+1}: {field}"` format instead of raw `items.0.field`. Readable for batch-of-1, scales to future batch support without code changes.
- **D-08a:** UNSET filtering switches from fragile string matching (`"_Unset" in e["msg"]`) to ctx-based matching (`e.get("ctx", {}).get("class") == "_Unset"`). More robust against Pydantic message changes across versions.

### Docstrings

- **D-09:** Leave the 45-55 line field-listing docstrings as-is during this migration. After typed params are live, inspect actual agent experience via MCP Inspector, then manually iterate on docstring density. **UAT must verify docstrings have been updated by the human before phase is considered complete.**

### PydanticJsonSchemaWarning

- **D-10:** Suppress the `PydanticJsonSchemaWarning` about UNSET defaults not being JSON-serializable. Expected and correct — UNSET excluded from schema by design. Suppression location is implementation detail.

### Canary Test

- **D-11:** Include a canary test verifying that FastMCP's validation still happens inside `call_next()` (i.e., the middleware can catch it). Guards against future FastMCP version moving validation before middleware. **The test MUST have a clear docstring and failure message explaining what it guards and what to do when it fails** — future maintainers will have forgotten the context.

### Folded Todos

- **Migrate write tools to typed params with validation middleware** (2026-03-29) — this todo IS the phase. Its 7 brainstormed decisions are incorporated as D-01 through D-11 above.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ToolLoggingMiddleware` in `middleware.py` — template for the new middleware's structure (inherits `Middleware`, implements `on_call_tool`)
- `_format_validation_errors` + `_clean_loc` in `server.py` — existing logic to move into middleware module; needs `items.N.` prefix stripping and ctx-based UNSET filtering
- `AddTaskCommand` / `EditTaskCommand` — already exist with full Pydantic validation, `extra="forbid"`, camelCase aliases

### Established Patterns
- FastMCP middleware: `Middleware` base class, `MiddlewareContext`, `call_next` pattern
- Error flow: `ValueError` raised in handlers → FastMCP wraps to `ToolError` for MCP clients
- Middleware registration: `mcp.add_middleware()` in `create_server()`, LIFO order (last added = outermost)

### Integration Points
- `create_server()` in `server.py` — where middleware is registered
- `_register_tools()` in `server.py` — where handler signatures and bodies change
- Test files asserting on validation error paths (ToolError type, error messages)

</code_context>

<specifics>
## Specific Ideas

- The canary test is specifically about future-proofing against FastMCP internal changes — the middleware relies on `FunctionTool.run()` → `type_adapter.validate_python()` happening inside the middleware chain, which is undocumented behavior
- Error messages should maintain parity with current output where possible (10/12 exact match per FINDINGS.md integration tests)

</specifics>

<deferred>
## Deferred Ideas

- **Docstring restructuring** — remove structural field docs (redundant with rich inputSchema), keep behavioral docs. Deferred to post-migration MCP Inspector review.
- **Custom UNSET error type** (`omitted_field_sentinel`) — modifies `_Unset` internals for even more robust filtering. Separate from ctx-based filtering (included in this phase).
- **Stale `$ref` discriminator mapping** — FastMCP/Pydantic interaction bug. Likely moot with flat frequency model (v1.2.3). Monitor if it surfaces.

</deferred>
