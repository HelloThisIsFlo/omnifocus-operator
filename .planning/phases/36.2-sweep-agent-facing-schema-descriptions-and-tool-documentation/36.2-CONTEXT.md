<decisions>
## Implementation Decisions

### Docstring Cleanup Strategy
- **D-01:** Two-tier approach: strip implementation notes from all model docstrings that appear in schema. Keep brief agent-useful behavioral summaries on nested specs agents construct (e.g., FrequencyEditSpec → "Patch individual frequency sub-fields; omit fields to leave unchanged"). Remove docstrings entirely where the model name is self-explanatory.
- **D-02:** Specific docstrings to clean (confirmed by three-model agent review):
  - `FrequencyAddSpec`: remove "Same field shapes as Frequency, with extra='forbid' from CommandModel. Validators delegate to shared functions in models.repetition_rule."
  - `FrequencyEditSpec`: remove "Single-field boundary checks (interval >= 1) are validated here. Cross-type validation fires when the merged Frequency is constructed in the service layer."
  - `RepetitionRuleEditSpec`: remove "All fields default to UNSET (no change)... Frequency uses FrequencyEditSpec with Patch/PatchOrClear fields."
  - `Schedule`: remove "derived from scheduleType + catchUp"
  - `TagAction`: remove "by the model validator" — just say "rejected" or "returns an error"
  - `MoveAction`: replace `None` with `null` (Python docstring convention leaking into JSON-facing description)
  - `Task` module docstring: remove "Maps to the flattenedTasks.map() output in the bridge script."
  - `Project` module docstring: same bridge reference removal
  - `Tag` module docstring: same
  - `AddTaskCommand`: "Agent instruction to create a task." → either remove or replace with something useful
  - `EditTaskCommand`: same

### Tool Description Density
- **D-03:** Field descriptions on Pydantic models + slim tool docstrings. Add `Field(description=...)` to key fields where the field name/type leave ambiguity. Trim tool docstrings to remove all field listings that the schema already communicates. Keep: patch semantics contract, repetition rule examples, tag/move/lifecycle mode constraints, batch limit note.
- **D-04:** Fluency filter for `Field(description=...)`: describe only when the field name + type leave ambiguity about behavior or valid values. `flagged: bool` needs nothing. `tags: list[str]` needs "Tag names (case-insensitive) or IDs". `dueDate: AwareDatetime` needs timezone guidance. Apply same filter to both write-side and read-side models.
- **D-05:** Read-side model fields that need descriptions (fluency filter applied):
  - `effective_due_date`, `effective_defer_date`, `effective_planned_date`, `effective_flagged`, `effective_completion_date`, `effective_drop_date` — "Inherited from parent project/task if not set directly on this entity"
  - `children_are_mutually_exclusive` on Tag — explain what this means in practice
  - `next_task` on Project — explain what this field represents
  - Other fields (name, id, flagged, note, etc.) — no description needed, self-explanatory
  - Apply the fluency test: would an agent reading just the field name + type understand it? If yes, skip.

### Gap Documentation
- **D-06:** Tag resolution behavior — document in BOTH tool descriptions (add_tasks, edit_tasks) and on relevant `Field(description=...)`:
  - Tags can be names (case-insensitive) or IDs; can mix both in one list
  - ID match takes precedence over name match
  - Non-existent names are rejected (error, not auto-created)
  - Ambiguous names (case-insensitive collision) return an error
  - Applies to `add_tasks.tags`, `edit_tasks.actions.tags.add`, `edit_tasks.actions.tags.remove`, `edit_tasks.actions.tags.replace`
- **D-07:** Timezone behavior — document on date `Field(description=...)` and as a concise callout in tool descriptions:
  - Datetimes must include timezone info (ISO 8601 with offset or Z)
  - Naive datetimes are rejected
- **D-08:** Partial repetitionRule on task with no existing rule — document in edit_tasks tool description:
  - When editing a task that has no existing repetition rule, you must provide a complete rule (all three root fields: frequency, schedule, basedOn)
  - Partial updates (omitting root fields) only work when a rule already exists on the task
- **D-09:** `plannedDate` / `basedOn: "planned_date"` — add brief explanation of what planned date means in OmniFocus context

### Read Tool Descriptions
- **D-10:** Improve read tool descriptions:
  - `get_all`: hint at top-level response structure (tasks, projects, tags, folders, perspectives arrays)
  - `get_task`, `get_project`, `get_tag`: mention key non-obvious fields agents will encounter
  - Note: outputSchema is NOT advertised by FastMCP v3 — agents genuinely cannot see response shape before calling. This makes read tool descriptions more important.
- **D-11:** Apply same `Field(description=...)` fluency filter to read-side models (Task, Project, Tag, common fields)

### Additional Findings from Agent Review
- **D-12:** Remove field-by-field listings from tool docstrings that duplicate the inputSchema (e.g., "name (required): Task name", "dueDate: Due date (ISO 8601)"). Schema already communicates types, required status, defaults, enums.
- **D-13:** Repetition rule examples are non-redundant and must be kept — they communicate behavioral rules (on/onDates mutual exclusivity, type inference) that the schema cannot express.
- **D-14:** `actions.lifecycle: "drop"` — add brief explanation for agents unfamiliar with OmniFocus ("drop = skip/cancel without completing")
- **D-15:** camelCase note on all read tools — add "The response uses camelCase field names." to all four read tool descriptions (get_all, get_task, get_project, get_tag). Agents have no outputSchema visibility, so this is genuinely useful for first-time callers.

### Verbatim Tool Docstrings
- **D-16:** The `add_tasks` and `edit_tasks` tool docstrings have been collaboratively drafted and approved. Downstream agents MUST use these verbatim (not paraphrase or restructure). The docstrings are in the "Approved Docstrings" section below.
- **D-17:** All three date fields — `dueDate`, `deferDate`, and `plannedDate` — get `Field(description=...)` annotations informed by `docs/omnifocus-concepts.md`. Each description should be a concise summary of that date's purpose (not just "ISO 8601 date"). The implementing agent MUST read `docs/omnifocus-concepts.md` before writing these descriptions. The goal: an agent reading only the field description understands when to use each date type and doesn't conflate them (e.g., setting a due date for a mere intention).
- **D-18:** `Schedule` enum description "derived from scheduleType + catchUp" is implementation leakage — replace with agent-useful text or remove.

- Exact wording of other `Field(description=...)` annotations — behavioral intent is captured above, phrasing is implementation detail
- Whether to keep or remove the `"Agent instruction to create/edit a task."` item descriptions — leaning toward replacing with something minimally useful or removing

<code_context>
## Existing Code Insights

### Key Finding: outputSchema Not Advertised
FastMCP v3 does NOT include `outputSchema` in the MCP `tools/list` response. Return type annotations (`-> Task`, `-> AllEntities`) are used for serialization but not schema generation. Agents have zero visibility into response shape until they call a tool. This makes read tool descriptions and read-model Field descriptions more important than initially assumed.

### Schema Description Sources
Two distinct layers produce agent-visible text:
1. **Tool docstrings** (`server.py`) → tool-level `description` in MCP
2. **Model docstrings** (contracts/, models/) → `description` fields on schema objects/properties via Pydantic `model_json_schema()`

The tool docstrings are generally agent-friendly but verbose. The model docstrings are where implementation details leak. These two layers need different treatment.

### Current Field Description State
No Pydantic fields currently have `Field(description=...)` annotations on write-side commands or read-side models. All field-level documentation is in the tool docstrings. Adding `Field(description=...)` creates a single source of truth at the schema level.

### Established Patterns
- `test_output_schema.py` validates that serialized output matches MCP outputSchema expectations — must pass after changes
- camelCase aliasing via `alias_generator` on base models — field descriptions will appear alongside camelCase names in the schema
- `CommandModel` base: `extra="forbid"`, camelCase aliases — all contract models inherit this

</code_context>

- The three-model agent review produced a prioritized list of concrete issues. The consensus items (flagged by all three: Haiku, Sonnet, Opus) are highest priority fixes.
- The user's optimization target is **agent fluency** — effort/file count is irrelevant; what matters is whether the description helps an agent call the tool correctly on the first try.

<deferred>
## Deferred Ideas

- **outputSchema enablement** — FastMCP v3 doesn't advertise outputSchema. Investigating why and enabling it would be a separate enhancement. For now, the sweep adds what descriptions it can to read tool docstrings.
- **Schema constraint tightening** — Adding `maxItems: 1` to items arrays, `oneOf` for move action mutual exclusivity, enum constraints on `on`/`onDays`/`onDates` keys. These are structural schema changes, not documentation.
- **Error catalog** — Documenting common error shapes and validation error formats. Useful but separate from the description sweep.

<approved_docstrings>
## Approved Docstrings

**These are the final, user-approved tool docstrings. Use verbatim.**

### add_tasks

```
Create tasks in OmniFocus. Limited to 1 item per call.

Tags accept names (case-insensitive) or IDs; you can mix both.
Non-existent names are rejected. Ambiguous names (case-insensitive
collision) return an error.

All date fields require timezone info (ISO 8601 with offset or Z).
Naive datetimes are rejected.

repetitionRule requires all three root fields (frequency, schedule,
basedOn) when creating. on and onDates within frequency are
mutually exclusive.

Examples (repetitionRule):
  Every 3 days from completion:
    {frequency: {type: "daily", interval: 3}, schedule: "from_completion", basedOn: "defer_date"}

  Every 2 weeks on Mon and Fri, stop after 10:
    {frequency: {type: "weekly", interval: 2, onDays: ["MO", "FR"]}, schedule: "regularly", basedOn: "due_date", end: {occurrences: 10}}

  Last Friday of every month:
    {frequency: {type: "monthly", on: {"last": "friday"}}, schedule: "regularly", basedOn: "due_date"}

Returns: [{success, id, name, warnings?}]
```

### edit_tasks

```
Edit existing tasks in OmniFocus using patch semantics. Limited to 1 item per call.

Patch contract: omit a field to leave it unchanged, set to null to
clear it, set to a value to update.

Tags (in all tag fields) accept names (case-insensitive) or IDs;
you can mix both. Non-existent names are rejected. Ambiguous names
(case-insensitive collision) return an error.

All date fields require timezone info (ISO 8601 with offset or Z).
Naive datetimes are rejected.

repetitionRule partial updates:
  - Task has no existing rule: all three root fields required
    (frequency, schedule, basedOn) — same as creation.
  - Task has existing rule: omitted root fields are preserved.
  - frequency.type can be omitted (inferred from existing rule)
    unless changing to a different type.
  - Same type: omitted frequency sub-fields preserved.
  - Different type: full replacement with creation defaults.
  - on and onDates are mutually exclusive — setting one clears
    the other.
  - null clears the entire repetition rule.

Examples (repetitionRule):
  Change just the interval (type inferred from existing):
    {frequency: {interval: 5}}

  Add specific days to a weekly task (no type change):
    {frequency: {onDays: ["MO", "WE", "FR"]}}

  Remove day constraint from weekly:
    {frequency: {onDays: null}}

  Switch monthly from dates to weekday pattern (onDates auto-cleared):
    {frequency: {on: {"last": "friday"}}}

  Change from daily to weekly (type required):
    {frequency: {type: "weekly", onDays: ["MO", "FR"]}}

  Clear:
    null

actions.move: exactly one key must be set. ending/beginning with
null moves to inbox.

actions.lifecycle:
  - "complete": marks the task as complete.
  - "drop": skips/cancels the task without completing it.
  On repeating tasks, both actions apply to the current occurrence
  only — the next occurrence is automatically created. Dropping an
  entire repeating sequence is not supported via this API.

actions.tags: replace is standalone. add/remove are combinable with
each other but not with replace.

Returns: [{success, id, name, warnings?}]
```
