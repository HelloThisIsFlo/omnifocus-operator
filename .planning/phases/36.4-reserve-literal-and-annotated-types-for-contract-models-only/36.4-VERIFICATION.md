### Gaps Summary

The phase achieved most of its goals: the four public type aliases are correctly located in contracts/, the JSON Schema constraints (minimum: 1) are advertised on contract fields, the convention is documented in model-taxonomy.md, and the AST enforcement test exists and passes.

Two specific requirements are not met as stated:

**TYPE-02 and TYPE-03** — The plan's must-haves required Frequency.type to be plain `str` and OrdinalWeekday fields to be plain `str | None`. The executor auto-fixed a schema regression (test_output_schema.py guards enum constraints in agent-facing read models) by introducing private `_FrequencyType` and `_DayName` Literal aliases in models/. These private aliases are used directly on class field annotations, which means models/ still contains Literal types on its class fields — just under private names.

This is a genuine deviation from the requirements. The executor's reasoning (preserving JSON Schema enum in agent-facing read output) is valid, but it contradicts TYPE-02/TYPE-03 as written. The resolution options are:

1. **CHOSEN →** Change Frequency.type to plain `str` and OrdinalWeekday fields to `str | None`, accepting that read model JSON Schema loses its enum constraint — and update test_output_schema.py to not assert enum on those fields. The whole point of this phase is that Literal on core models forces `cast`/`type: ignore` at construction sites. Private aliases cause the exact same problem — the private name doesn't change the type system pressure.
2. ~~Accept the private alias approach~~ — REJECTED. Private Literal aliases are semantically identical to public ones; they don't solve the original problem.

The enforcement test must also be strengthened to detect private Literal aliases (not just the name "Literal"), so this pattern can't sneak back in. Delete `_FrequencyType` and `_DayName` entirely from models/.
