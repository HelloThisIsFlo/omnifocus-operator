<decisions>
## Implementation Decisions

### Scope: Full BRIDGE-SPEC alignment
- Fix ALL gaps between current codebase and BRIDGE-SPEC, not just the two todos
- One clean break with the pre-audit codebase

### Bridge JS: Per-entity status resolvers
- Replace broken `p.status.name` / `g.status.name` / `f.status.name` patterns (opaque enums, `.name` returns `undefined`)
- Create three `===` switch resolvers: `ps()` (Project, 4 values), `gs()` (Tag, 3 values), `fs()` (Folder, 2 values)
- Fix `ts()` fallback: change `return null` to `throw new Error("Unknown TaskStatus: " + String(s))`
- All enum resolvers must throw on unknown values — fail-fast at the bridge boundary

### Bridge JS: RepetitionRule resolvers
- Create `rst()` resolver for `Task.RepetitionScheduleType` (3 values: Regularly, FromCompletion, None)
- Create `adk()` resolver for `Task.AnchorDateKey` (3 values: DueDate, DeferDate, PlannedDate)
- Update `rr()` to extract all 4 fields: `ruleString`, `scheduleType` (via `rst()`), `anchorDateKey` (via `adk()`), `catchUpAutomatically`

### Bridge JS: Missing Project fields
- Add `active: p.task.active`, `effectiveActive: p.task.effectiveActive`, `added: d(p.task.added)`, `modified: d(p.task.modified)`
- These 4 fields are undefined on `p.*` — MUST read from `p.task.*`
- Use `p.task.taskStatus` (not `p.taskStatus`) for consistency with BRIDGE-SPEC

### Bridge JS: Tags as TagRef (id + name)
- Change tag serialization from `g.name` to `{id: g.id.primaryKey, name: g.name}`
- Both tasks and projects affected
- **BRIDGE-SPEC override (documented):** BRIDGE-SPEC says tags should be IDs only. We chose to capture both id and name because: (1) names are more ergonomic for agents, (2) we need IDs for future write operations, (3) cheap to capture both. At the MCP presentation layer, expose tag names only for agent readability.

### Bridge JS: URL fields
- Add `url: t.url.toString()` to all entity mappings (tasks, projects, tags, folders)
- Serialize OmniJS URL objects via `.toString()` to get `omnifocus:///task/xxx` deep link strings

### Bridge JS: Tag `childrenAreMutuallyExclusive`
- Add `childrenAreMutuallyExclusive: g.childrenAreMutuallyExclusive` to tag mapping

### Bridge JS: Remove `assignedContainer`
- Remove from task mapping — always null on 2825/2825 tasks per BRIDGE-SPEC
- Keep `shouldUseFloatingTimeZone` — it IS a real property, just happens to always be true in observed data

### Bridge JS: Perspective simplification
- Change `id` mapping from `p.identifier || null` to `p.id ? p.id.primaryKey : null` (fix id/identifier confusion)
- Remove `builtin` from bridge output — derive in Python from `id is None`
- Perspective bridge output: just `id` and `name` (minimal, per "Dumb Bridge, Smart Python")

### Python: Split EntityStatus into per-entity enums
- Replace single `EntityStatus` (Active, Done, Dropped) with three separate enums:
  - `ProjectStatus`: Active, OnHold, Done, Dropped
  - `TagStatus`: Active, OnHold, Dropped
  - `FolderStatus`: Active, Dropped
- `OnHold` was missing entirely — used by both Project and Tag
- Delete `EntityStatus` after migration

### Python: New RepetitionRule enums
- `ScheduleType(StrEnum)`: Regularly, FromCompletion, None
- `AnchorDateKey(StrEnum)`: DueDate, DeferDate, PlannedDate

### Python: RepetitionRule model redesign
- Expand from 2 fields to 4 (all required):
  - `rule_string: str`
  - `schedule_type: ScheduleType` (was `str | None = None`)
  - `anchor_date_key: AnchorDateKey` (new)
  - `catch_up_automatically: bool` (new)
- Remove `# TEMPORARY` comment block

### Python: Make incorrectly-nullable fields required
- **6 timestamp fields** (confirmed always-present by BRIDGE-SPEC):
  - Task: `added`, `modified`
  - Tag: `added`, `modified`
  - Folder: `added`, `modified`
- **3 status fields** (after per-entity resolvers working):
  - Project: `status: ProjectStatus` (was `EntityStatus | None`)
  - Tag: `status: TagStatus` (was `EntityStatus | None`)
  - Folder: `status: FolderStatus` (was `EntityStatus | None`)
- **3 Project review fields** (BRIDGE-SPEC marks all as always-present):
  - `last_review_date: AwareDatetime` (was nullable)
  - `next_review_date: AwareDatetime` (was nullable)
  - `review_interval: ReviewInterval` (was nullable)

### Python: Add missing fields
- **Project**: `active: bool`, `effective_active: bool`, `added: AwareDatetime`, `modified: AwareDatetime`
- **Tag**: `children_are_mutually_exclusive: bool`
- **All entities** (Task, Project, Tag, Folder): `url: str`
- **Task/Project tags**: Change from `list[str]` to `list[TagRef]` where `TagRef` is a small model with `id: str` and `name: str`

### Python: Remove fields
- **Task**: Remove `assigned_container` (always null)

### Python: Perspective model
- Fix `id` semantics (now `p.id.primaryKey`, null for built-in)
- Add `builtin` as a derived field (computed from `id is None`) — not from bridge
- Model stays minimal: `id: str | None`, `name: str`, `builtin: bool` (computed)

### Python: Dual status on Project (kept)
- `status: ProjectStatus` = lifecycle (Active/OnHold/Done/Dropped)
- `task_status: TaskStatus` = computed availability of root task
- Both are required, both are useful, both are in BRIDGE-SPEC

### Model hierarchy
- Claude's discretion — `active`, `effectiveActive`, `added`, `modified` are shared across all 4 entity types. The planner should restructure the base class hierarchy as appropriate.

### Claude's Discretion
- Model hierarchy restructuring (where shared fields live in the inheritance chain)
- Exact test refactoring approach
- Seed data update strategy
- Order of implementation (bridge-first vs Python-first)

<specifics>
## Specific Ideas

- "The BRIDGE-SPEC is extremely clear — make sure to refer to it in context to tell the agent to actually look at the BRIDGE-SPEC"
- "Dumb Bridge, Smart Python" — every derivation that can be done in Python should be done in Python (e.g., `builtin` on Perspective)
- Tags stored as TagRef (id + name) internally, but MCP surface should expose names only for agent readability. The MCP presentation change is a separate concern from this phase — this phase fixes the bridge and models.
- Tag names chosen over IDs for agent ergonomics: "when we interact with this database, it's going to be really okay to add this tag and that tag" — names are more natural for agents
