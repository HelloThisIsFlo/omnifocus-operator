# Phase 08.2: Enforce fail-fast model fields, fix bridge status helpers, and redesign RepetitionRule - Research

**Researched:** 2026-03-06
**Domain:** Bridge JS / Python Pydantic models / enum alignment with empirical BRIDGE-SPEC
**Confidence:** HIGH

## Summary

This phase aligns the entire bridge-and-model stack with the empirically-verified BRIDGE-SPEC. The changes span two codebases (bridge.js in OmniJS/JavaScriptCore and Python Pydantic models) and touch every entity type. The core problems are: (1) broken status enum resolution in bridge.js (`.name` returns `undefined` on opaque OmniFocus enums), (2) a shared `EntityStatus` Python enum that doesn't match OmniFocus's per-entity enum namespaces, (3) an incomplete RepetitionRule model (2/4 fields), (4) 8+ incorrectly nullable fields, and (5) missing fields on Project and Tag.

The scope is well-defined because every change maps directly to a discrepancy between the current codebase and the BRIDGE-SPEC. There is no ambiguity about what to build -- the BRIDGE-SPEC specifies exact access paths, types, required/nullable classifications, and enum values. The risk is purely in execution: many files change simultaneously, test data (conftest factories, seed data, Vitest mocks) must all stay consistent, and the `models/__init__.py` namespace rebuild must include all new types.

**Primary recommendation:** Implement bridge-side changes first (enum resolvers, field additions), then Python-side changes (new enums, model field updates), then update all test infrastructure (conftest factories, Vitest mocks, seed data). This order ensures bridge output is correct before Python models try to parse it.

### Pattern 1: Hierarchy restructuring for shared fields
**What:** `active`, `effectiveActive`, `added`, `modified` are currently on Task only but BRIDGE-SPEC shows they exist on ALL four entity types (Task, Project, Tag, Folder). Moving them into the base hierarchy eliminates duplication.

**Options:**
- **Option A: Move to OmniFocusEntity.** All four entity types inherit these fields. Tag and Folder already have them on their own classes; Task has them; Project needs them added. This is the cleanest approach.
- **Option B: Keep on each entity separately.** Less DRY but avoids changing the base class. Project gets new fields, others stay as-is.

**Recommendation: Option A (move to OmniFocusEntity or create a new intermediate base).** Since all 4 entities have `added`, `modified`, `active`, `effectiveActive`, these are universal entity properties. The `url` field is also universal. Moving them up reduces duplication. However, Project's `added`/`modified` come from `p.task.*` in the bridge -- this is a bridge concern, not a model concern. The Python model just has the fields regardless of where the bridge reads them.

**Specific restructuring:**
- `OmniFocusEntity` gains: `url: str`, `added: AwareDatetime`, `modified: AwareDatetime`, `active: bool`, `effective_active: bool`
- Remove these from Task (currently has `added`, `modified`, `active`, `effective_active`)
- Remove these from Tag (currently has `added`, `modified`, `active`, `effective_active`)
- Remove these from Folder (currently has `added`, `modified`, `active`, `effective_active`)
- Project now inherits them via ActionableEntity -> OmniFocusEntity

**Caveat for Perspective:** Perspective does NOT extend OmniFocusEntity (it has nullable id). Perspective does NOT have `added`, `modified`, `active`, `effectiveActive` in our scope (deferred). This is fine -- Perspective stays on OmniFocusBaseModel.

### Pattern 2: Computed field for Perspective.builtin
**What:** The `builtin` field should be derived from `id is None` in Python, not sent from bridge.
**Implementation:** Use a Pydantic `@model_validator(mode='before')` or `@computed_field`:
```python
from pydantic import computed_field

class Perspective(OmniFocusBaseModel):
    id: str | None
    name: str

    @computed_field  # type: ignore[prop-decorator]
    @property
    def builtin(self) -> bool:
        return self.id is None
```

**Alternative:** Use `model_validator(mode='before')` to strip `builtin` from input and let it be computed. The `@computed_field` approach is cleaner for Pydantic v2.

### Pattern 3: TagRef as a small sub-model
**What:** Tags on Task/Project change from `list[str]` (tag names) to `list[TagRef]` (id + name objects).
```python
class TagRef(OmniFocusBaseModel):
    id: str
    name: str
```
**Bridge output changes:** `tags: t.tags.map(g => ({id: g.id.primaryKey, name: g.name}))` instead of `tags: t.tags.map(g => g.name)`.

### Anti-Patterns to Avoid
- **Shared status enum across entities:** OmniFocus uses isolated enum namespaces per entity type. A single Python `EntityStatus` cannot work -- `ProjectStatus.ON_HOLD` is valid but `FolderStatus.ON_HOLD` is not.
- **Reading `.name` on OmniFocus enum objects:** Always `undefined`. Must use `===` comparison.
- **Silent null fallback on unknown enums:** Use `throw new Error(...)` in bridge, Pydantic `ValidationError` in Python.
- **Reading project fields from `p.*` that only exist on `p.task.*`:** `p.active`, `p.effectiveActive`, `p.added`, `p.modified` are all `undefined`. Must read from `p.task.*`.

## Common Pitfalls

### Pitfall 1: Forgetting to update `models/__init__.py` namespace rebuild
**What goes wrong:** New enum types (`ProjectStatus`, `TagStatus`, `FolderStatus`, `ScheduleType`, `AnchorDateKey`) and new models (`TagRef`) must be added to the `_ns` dict in `__init__.py` and included in the `model_rebuild()` calls, otherwise Pydantic cannot resolve forward references.
**How to avoid:** After adding any new type to `_enums.py` or `_common.py`, immediately update `__init__.py`:
1. Add import
2. Add to `_ns` dict
3. Add to `__all__`
4. Add `model_rebuild()` call if the type needs it
5. Remove `EntityStatus` from all three locations

### Pitfall 2: Conftest factory field counts hardcoded in tests
**What goes wrong:** `test_models.py` asserts exact field counts (e.g., `len(d) == 32` for tasks, `len(Task.model_fields) == 32`). When fields are added/removed, these tests fail.
**How to avoid:** Update the field count assertions in lockstep with factory and model changes. The new counts will be:
- Task: 32 fields -> 31 fields (remove `assignedContainer`, add `url`; net 0, but `added`/`modified`/`active`/`effectiveActive` may move to base)
- Project: 31 fields -> 35+ fields (add `active`, `effectiveActive`, `added`, `modified`, `url`)
- Tag: 9 fields -> 12 fields (add `url`, `childrenAreMutuallyExclusive`, status becomes required)
- Folder: 8 fields -> 9 fields (add `url`, status becomes required)
- Perspective: 3 fields -> stays 3 (id, name, builtin -- but builtin is now computed)

**Important:** The model field count includes inherited fields. When fields move to a base class, the subclass count stays the same (they still have the field, just inherited).

### Pitfall 3: Breaking `make_snapshot_dict()` cascade
**What goes wrong:** `make_snapshot_dict()` calls `make_task_dict()`, `make_project_dict()`, etc. If factory defaults don't include new required fields, every test using `make_snapshot_dict()` fails -- including tests in `test_service.py`, `test_repository.py`, and `test_models.py`.
**How to avoid:** Update ALL factory functions in conftest.py first, ensuring they have correct defaults for every required field. Run `pytest` immediately after to catch cascading failures.

### Pitfall 4: Pydantic `computed_field` serialization for Perspective.builtin
**What goes wrong:** If the bridge still sends `builtin` in its JSON, and Perspective now has `builtin` as a `@computed_field`, Pydantic may raise a validation error on the unexpected field, or silently ignore it, depending on model config.
**How to avoid:** Two options: (a) remove `builtin` from bridge output AND use `@computed_field`, or (b) if backward compatibility is needed for a transition period, use `model_config = ConfigDict(..., extra='ignore')`. The CONTEXT.md says to remove `builtin` from bridge and derive in Python, so option (a) is correct.

### Pitfall 5: Bridge Vitest mock data out of sync
**What goes wrong:** The Vitest test file (`bridge/tests/bridge.test.js`) mocks OmniFocus globals (`flattenedTasks`, `flattenedProjects`, etc.) with specific structures. When the bridge mapping changes (new fields, tag format, removed fields), the mock data must match or tests break.
**How to avoid:** Update Vitest mock data in the same commit as bridge.js changes. Also add new mocks for `Project.Status`, `Tag.Status`, `Folder.Status`, `Task.RepetitionScheduleType`, `Task.AnchorDateKey`.

### Pitfall 6: `StrEnum` member named `NONE` conflicts with Python
**What goes wrong:** `ScheduleType.NONE = "None"` -- `NONE` is a valid Python identifier but `None` is a reserved keyword. The member value `"None"` (string) is fine, but naming the member `NONE` may cause confusion. Python's `StrEnum` handles this correctly -- `NONE` is just an attribute name.
**How to avoid:** Use `NONE = "None"` -- StrEnum is fine with this. The string value `"None"` is what Pydantic validates against. Test explicitly.

### Pitfall 7: Seed data in bridge factory
**What goes wrong:** `src/omnifocus_operator/bridge/_factory.py` has inline seed data for the `inmemory` case. This data uses the OLD format (no `url`, no `added`/`modified` on project, tags as strings). If not updated, `create_bridge("inmemory")` returns data that fails model validation.
**How to avoid:** Update the seed data in `_factory.py` to include all new required fields and new formats (TagRef, url, etc.).

```javascript
// Pattern for all per-entity status resolvers (ps, gs, fs)
// Follow existing ts() pattern but with entity-specific constants and throw on unknown

function ps(s) {
    if (s === Project.Status.Active) return "Active";
    if (s === Project.Status.OnHold) return "OnHold";
    if (s === Project.Status.Done) return "Done";
    if (s === Project.Status.Dropped) return "Dropped";
    throw new Error("Unknown ProjectStatus: " + String(s));
}

function gs(s) {
    if (s === Tag.Status.Active) return "Active";
    if (s === Tag.Status.OnHold) return "OnHold";
    if (s === Tag.Status.Dropped) return "Dropped";
    throw new Error("Unknown TagStatus: " + String(s));
}

function fs(s) {
    if (s === Folder.Status.Active) return "Active";
    if (s === Folder.Status.Dropped) return "Dropped";
    throw new Error("Unknown FolderStatus: " + String(s));
}
```

### Bridge JS: Updated rr() with all 4 fields
```javascript
function rst(s) {
    if (s === Task.RepetitionScheduleType.Regularly) return "Regularly";
    if (s === Task.RepetitionScheduleType.FromCompletion) return "FromCompletion";
    if (s === Task.RepetitionScheduleType.None) return "None";
    throw new Error("Unknown RepetitionScheduleType: " + String(s));
}

function adk(s) {
    if (s === Task.AnchorDateKey.DueDate) return "DueDate";
    if (s === Task.AnchorDateKey.DeferDate) return "DeferDate";
    if (s === Task.AnchorDateKey.PlannedDate) return "PlannedDate";
    throw new Error("Unknown AnchorDateKey: " + String(s));
}

function rr(v) {
    if (!v) return null;
    return {
        ruleString: v.ruleString,
        scheduleType: rst(v.scheduleType),
        anchorDateKey: adk(v.anchorDateKey),
        catchUpAutomatically: v.catchUpAutomatically,
    };
}
```

## Open Questions

1. **Project `taskStatus` access path**
   - What we know: BRIDGE-SPEC says use `p.task.taskStatus` for consistency. Current bridge uses `p.taskStatus` which works (proxied).
   - What's unclear: Whether to change the access path. Both work identically per BRIDGE-SPEC ("proxied").
   - Recommendation: Change to `p.task.taskStatus` for consistency with `p.task.active`, `p.task.effectiveActive`, `p.task.added`, `p.task.modified` -- all the `p.task.*` reads are in one place.

2. **Perspective `@computed_field` serialization behavior**
   - What we know: `@computed_field` includes the field in `model_dump()` output. The bridge will NO LONGER send `builtin`. Existing test `make_perspective_dict()` sends `builtin`.
   - What's unclear: Whether `model_validate()` accepts extra `builtin` field during transition.
   - Recommendation: Remove `builtin` from bridge output and from test factory defaults. Use `@computed_field` on model. With default Pydantic config (no `extra='forbid'`), extra input fields are silently ignored, so transition is safe even if old bridge data is encountered.

3. **`model_fields` count in tests after hierarchy restructuring**
   - What we know: `len(Task.model_fields)` counts all fields including inherited. Moving fields up the hierarchy does not change the count.
   - What's unclear: Whether Pydantic's `model_fields` includes `@computed_field`.
   - Recommendation: Verify with a quick test. `computed_field` properties do appear in `model_computed_fields` (separate dict), NOT in `model_fields`. So Perspective's field count will drop from 3 to 2 in `model_fields`. Tests should check `model_computed_fields` separately for `builtin`.
