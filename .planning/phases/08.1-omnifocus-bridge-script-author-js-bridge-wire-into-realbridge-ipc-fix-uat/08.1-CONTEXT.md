<decisions>
## Implementation Decisions

### Bridge folder structure
- npm project in `bridge/` at project root — devDependencies only (test runner), zero production deps
- Canonical `bridge.js` lives at `src/omnifocus_operator/bridge/bridge.js` (inside the Python package)
- `bridge/` npm project references the canonical file for testing via relative path (`../src/omnifocus_operator/bridge/bridge.js`)
- Python loads `bridge.js` via `importlib.resources` at runtime — PyPI-compatible
- RealBridge reads and caches the script once at `__init__()` time (IPC dir doesn't change during a session)

### JS testing strategy
- Test runner: **Vitest** (one devDependency in `bridge/package.json`)
- Scope: Pure logic only — argument parsing, dispatch routing, response envelope shaping, error handling
- OmniFocus API calls (`flattenedTasks`, `FileWrapper`, etc.) are mocked at the boundary, not deeply simulated
- Script uses **dual-format**: exports internal functions for testability via `module.exports`, production entry point wraps in IIFE
- Unified test command: a top-level coordination mechanism (Makefile or script) runs both `uv run pytest` and `cd bridge && npm test`

### Argument protocol design
- **Template injection**: Python reads `bridge.js`, replaces `__IPC_DIR__` placeholder with the actual IPC directory path, URL-encodes the result as `script=` parameter
- **Argument is a file prefix only**: `arg={pid}_{uuid}` — the script uses this to construct request/response filenames
- URL format: `omnifocus:///omnijs-run?script={encoded_customized_script}&arg={pid}_{uuid}`
- The script reads `{IPC_DIR}/{argument}.request.json`, processes the operation, writes `{IPC_DIR}/{argument}.response.json`
- The `::::` delimiter is removed from the dispatch flow — no longer needed since UUID is in the filename

### Request/response envelope
- **Request file** (clean JSON): `{"operation": "snapshot", "params": {}}` — no more `{"dispatch": "uuid::::op"}` format
- **Response file** (unchanged): `{"success": true, "data": {...}}` or `{"success": false, "error": "message"}`
- This requires updating `_write_request()` in `_real.py` and the simulator's `_handle_request()` to use the new envelope

### Operation naming
- Rename `dump_all` → `snapshot` across the codebase (Python service, repository, bridge, simulator, tests)
- Only one operation for now: `snapshot`. Design for extensibility but only implement snapshot.
- Aligns with the existing `DatabaseSnapshot` model name

### Error handling in JS
- Always write a response file on error: `{"success": false, "error": "message"}` — Python never times out for catchable failures
- Also log errors to OmniFocus developer console (`console.error`) for debugging
- Covers: can't read request file, unknown operation, OmniFocus API exceptions

### File I/O in OmniFocus
- Use `FileWrapper.withContents()` for writing (proven by research script)
- Use `URL.fromPath()` for path construction
- Atomic writes via FileWrapper's built-in atomic option

</decisions>

<specifics>
## Specific Ideas

- The research script (`.research/operatorBridgeScript.js`) shows the OmniFocus API surface: `flattenedTasks`, `flattenedProjects`, `flattenedTags`, `flattenedFolders`, `Perspective.all`, `Task.Status.*`, `FileWrapper`, `URL.fromPath()`
- The simulator (`simulator/__main__.py`) is the Python reference implementation of the IPC protocol — the JS bridge mirrors its behavior
- Script should be an IIFE in production: `(function() { ... })()` — self-contained, no global pollution
- Template placeholder `__IPC_DIR__` must be a string that won't appear in normal JS code

</specifics>
