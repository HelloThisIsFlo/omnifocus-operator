# Phase 16.1: Introduce Actions Grouping to edit_tasks - Context

**Gathered:** 2026-03-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Restructure the `edit_tasks` API to separate idempotent field setters (top-level) from stateful operations (`actions` block). Tags and moveTo graduate into `actions`. Lifecycle slot reserved but not implemented. Bridge.js is untouched — only the Python API layer (models, service, MCP tool, tests) changes.

</domain>

<decisions>
## Implementation Decisions

### No backward compatibility
- This is an alpha with a single user — no external consumers
- New shape IS the shape, as if designed this way from the start
- Old fields removed completely, no migration, no detection of old field names
- All tests rewritten to new shape (old tests would fail)

### Actions block structure
- New `actions` field on TaskEditSpec containing an `ActionsSpec` Pydantic model
- `ActionsSpec` has nested models: `TagActionSpec`, reused `MoveToSpec`, and `lifecycle` string
- All models inherit `OmniFocusBaseModel` with UNSET sentinel pattern

### Tag actions (actions.tags)
- Nested object: `{ add: [...], remove: [...] }` or `{ replace: [...] }`
- Mutual exclusivity: `replace` is standalone, `add` + `remove` are combinable
- Same resolution logic as before (case-insensitive name match, ID fallback)
- Old top-level fields (`tags`, `addTags`, `removeTags`) removed entirely

### Move actions (actions.move)
- Same "key IS the position" pattern, relocated under `actions.move`
- Exactly one key: `beginning`, `ending`, `before`, `after`
- Null value for `beginning`/`ending` means inbox
- Old top-level `moveTo` removed entirely

### Lifecycle slot (actions.lifecycle)
- Reserved in the model as a string field
- Service rejects with clear "not yet implemented (coming in Phase 17)" error
- Documents the intent and validates the shape early

### Top-level setters (unchanged)
- `id` (required), `name`, `flagged`, `note`, `dueDate`, `deferDate`, `plannedDate`, `estimatedMinutes`
- These are pure idempotent setters — no graduation needed

### Warning tiers
- **Setters**: generic no-op warnings ("No changes detected", "Field already has this value — omit to skip")
- **Actions**: domain-specific warnings ("Tag 'X' is already on this task", "Task is already at this position — omit actions.move to skip movement")

### Bridge interface unchanged
- Bridge.js keeps current interface: `tagMode`, `tagIds`, `addTagIds`, `removeTagIds`, position payload
- Python service layer translates from new `ActionsSpec` to existing bridge payload format
- Bridge simplification is a future phase (not in scope)

### Claude's Discretion
- Exact Pydantic model_json_schema override for ActionsSpec/TagActionSpec (same pattern as existing)
- How to structure the service layer translation (inline vs helper methods)
- Test file organization for the rewritten tests
- Exact warning message wording

</decisions>

<specifics>
## Specific Ideas

- The field graduation pattern: any field can "graduate" from top-level setter to action group when it needs more than replacement. Tags are the first graduate. Each graduation is independent.
- Lifecycle reservation means Phase 17 gets a clean slot to implement into — no model changes needed, just service logic
- Bridge simplification noted as future phase — keep translation in Python service for now

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `MoveToSpec` (`models/write.py`): reused directly under `ActionsSpec.move` — same model, just relocated
- `_Unset` sentinel + `_clean_unset_from_schema` (`models/write.py`): reused for all new models
- `OmniFocusBaseModel` with camelCase aliases: all new models inherit this
- `OperatorService._resolve_tags()` / `_resolve_parent()` (`service.py`): reused, just called from different code path

### Established Patterns
- UNSET sentinel for patch semantics (omit = no change, null = clear, value = set)
- `model_json_schema` override to strip `_Unset` from JSON schema
- Service validates before bridge executes
- `model_dump(by_alias=True, exclude_none=True)` for camelCase bridge payloads

### Integration Points
- `models/write.py`: add `TagActionSpec`, `ActionsSpec`; modify `TaskEditSpec` (remove old fields, add `actions`)
- `service.py`: refactor `edit_task` to read from `spec.actions` instead of top-level tag/move fields
- `server.py`: update `edit_tasks` tool docstring to document new shape
- `tests/`: rewrite all edit-related tests to new shape
- `uat/`: update UAT scripts to new shape
- `.claude/skills/test-edit-operations`: verify and update if it hardcodes tool shape

</code_context>
