# Phase 16.1: Introduce Actions Grouping to edit_tasks - Research

**Researched:** 2026-03-09

## Summary

This phase restructures `edit_tasks` to separate idempotent field setters (top-level) from stateful operations (nested `actions` block). Tags and moveTo graduate into `actions`. A lifecycle slot is reserved but rejected at runtime. Bridge.js is untouched -- only Python models, service, server docstring, tests, and UAT skill change.

The scope is well-bounded: all patterns (UNSET sentinel, model_json_schema override, OmniFocusBaseModel inheritance, camelCase aliases) are established in the codebase. No new dependencies, no new infrastructure -- purely restructuring existing code.

**Primary recommendation:** Work in three waves: (1) new models + model tests, (2) service refactor + service tests, (3) server docstring + UAT skill update.

### Service Translation Layer

The service must translate from new `ActionsSpec` to existing bridge payload format:

| New API | Bridge Payload |
|---------|---------------|
| `actions.tags.replace: [...]` | `tagMode: "replace", tagIds: [resolved]` |
| `actions.tags.add: [...] + remove: [...]` | `tagMode: "add_remove", addTagIds: [...], removeTagIds: [...]` |
| `actions.tags.add: [...]` only | `tagMode: "add", tagIds: [resolved]` |
| `actions.tags.remove: [...]` only | `tagMode: "remove", removeTagIds: [resolved]` |
| `actions.move: {ending: id}` | `moveTo: {position: "ending", containerId: id}` |
| `actions.lifecycle: "complete"` | **Rejected** -- ValueError before bridge |

### Mutual Exclusivity for Tags

`TagActionSpec` enforces: `replace` is standalone; `add`/`remove` are combinable.

```python
@model_validator(mode="after")
def _tag_mutual_exclusivity(self) -> TagActionSpec:
    has_replace = not isinstance(self.replace, _Unset)
    has_add = not isinstance(self.add, _Unset)
    has_remove = not isinstance(self.remove, _Unset)
    if has_replace and (has_add or has_remove):
        raise ValueError("Cannot use 'replace' with 'add' or 'remove'")
    if not has_replace and not has_add and not has_remove:
        raise ValueError("tags must specify at least one of: add, remove, replace")
    return self
```

## Common Pitfalls

### Pitfall 1: `_clean_unset_from_schema` doesn't recurse into nested `$defs`
**What goes wrong:** New nested models (`ActionsSpec`, `TagActionSpec`) add new entries to `$defs`. Current cleanup only handles `_Unset` in top-level `$defs` but may miss deeply nested references.
**How to avoid:** After adding new models, test `TaskEditSpec.model_json_schema()` and verify no `_Unset` references remain anywhere in the output. The existing function already cleans nested `$defs` (lines 238-248 of write.py), but verify it handles the new depth.

### Pitfall 2: model_rebuild for new models
**What goes wrong:** `models/__init__.py` calls `model_rebuild(_types_namespace=_ns)` for models using `TYPE_CHECKING` imports. New models need the same treatment.
**How to avoid:** Add `ActionsSpec.model_rebuild()` and `TagActionSpec.model_rebuild()` in `models/__init__.py`. Also add exports.

### Pitfall 3: camelCase alias for `actions` sub-fields
**What goes wrong:** `add`, `remove`, `replace`, `move`, `lifecycle` are already camelCase-safe, but `TagActionSpec` inherits `OmniFocusBaseModel` which auto-generates aliases. Verify `add` doesn't get aliased to something unexpected.
**How to avoid:** Test round-trip: `model_validate({"add": ["x"]})` should work. The `to_camel` alias generator leaves single words unchanged.

### Pitfall 4: Empty actions block
**What goes wrong:** User sends `{"id": "x", "actions": {}}` -- all fields inside are UNSET.
**How to avoid:** Detect this case in service and treat same as "no actions specified." Could be part of the "empty edit" detection (currently checks `len(payload) == 1`).

### Pitfall 5: Test rewrite scope
**What goes wrong:** Underestimating the test blast radius. Every test that uses `tags`, `add_tags`, `remove_tags`, or `move_to` on `TaskEditSpec` must change to `actions.tags.replace`, `actions.tags.add`, `actions.tags.remove`, `actions.move`.
**How to avoid:** The changes touch:
- `tests/test_service.py`: ~60 tests in `TestEditTask` class (lines 441-1424)
- `tests/test_server.py`: ~15 tests in `TestEditTasks` class (lines 893-1238)
- UAT skill: `.claude/skills/test-edit-operations/SKILL.md` -- all tag/move syntax references

### Pitfall 6: Lifecycle rejection timing
**What goes wrong:** If lifecycle validation happens after tag resolution, you waste API calls before rejecting.
**How to avoid:** Check lifecycle field early in `edit_task()`, before any resolution work. Fail fast with clear message.

## Code Examples

### New Model Definitions (write.py)

```python
class TagActionSpec(OmniFocusBaseModel):
    """Tag actions: add/remove (combinable) or replace (standalone)."""
    add: list[str] | _Unset = UNSET
    remove: list[str] | _Unset = UNSET
    replace: list[str] | None | _Unset = UNSET  # None = clear all

    @model_validator(mode="after")
    def _tag_mutual_exclusivity(self) -> TagActionSpec:
        has_replace = not isinstance(self.replace, _Unset)
        has_add = not isinstance(self.add, _Unset)
        has_remove = not isinstance(self.remove, _Unset)
        if has_replace and (has_add or has_remove):
            msg = "Cannot use 'replace' with 'add' or 'remove' -- use either replace mode or add/remove mode"
            raise ValueError(msg)
        if not has_replace and not has_add and not has_remove:
            msg = "tags must specify at least one of: add, remove, replace"
            raise ValueError(msg)
        return self

    @classmethod
    def model_json_schema(cls, ...) -> dict[str, Any]:
        schema = super().model_json_schema(...)
        return _clean_unset_from_schema(schema)


class ActionsSpec(OmniFocusBaseModel):
    """Stateful operations grouped under the actions block."""
    tags: TagActionSpec | _Unset = UNSET
    move: MoveToSpec | _Unset = UNSET
    lifecycle: str | _Unset = UNSET

    @classmethod
    def model_json_schema(cls, ...) -> dict[str, Any]:
        schema = super().model_json_schema(...)
        return _clean_unset_from_schema(schema)
```
