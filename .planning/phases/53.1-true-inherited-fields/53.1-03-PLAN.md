<objective>
Fix _walk_one to compute actual ancestor values for inherited fields instead of just checking ancestor presence.

Purpose: The current walk correctly identifies WHICH fields are truly inherited (strips self-echoes) but keeps OmniFocus's effective value instead of computing the ancestor's actual value. When a task's own dueDate (2026-05-01) is earlier than its project's (2036-03-01), inheritedDueDate shows 2026-05-01 (= own, because OF resolves to min) instead of 2036-03-01 (= the project's actual date). The fix: track the actual ancestor values during the walk and use them in the model_copy update.

Output: Corrected _walk_one that populates inherited fields with computed ancestor values (soonest date for dates, any-True for flags).
</objective>

  <name>Task 2: Fix _walk_one to compute ancestor values (GREEN)</name>
  <files>src/omnifocus_operator/service/domain.py, tests/test_service_domain.py</files>
  <action>
    Modify `_walk_one` in domain.py to track actual ancestor values, not just presence booleans.

    Current approach (BROKEN):
    ```python
    ancestor_sets: dict[str, bool] = {inh: False for _, inh in _INHERITED_FIELD_PAIRS}
    # ... walks ancestors, sets ancestor_sets[inherited] = True/False
    # ... if not ancestor_sets[inherited]: updates[inherited] = None/False
    # (when ancestor_sets[inherited] is True, leaves OF effective value unchanged)
    ```

    New approach (CORRECT):
    ```python
    # Track actual ancestor values, not just booleans
    ancestor_vals: dict[str, object] = {inh: None for _, inh in _INHERITED_FIELD_PAIRS}
    # Special case: inherited_flagged starts as False (not None)
    ancestor_vals["inherited_flagged"] = False
    ```

    In the ancestor task loop and project check:
    - For `flagged`: if ancestor.flagged is True, set `ancestor_vals["inherited_flagged"] = True` (any-True semantics -- once True, stays True)
    - For date fields: collect the ancestor's actual date value. Compute the soonest (minimum) date among all ancestors that have the field set. Track all seen values or update with min logic.

    Date computation strategy (soonest ancestor date):
    - If `ancestor_vals[inherited]` is None and ancestor has the field set: store the ancestor's value
    - If `ancestor_vals[inherited]` is not None and ancestor has the field set: keep the min (soonest) of the two

    After the walk, build updates dict:
    - For each field pair: if `ancestor_vals[inherited]` is the "empty" value (False for flagged, None for dates), that means no ancestor sets it -> strip (same as before)
    - If `ancestor_vals[inherited]` has a real value -> SET `updates[inherited] = ancestor_vals[inherited]` (this is the NEW behavior: replace OF's effective with computed ancestor value)

    Critical: always build the update for inherited fields, whether stripping (no ancestor) or replacing (has ancestor). The update dict should cover ALL 6 inherited fields, not just the ones being stripped. This ensures OF's effective value is always replaced with the computed ancestor value.

    Type note: date field values from ancestors are `AwareDatetime` objects (from Pydantic parsing). They compare with standard Python `<` / `min()`. No string parsing needed.

    Run `uv run pytest tests/test_service_domain.py -q --no-cov -k "TrueInheritance"` -- ALL tests (old and new) should pass (GREEN).
    Then run `uv run pytest tests/ -q --no-cov` to verify full suite.
