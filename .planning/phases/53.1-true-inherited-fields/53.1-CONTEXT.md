# Phase 53.1: True Inherited Fields - Context

**Gathered:** 2026-04-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Strip self-echoed `inherited*` fields from tasks by walking the parent hierarchy, and remove structurally impossible `inherited*` fields from projects via model surgery. After this phase, `inheritedX` on a task means "an ancestor above me set X" — not "OmniFocus echoed my own value back."

Two parts:
- **Part A (tasks):** Service-layer hierarchy walk in `DomainLogic` that checks whether any ancestor (task chain + containing project) sets each inherited field. Self-echoes are set to None/False; server-layer D-03 stripping removes them from output.
- **Part B (projects):** Move inherited fields from `ActionableEntity` to `Task` only. Projects can't meaningfully inherit (folders have no dates/flags), so inherited fields on projects are structurally always self-echoes.

</domain>

<decisions>
## Implementation Decisions

### Walk depth
- **D-01:** Walk includes the containing project as the final ancestor. After exhausting the `parent.task.id` task chain, check `project_map[task.project.id]` for the containing project's direct fields. Projects CAN have dates/flags that tasks inherit — stopping at the project boundary would incorrectly strip inherited fields for tasks under flagged/dated projects.
- Build both `task_map: dict[id, Task]` and `project_map: dict[id, Project]` from the same `get_all()` call. Zero additional cost.

### Stripping mechanism
- **D-02:** Keep `inherited_flagged` as `bool`. `False` = no ancestor flags (stripped by D-03). Date fields use `None` = no ancestor sets date (stripped by D-03). No model type changes.
- Rationale: OmniFocus flagging is an OR — `effectiveFlagged` is True if ANY ancestor flags. A parent having `flagged=False` doesn't "contribute" anything. So `inherited_flagged` is binary: True (some ancestor flagged) or False (nobody did). No meaningful "ancestor explicitly set False" case exists.

### Walk method API
- **D-03:** `compute_true_inheritance(tasks: list[Task]) -> list[Task]` on `DomainLogic`. Method calls `self._repo.get_all()` internally, builds task_map + project_map, walks each task. Callers just pass their tasks. Follows `check_cycle` precedent. Repo cache makes the internal `get_all()` free.

### Service integration
- **D-04:** Walk applies at the service boundary, not the repo level. Internal `get_all()` callers (check_cycle, tag map building, the walk itself) always get raw data.
  - `get_all_data()`: calls `repo.get_all()`, applies walk, returns `raw.model_copy(update={"tasks": walked_tasks})`
  - `get_task()`: calls `resolver.lookup_task()`, applies walk to single-element list, returns walked task
  - `list_tasks()`: pipeline calls repo, applies walk to result items
  - `get_project()`: no walk needed — projects have no inherited fields after Part B

### Field pairs checked (Part A)
- **D-05:** Six field pairs. For each task, check if any ancestor (task chain + project) sets the direct field:
  - `flagged` ↔ `inheritedFlagged` — ancestor has `flagged=True`
  - `dueDate` ↔ `inheritedDueDate` — ancestor has `dueDate` not null
  - `deferDate` ↔ `inheritedDeferDate` — same pattern
  - `plannedDate` ↔ `inheritedPlannedDate` — same pattern
  - `dropDate` ↔ `inheritedDropDate` — same pattern
  - `completionDate` ↔ `inheritedCompletionDate` — same pattern

### Part B cascade scope (8 files)
- **D-06:** Move `inherited_flagged`, `inherited_due_date`, `inherited_defer_date`, `inherited_planned_date`, `inherited_drop_date` from `ActionableEntity` to `Task` (alongside existing `inherited_completion_date`). Full cascade:
  - `models/common.py` — remove 5 inherited fields from `ActionableEntity`
  - `models/task.py` — add 5 inherited fields (move from ActionableEntity)
  - `repository/hybrid/hybrid.py` — `_map_project_row`: remove 5 inherited key assignments
  - `repository/bridge_only/adapter.py` — add `effectiveFlagged`, `effectiveDueDate`, `effectiveDeferDate`, `effectivePlannedDate`, `effectiveDropDate` to `_PROJECT_EXTRA_DEAD_FIELDS` (precedent: `effectiveCompletionDate` already there)
  - `config.py` — `PROJECT_DEFAULT_FIELDS`: remove 4 inherited fields; `PROJECT_FIELD_GROUPS["metadata"]`: remove `inheritedDropDate`
  - `agent_messages/descriptions.py` — update `LIST_PROJECTS_TOOL_DOC`, `GET_PROJECT_TOOL_DOC`: remove inherited field references; rename "entity" → "task" in `_INHERITED_FIELD_DESC`
  - `conftest.py` — `make_model_project_dict`: remove inherited keys
  - `test_adapter.py` / `test_cross_path_equivalence.py` / `test_models.py` — update assertions for project shape

### Description text update
- **D-07:** `_INHERITED_FIELD_DESC` changes from "Inherited from parent hierarchy when not set directly on this entity." to "Inherited from parent hierarchy when not set directly on this task." — since inherited fields are now task-only.

<code_context>
## Existing Code Insights

### Reusable Assets
- `DomainLogic.check_cycle()` — exact precedent for hierarchy walk: calls `get_all()`, builds `task_map`, walks `parent.task.id`. New method follows same pattern but also builds `project_map` and checks the containing project.
- `model_copy(update={...})` — established Pydantic mutation pattern for creating modified immutable instances
- D-03 server stripping — already removes None, False, [], "", "none" from output. Self-echoed inherited fields set to None/False are naturally stripped.

### Established Patterns
- **Method Object pattern** — `_ListTasksPipeline`, `_EditTaskPipeline` etc. The walk call can be added to the pipeline's execute method.
- **Service returns full models, server shapes** — D-09 from Phase 53. The walk modifies models at the service layer; server stripping is downstream.
- **`_PROJECT_EXTRA_DEAD_FIELDS`** — bridge adapter precedent for dropping fields that don't map to the model. `effectiveCompletionDate` is already there for projects.

### Integration Points
- `OperatorService.get_all_data()` — currently a one-liner pass-through to `repo.get_all()`. Becomes: fetch, walk tasks, return model_copy.
- `OperatorService.get_task()` — currently delegates to `resolver.lookup_task()`. Needs walk applied to the returned task.
- `_ListTasksPipeline.execute()` — needs walk applied to result items before returning.
- `ActionableEntity` → `Task` field migration — inherited fields move down the class hierarchy.

</code_context>

<specifics>
## Specific Ideas

- The walk's field-pair mapping should be a constant (list of tuples or dict), not inline if/else chains. Makes it easy to add/remove pairs.
- For `get_task()`, the walk builds the full task_map + project_map just to process one task. This is fine — `get_all()` is cached, and the walk itself is O(depth) per task. Don't optimize for the single-task case.
- The `_INHERITED_FIELD_DESC` text change ("entity" → "task") is a one-line edit but worth its own test assertion (AST enforcement tests likely catch this automatically).

</specifics>
