# Phase 53.1: True Inherited Fields - Research

**Researched:** 2026-04-14
**Domain:** Service-layer hierarchy walk + model refactoring (Python/Pydantic)
**Confidence:** HIGH

## Summary

This phase has two parts: (A) a service-layer method that walks the parent hierarchy to determine whether each `inherited*` field on a task is truly inherited from an ancestor, and (B) a model-level refactoring that moves 5 inherited fields from `ActionableEntity` to `Task`, eliminating structurally impossible inherited fields from projects.

All implementation details are locked by CONTEXT.md decisions D-01 through D-07. The codebase already has exact precedent for every operation needed: `check_cycle()` for hierarchy walks with `get_all()` + task_map, `model_copy(update=...)` for immutable Pydantic mutations, and the server-layer `strip_entity()` already strips None/False values. The research confirms the decisions are sound and surfaces implementation details the planner needs.

**Primary recommendation:** Execute Part B (model surgery) first -- it's mechanical and isolates the project-side cleanup. Then Part A (hierarchy walk) builds on the simplified model.

## Architecture Patterns

### Part A: Hierarchy Walk

The walk method follows the exact `check_cycle()` pattern already in DomainLogic (line 718 of domain.py):

```python
# Precedent: check_cycle
async def check_cycle(self, task_id: str, container_id: str) -> None:
    all_data = await self._repo.get_all()
    task_map = {t.id: t for t in all_data.tasks}
    current = container_id
    while current is not None:
        if current == task_id:
            raise ValueError(CIRCULAR_REFERENCE)
        t = task_map.get(current)
        if t is None:
            break
        if t.parent.task is None:
            break
        current = t.parent.task.id
```

### Anti-Patterns to Avoid
- **Applying walk at repository level:** Internal consumers (`check_cycle`, tag map building, the walk itself) need raw `get_all()` data. Walk is service-boundary only.
- **Heuristic comparison (inherited == direct):** Rejected in discussion. Unreliable for booleans (coincidental equality).
- **Per-task `get_all()` calls:** The walk method calls `get_all()` once internally. Callers pass task lists, not IDs.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Immutable model mutation | Dict manipulation + re-validate | `model_copy(update={...})` | Type-safe, Pydantic-native, established pattern |
| Response stripping | Custom None/False filtering | Existing `strip_entity()` in projection.py | Already strips None, False, empty -- inherited fields set to None/False are automatically removed |
| Task hierarchy walk | Recursive approach | Iterative while-loop (check_cycle precedent) | OmniFocus hierarchies are shallow (2-5 levels), iterative is clearer and avoids stack depth issues |

## Common Pitfalls

### Pitfall 1: Forgetting the project as final ancestor
**What goes wrong:** Walk stops at the first project-parent ParentRef but doesn't check the project's own fields.
**Why it happens:** `task.parent.project` indicates "parent is a project," but the project itself can have `flagged=True` or dates set.
**How to avoid:** After exhausting the task chain (while `parent.task` is not None), always look up the containing project from `project_map[task.project.id]` and check its direct fields.
**Warning signs:** Tasks under flagged projects show no `inheritedFlagged`.

### Pitfall 2: Bridge adapter still renames for projects
**What goes wrong:** `adapt_snapshot()` calls `_rename_inherited_fields(project)` for every project, but projects no longer have those fields.
**Why it happens:** The rename step processes both tasks and projects; after Part B, projects don't have the target fields.
**How to avoid:** Add the `effective*` keys to `_PROJECT_EXTRA_DEAD_FIELDS` so they're removed before renaming runs. The rename is a no-op on missing keys anyway, but dead-field removal is cleaner. [VERIFIED: `_rename_inherited_fields` uses `if old_key in raw` -- safe on missing keys, but confusing to keep renaming fields that don't map to the model]
**Warning signs:** Pydantic validation error on Project construction with unexpected fields.

### Pitfall 3: Confusing `task.project` vs `task.parent.project`
**What goes wrong:** Walk uses `task.parent.project.id` to look up the project, but this is the direct parent when the parent IS a project. `task.project` is the containing project at any depth.
**Why it happens:** Two different references to "project" on a task.
**How to avoid:** For the final ancestor check, use `task.project.id` (containing project at any depth). `task.parent.project.id` would miss intermediate project parents in edge cases.
**Warning signs:** Deep subtasks don't see inheritance from their containing project.

### Pitfall 4: Inbox tasks have no real project
**What goes wrong:** Walk tries to look up `$inbox` in project_map.
**Why it happens:** Inbox tasks have `project.id = "$inbox"` which is a synthetic system location.
**How to avoid:** Check `project_map.get(task.project.id)` -- returns None for inbox, walk terminates naturally. No special-casing needed.
**Warning signs:** KeyError on `$inbox` during project lookup.

### Pitfall 5: Field group config drift
**What goes wrong:** After removing inherited fields from `PROJECT_DEFAULT_FIELDS`, the `LIST_PROJECTS_TOOL_DOC` description still mentions them.
**Why it happens:** Description text is maintained separately from config.py field sets.
**How to avoid:** Update all three locations: config.py field sets, descriptions.py tool docs, and descriptions.py reusable fragments.
**Warning signs:** Agent sees inherited fields listed in tool description but they never appear in output.

### Pitfall 6: Test fixture projects still have inherited fields
**What goes wrong:** `make_model_project_dict()` returns inherited fields that no longer exist on the model, causing Pydantic validation to either ignore them (extra="ignore") or fail.
**Why it happens:** Test factories aren't updated alongside model changes.
**How to avoid:** Part B must update `conftest.py` `make_model_project_dict()` to remove `inheritedFlagged`, `inheritedDueDate`, `inheritedDeferDate`, `inheritedPlannedDate`, `inheritedDropDate`.
**Warning signs:** Test failures in test_models.py, test_adapter.py, test_output_schema.py.

## Code Examples

### Walk one task (Part A core logic)
```python
# Source: derived from check_cycle pattern + D-01 through D-05
_INHERITED_FIELD_PAIRS: tuple[tuple[str, str], ...] = (
    ("flagged", "inherited_flagged"),
    ("due_date", "inherited_due_date"),
    ("defer_date", "inherited_defer_date"),
    ("planned_date", "inherited_planned_date"),
    ("drop_date", "inherited_drop_date"),
    ("completion_date", "inherited_completion_date"),
)

def _walk_one(
    self,
    task: Task,
    task_map: dict[str, Task],
    project_map: dict[str, Project],
) -> Task:
    # Determine which fields have an ancestor that sets them
    ancestor_sets: dict[str, bool] = {inh: False for _, inh in _INHERITED_FIELD_PAIRS}

    # Walk task chain
    current_id: str | None = task.parent.task.id if task.parent.task else None
    while current_id is not None:
        ancestor = task_map.get(current_id)
        if ancestor is None:
            break
        for direct, inherited in _INHERITED_FIELD_PAIRS:
            if not ancestor_sets[inherited]:
                val = getattr(ancestor, direct)
                if direct == "flagged":
                    ancestor_sets[inherited] = val is True
                else:
                    ancestor_sets[inherited] = val is not None
        current_id = ancestor.parent.task.id if ancestor.parent.task else None

    # Check containing project as final ancestor (D-01)
    project = project_map.get(task.project.id)
    if project is not None:
        for direct, inherited in _INHERITED_FIELD_PAIRS:
            if not ancestor_sets[inherited] and hasattr(project, direct):
                val = getattr(project, direct)
                if direct == "flagged":
                    ancestor_sets[inherited] = val is True
                else:
                    ancestor_sets[inherited] = val is not None

    # Build update dict for fields where no ancestor sets the value
    updates: dict[str, object] = {}
    for _, inherited in _INHERITED_FIELD_PAIRS:
        if not ancestor_sets[inherited]:
            if inherited == "inherited_flagged":
                updates[inherited] = False
            else:
                updates[inherited] = None

    return task.model_copy(update=updates) if updates else task
```
