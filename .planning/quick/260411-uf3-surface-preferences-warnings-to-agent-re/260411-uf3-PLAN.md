<objective>
Wire `OmniFocusPreferences.get_warnings()` into all three pipeline families so
preferences warnings (settings fallback, unknown DueSoon pair) flow through to
agent-visible result objects instead of only being logged.

Purpose: Agents currently get `warnings: null` when OmniFocus preferences fail to
load or produce unusual values. The warnings exist in `preferences._warnings` but
`get_warnings()` is never called from any pipeline. This is a dead integration path.

Output: Three pipelines drain preferences warnings; 4 existing tests in
`tests/test_preferences_warnings_surfacing.py` pass (currently all fail).
</objective>

Three pipelines call preferences methods but never drain warnings. Fix each:

**1. `_ReadPipeline._resolve_date_filters()` (line ~330)**
After `await self._preferences.get_due_soon_setting()` (or unconditionally after
the conditional block), drain preferences warnings and extend `self._warnings`:
```python
self._warnings.extend(await self._preferences.get_warnings())
```
Place this AFTER the `due_soon_setting` call but BEFORE (or alongside) the
`self._warnings.extend(self._date_result.warnings)` line at ~338. Since
`get_warnings()` calls `_ensure_loaded()` internally, it is safe to call even
when the `due="soon"` branch was not taken — but the tests only trigger on
`due="soon"`, so placing it after the conditional block is fine either way. The
key constraint: it must be called AFTER any preferences method that could
produce warnings, and the result must land in `self._warnings` before
`_result_from_repo` packages them.

**2. `_AddTaskPipeline._normalize_dates()` (line ~549)**
After the `for` loop that calls `get_default_time`, drain preferences warnings.
Store them alongside `self._repetition_warnings` (or in a new
`self._preferences_warnings` list). Then merge them in `_delegate()` at line
~639. The simplest approach: extend `self._repetition_warnings` directly after
`_normalize_dates()` completes — the name is slightly misleading but avoids
adding a new field for a 1-line change. Alternatively, add
`self._preferences_warnings: list[str] = []` in `execute()` next to
`self._repetition_warnings`, populate after `_normalize_dates()`, and include
in the `warnings` merge in `_delegate()`. Use whichever is cleanest.

**3. `_EditTaskPipeline._normalize_dates()` (line ~689)**
Same pattern as AddTask. Drain warnings after the `for` loop. The final merge
point is `_build_payload()` at line ~943 where `self._all_warnings` is
assembled. Either extend one of the existing warning lists
(`self._lifecycle_warns` is wrong semantically; better to add a
`self._preferences_warns` field) or extend `self._all_warnings` concatenation.

For all three: `await self._preferences.get_warnings()` is the call. It returns
`list[str]` and is idempotent (returns a copy of accumulated warnings).
