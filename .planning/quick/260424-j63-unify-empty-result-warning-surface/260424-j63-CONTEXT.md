# Quick Task 260424-j63: Unify empty-result warning surface — Context

**Gathered:** 2026-04-24
**Status:** Ready for planning

<domain>
## Task Boundary

Implement the design in `.planning/todos/pending/2026-04-24-unify-empty-result-warning-surface.md` verbatim. Two-layer model:

- **Layer 1 — unified empty-result warning**: fires when `items == []` AND at least one query field `is_non_default`. Text parameterized by active-filter count (1 vs 2+).
- **Layer 2 — did-you-mean**: unchanged in role, fires on top when an unresolved name has fuzzy candidates. Needs a small standalone reword since `FILTER_NO_MATCH` is retiring.

**Retired warnings**: `EMPTY_SCOPE_INTERSECTION_WARNING`, `FILTER_NO_MATCH`.

**Scope guard (Flo's steer)**: "change should be pretty minimal". Do not expand beyond the todo's sketch. No new helpers beyond `_active_filter_names(query)`. No refactors of adjacent warnings or domain methods. Service layer only.

</domain>

<decisions>
## Implementation Decisions

### Filter-name ordering in warning text
**Alphabetical** (ascending, stable `sorted()` over the active filter names).

Rationale: the agent doesn't semantically parse warning order, so pedagogical grouping has no payoff. Alphabetical is deterministic, requires zero new ordering constants, and won't drift when new fields are added. Flo: "Who cares? The agent is going to read this, so the ordering doesn't really matter."

Lock with a test that enforces the alphabetical invariant across a mixed set of active filters.

### Field name surface in warning text
**camelCase JSON-schema alias names** — exactly what the agent sent in its MCP tool input.

Examples:
- `in_inbox` → `"inInbox"`
- `estimated_minutes_max` → `"estimatedMinutesMax"`
- `project`, `parent`, `flagged` → unchanged (no case shift)

Mechanism: read each `ListTasksQuery.model_fields[name].alias` (or fall back to the raw field name when no alias is declared). The MCP tool input schema already generates camelCase via these aliases — reusing them guarantees the warning echoes back the exact key the agent typed, with zero drift risk.

### `$inbox` zero-result behavior
**No special-case branches.** The uniform rule is: `is_non_default(query, field) → counts as an active filter`. `project="$inbox"` is non-default on `project`, so if results are empty, the warning fires — naturally, without any `if value == "$inbox"` branch.

Flo's directive: "whatever has the least amount of special case, we should really have zero special case. It's just like filter, no filter, no weird introspection." Whatever the uniform predicate says is what happens; we do not introspect sentinels.

### Emit site
Single emit point in `_ListTasksPipeline.execute()` — **both** after the short-circuit return AND after `_delegate()`, OR before both via a common tail path. The natural place is:

1. Build `active_filters = _active_filter_names(self._query)` after resolution.
2. On both the short-circuit path (service.py:430-440) and the post-`_delegate` path, before returning: if `result.items == [] and active_filters`, append `EMPTY_RESULT_WARNING.format(filters=...)`.

The existing `_emit_empty_intersection_warning_if_applicable` helper (service.py:642-660) is deleted outright; its behavior is subsumed.

### DYM reword (Claude's discretion)
Keep DYM minimal. Drop the redundant "No {entity_type} found matching '{value}'." prefix (now handled by the unified empty-result warning). Proposed new text:

```
FILTER_DID_YOU_MEAN = "Did you mean: {suggestions}? (no {entity_type} matched '{value}')"
```

The parenthetical keeps `value` attached to the suggestion without re-stating the empty outcome. Minimal diff from the current constant.

### Warning text templates
Locked from the todo, verbatim:

- 1 filter: `"The 'X' filter resolved to zero tasks. No results."`
- 2+ filters: `"The combination of filters 'X', 'Y', 'Z' resolved to zero tasks. No results."`

Quote characters: single quotes around each filter name (matches existing style in `LIST_TASKS_INBOX_PROJECT_WARNING`). Comma-separated, Oxford comma irrelevant for N≥3 since adjacent commas and "and" are omitted (no conjunction between the last two — simpler, consistent with existing warnings).

### Zero-filter case
Skip entirely. Per todo + Flo: "almost impossible to get zero results without any filter — not worried about the edge case." No warning when `active_filters == []`.

### Claude's Discretion
- Exact helper name (`_active_filter_names` per todo sketch, or `_active_filter_aliases` — whichever reads better in context). Default: use todo's name.
- Placement of `_active_filter_names` — free-function module-level vs pipeline method. Default: pipeline method (`self._active_filter_names()`) for cohesion with other `_Pipeline` helpers.
- Test organization: one new test class `TestEmptyResultWarning` in `tests/test_list_pipelines.py` covering the 8-case matrix. Retire `TestEmptyScopeShortCircuit` and `TestFilterNoMatchWarningText` in the same patch.

</decisions>

<specifics>
## Specific Ideas

### Exact retirements
- `EMPTY_SCOPE_INTERSECTION_WARNING` constant → deleted from `warnings.py:204`
- `FILTER_NO_MATCH` constant → deleted from `warnings.py:160`
- `_emit_empty_intersection_warning_if_applicable` method → deleted from `service.py:642-660`
- `check_filter_resolution` no-match-no-suggestions branch → deleted from `domain.py:593` (falls through to empty list; unified warning handles the surface)

### Exact additions
- `EMPTY_RESULT_WARNING` constant in `warnings.py` with `{filters}` placeholder
- `_active_filter_names` helper (free fn or method) iterating `ListTasksQuery.model_fields` and filtering by `is_non_default`
- Single emit point in `execute()` covering both the short-circuit and delegated paths

### Test matrix (≈8 new tests)
1. scope-empty (project resolves to empty project) → single-filter warning
2. intersection-empty (project ∩ parent disjoint) → 2-filter warning
3. pruning-empty (flagged=true + no flagged tasks) → single-filter warning
4. no-match name (project="NonExistent") → single-filter warning + DYM if suggestions
5. combos (3+ active filters, empty result) → N-filter warning with alphabetical names
6. default-vs-non-default (availability=[REMAINING] only, empty result) → NO warning (defaults don't count)
7. zero-filters-skip (no filters, empty result — synthesized) → NO warning
8. DYM standalone (name no-match with fuzzy candidates) → both warnings fire

### Alphabetical ordering lock
One dedicated test asserts that with filters `{parent, flagged, inInbox, due}` active, the warning text lists them as `'due', 'flagged', 'inInbox', 'parent'`.
