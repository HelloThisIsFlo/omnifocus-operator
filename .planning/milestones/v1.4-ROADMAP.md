## Phase Details

### Phase 53: Response Shaping

**Goal**: Agents receive clean, minimal responses by default — null/empty/false fields stripped, inherited field names aligned with OmniFocus semantics, field selection available on list tools, server refactored into a package

**Depends on**: v1.3.3 (ordering infrastructure)

**Requirements**: STRIP-01, STRIP-02, STRIP-03, RENAME-01, FSEL-01 through FSEL-13, COUNT-01

**Success Criteria** (what must be TRUE):
  1. Entity responses strip null, `[]`, `""`, `false`, `"none"` from all fields automatically
  2. `availability` and envelope fields (`hasMore`, `total`, `status`) are never stripped
  3. `effective*` fields renamed to `inherited*` across all 6 pairs in all responses
  4. `include` parameter on `list_tasks`/`list_projects` adds semantic field groups
  5. `only` parameter allows individual field selection (id always included)
  6. `include` + `only` mutually exclusive — providing both returns validation error
  7. `include: ["*"]` returns all fields
  8. `limit: 0` returns count-only response `{items: [], total: N, hasMore: <bool>}`
  9. `server.py` → `server/` package with `__init__.py`, `handlers.py`, `lifespan.py`, `projection.py`
  10. Field group definitions centralized in `config.py`; projection server-only concern

**Plans:** 5/5 plans complete

### Phase 53.1: True Inherited Fields (INSERTED)

**Goal**: `inherited*` fields on tasks reflect genuine inheritance from ancestor hierarchy, not OmniFocus self-echo values; each field uses correct aggregation semantics

**Depends on**: Phase 53 (inherited field names, server/ package)

**Requirements**: INHERIT-01 through INHERIT-10

**Success Criteria** (what must be TRUE):
  1. `inherited*` fields only appear when truly inherited from an ancestor — self-echoes stripped
  2. Projects structurally cannot have `inherited*` fields (moved to Task model only)
  3. `inheritedDueDate` uses **min** across all ancestors (tightest deadline wins)
  4. `inheritedDeferDate` uses **max** across all ancestors (latest block wins)
  5. `inheritedPlannedDate`, `inheritedDropDate`, `inheritedCompletionDate` use **first-found**
  6. `inheritedFlagged` uses **any-True** (boolean OR across ancestor chain)
  7. Walk applies to `get_all`, `get_task`, and `list_tasks`

**Plans:** 4/4 plans complete

- [x] 53.1-02-PLAN.md — Service hierarchy walk: _walk_one presence-check implementation
- [x] 53.1-03-PLAN.md — Ancestor value computation: ancestor_vals dict, soonest/any-True semantics
- [x] 53.1-04-PLAN.md — Per-field aggregation: min/max/first-found/any-True strategy constants

### Phase 54: Batch Processing

**Goal**: `add_tasks` and `edit_tasks` accept arrays of up to 50 items with per-item result reporting; add uses best-effort, edit uses fail-fast

**Depends on**: Phase 53 (response shaping infrastructure, server/ package)

**Requirements**: BATCH-01 through BATCH-09, BATCH-10 (added via Quick 260417-oiw)

**Success Criteria** (what must be TRUE):
  1. Both tools accept 1–50 items (Pydantic `maxItems`, 51+ rejected at schema level)
  2. `add_tasks` processes all items regardless of failures (best-effort)
  3. `edit_tasks` stops at first error; remaining items get `status: "skipped"` (fail-fast)
  4. Response is flat array: `{status, id?, name?, error?, warnings?}` per item
  5. `status: "success" | "error" | "skipped"` — `name` on success only, `id` on success + edit errors/skips
  6. `warnings` array available on all status types
  7. Items processed serially in array order
  8. Same-task edits allowed — each sees prior item's committed result

**Plans:** 3/3 plans complete

Plans:
- [x] 54-01-PLAN.md — AddTaskResult/EditTaskResult model migration (status Literal)
- [x] 54-02-PLAN.md — Batch handler loops (best-effort add, fail-fast edit) + tool descriptions
- [x] 54-03-PLAN.md — 26 MCP-level batch tests covering all BATCH requirements

### Phase 55: Notes Graduation

**Goal**: `edit_tasks` graduates note editing to an `actions.note` block with explicit `append`/`replace` semantics; top-level `note` removed from edit

**Depends on**: Phase 54 (batch processing infrastructure)

**Requirements**: NOTE-01 through NOTE-05

**Success Criteria** (what must be TRUE):
  1. Top-level `note` removed from `edit_tasks` input schema (NOTE-01)
  2. `actions.note.append`: adds text with `\n` separator; `""` or whitespace-only is N1 no-op
  3. `actions.note.replace`: sets/clears note content; identical content is N2 no-op; `null` or `""` clears
  4. Append on empty/whitespace-only note sets directly (no leading separator)
  5. `add_tasks` retains top-level `note` for initial content

**Plans:** 3/3 plans complete

Plans:

## Milestone Summary

- Universal stripping via `_is_strip_value` helper (unhashable list/dict safety) — dict-based pure transform on `model_dump(by_alias=True)`
- Field group definitions centralized in `config.py` (TASK_DEFAULT_FIELDS, PROJECT_DEFAULT_FIELDS, TASK_FIELD_GROUPS, PROJECT_FIELD_GROUPS)
- `ancestor_vals` dict tracks actual computed values (dates/bools) not presence booleans; strategy constants as frozensets for O(1) lookup

- Batch result stripping asymmetry — entity stripping was applied to list/get responses but not batch write results
