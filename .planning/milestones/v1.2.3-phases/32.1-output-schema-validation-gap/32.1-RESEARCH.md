**Researched:** 2026-03-28

## Summary

This phase adds systemic test safeguards for three concerns: (1) serialized tool output validates against MCP `outputSchema`, (2) no model can silently break schema via `@model_serializer`, and (3) naming conventions from `docs/architecture.md` are programmatically enforced. All technical pieces are well-understood -- the codebase already has a basic `TestSerializationSchema` to migrate and expand, `jsonschema` 4.26.0 is available transitively, and the FastMCP serialization + schema generation pipeline has been fully traced.

Key finding: FastMCP applies `compress_schema(prune_titles=True)` to the raw Pydantic schema before advertising it as `outputSchema`. Tests **must** apply the same compression to match what MCP clients actually validate against. FastMCP serializes return values via `pydantic_core.to_jsonable_python()` which respects alias generators (camelCase). For `list[...]` return types (add_tasks, edit_tasks), FastMCP wraps them in `{"result": [...]}` with `x-fastmcp-wrap-result: true`.

**Primary recommendation:** Use `TypeAdapter(...).json_schema(mode="serialization")` + `compress_schema(prune_titles=True)` for schema generation, `pydantic_core.to_jsonable_python()` for serialization, and `jsonschema.validate()` for validation -- this exactly reproduces the FastMCP pipeline.

### Pattern 1: Schema-vs-Data Validation (per tool)
**What:** For each MCP tool, generate the outputSchema the same way FastMCP does, serialize realistic fixture data the same way FastMCP does, then validate with jsonschema.
**When to use:** Every tool's return type.
**Key insight:** Must reproduce FastMCP's exact pipeline:
1. `TypeAdapter(return_type).json_schema(mode="serialization")` -- raw schema
2. For non-object root types (lists): wrap in `_WrappedResult[return_type]`
3. `compress_schema(schema, prune_titles=True)` -- what clients see
4. `pydantic_core.to_jsonable_python(model_instance)` -- serialized data
5. For wrapped types: `{"result": serialized_list}` -- structured_content shape
6. `jsonschema.validate(data, schema)` -- client-side validation

**Example:**
```python
# Source: verified by tracing fastmcp/tools/function_parsing.py + tool.py
import pydantic_core
from pydantic import TypeAdapter
from fastmcp.utilities.json_schema import compress_schema
import jsonschema

def get_output_schema(return_type: type) -> dict:
    """Reproduce FastMCP's outputSchema generation."""
    schema = TypeAdapter(return_type).json_schema(mode="serialization")
    return compress_schema(schema, prune_titles=True)

def serialize_like_fastmcp(value: Any) -> Any:
    """Reproduce FastMCP's serialization path."""
    return pydantic_core.to_jsonable_python(value)

# For get_task:
schema = get_output_schema(Task)
data = serialize_like_fastmcp(task_instance)
jsonschema.validate(data, schema)

# For add_tasks (list return, wrapped):
from fastmcp.tools.function_parsing import _WrappedResult
wrapped_type = _WrappedResult[list[AddTaskResult]]
schema = TypeAdapter(wrapped_type).json_schema(mode="serialization")
schema["x-fastmcp-wrap-result"] = True
schema = compress_schema(schema, prune_titles=True)
data = {"result": serialize_like_fastmcp(result_list)}
jsonschema.validate(data, schema)
```

### Pattern 2: Union Branch Regression Guard
**What:** Extract each branch from discriminated/plain unions in serialization schema and assert it has `properties` (not just `{"type": "object"}`).
**When to use:** `Frequency` (8 branches, discriminated), `EndCondition` (2 branches, anyOf).

```python
# Source: existing TestSerializationSchema pattern
from pydantic import TypeAdapter

def assert_union_branches_not_erased(union_type, expected_count: int):
    schema = TypeAdapter(union_type).json_schema(mode="serialization")
    defs = schema.get("$defs", {})
    assert len(defs) == expected_count
    for name, branch in defs.items():
        assert "properties" in branch, (
            f"${name} lost its properties -- likely erased by @model_serializer"
        )
```

### Pattern 3: Package Class Scanning for Naming Convention
**What:** `pkgutil.walk_packages()` + `inspect.getmembers()` to discover all Pydantic model classes, then check naming rules.
**When to use:** Naming convention enforcement.

```python
import pkgutil, inspect, importlib
from pydantic import BaseModel

WRITE_SUFFIXES = ("Command", "Result", "RepoPayload", "RepoResult", "Action", "Spec")

def scan_package_models(package_name: str) -> list[tuple[str, type]]:
    """Yield (class_name, class) for all Pydantic models in package."""
    pkg = importlib.import_module(package_name)
    results = []
    for _importer, modname, _ispkg in pkgutil.walk_packages(
        pkg.__path__, prefix=f"{package_name}."
    ):
        mod = importlib.import_module(modname)
        for name, obj in inspect.getmembers(mod, inspect.isclass):
            if (issubclass(obj, BaseModel) and obj is not BaseModel
                    and obj.__module__ == modname):
                results.append((name, obj))
    return results
```

### Anti-Patterns to Avoid
- **Using `model_dump(mode='json', by_alias=True)` instead of `pydantic_core.to_jsonable_python()`**: These produce different output for datetime fields. FastMCP uses `to_jsonable_python`, so tests must match.
- **Validating against raw Pydantic schema without compress_schema**: The raw schema has `title` fields that `compress_schema` strips. While this doesn't affect validation correctness, it's better to test against the exact schema clients see.
- **Testing only Frequency union, ignoring EndCondition**: `EndCondition` is also a union (`EndByDate | EndByOccurrences`) and needs the same regression guard.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON Schema validation | Custom dict-walking validator | `jsonschema.validate()` | Exact same library MCP clients use; handles `$ref`, `anyOf`, `discriminator` correctly |
| Schema generation | Manual schema dict construction | `TypeAdapter.json_schema(mode="serialization")` | Pydantic's schema is authoritative for what FastMCP advertises |
| Schema compression | Manual title stripping | `fastmcp.utilities.json_schema.compress_schema` | Exactly reproduces FastMCP's pipeline |
| Model class discovery | Hardcoded class lists | `pkgutil.walk_packages()` + `inspect.getmembers()` | Auto-discovers new classes, no maintenance burden |

## Common Pitfalls

### Pitfall 1: FastMCP Wraps Non-Object Return Types
**What goes wrong:** `add_tasks` returns `list[AddTaskResult]` and `edit_tasks` returns `list[EditTaskResult]`. FastMCP wraps these in `_WrappedResult` because MCP requires `type: "object"` at schema root. The actual structured_content is `{"result": [...]}`, not the bare list.
**Why it happens:** MCP spec requires outputSchema to be an object type.
**How to avoid:** For list-returning tools, use `_WrappedResult[list[T]]` for schema and wrap serialized data in `{"result": data}`.
**Warning signs:** `jsonschema.validate` fails with "is not of type 'object'" at the root level.

### Pitfall 2: to_jsonable_python vs model_dump for Datetime Serialization
**What goes wrong:** `model_dump(mode='json')` serializes datetimes as ISO strings. `pydantic_core.to_jsonable_python()` also does this, but they may differ in edge cases (timezone formatting).
**Why it happens:** Different serialization paths.
**How to avoid:** Use `pydantic_core.to_jsonable_python()` consistently, matching FastMCP.
**Warning signs:** Schema validation passes but serialized dates don't match format expectations.

### Pitfall 3: EndCondition is Not a Discriminated Union
**What goes wrong:** `EndCondition = EndByDate | EndByOccurrences` uses `anyOf` in the schema (not `oneOf` with discriminator). Regression guard must handle both patterns.
**Why it happens:** EndCondition has no `type` discriminator field -- Pydantic uses `anyOf`.
**How to avoid:** The regression guard should check `$defs` entries for both `anyOf` and discriminated union patterns.
**Warning signs:** Test assumes all unions use `discriminator` field.

### Pitfall 4: Private/Base Classes in Naming Scan
**What goes wrong:** `_FrequencyBase` starts with underscore (private), `OmniFocusBaseModel`/`OmniFocusEntity`/`ActionableEntity` are base classes not meant to have suffixes. `CommandModel` and `EditTaskActions` in contracts don't end with recognized suffixes.
**Why it happens:** Naming conventions apply to public domain models, not infrastructure.
**How to avoid:** Exempt list must include: `CommandModel`, `EditTaskActions` (contracts-side), and filter out private classes (leading `_`) and abstract base classes (models-side).
**Warning signs:** False positives from base classes and infrastructure models.

### Pitfall 5: compress_schema Mutates In Place
**What goes wrong:** `compress_schema` calls `_single_pass_optimize` which pops keys from dicts in place. If you call it on a cached schema, subsequent calls see the already-mutated version.
**Why it happens:** The implementation uses `node.pop()` directly on dict objects.
**How to avoid:** Always work with a fresh copy: `compress_schema(copy.deepcopy(schema), ...)` or generate fresh schema per test.
**Warning signs:** Tests pass individually but fail when run together; schema missing expected keys.

## Open Questions

1. **Schema compression deepcopy concern**
   - What we know: `compress_schema` mutates dicts in place via `node.pop()`
   - What's unclear: Whether generating fresh schemas per test via `TypeAdapter.json_schema()` avoids this (likely yes -- Pydantic generates new dicts each call)
   - Recommendation: Verify in implementation; if schema caching is desired, use `copy.deepcopy()` before compress

2. **Exact FastMCP _WrappedResult import path stability**
   - What we know: `_WrappedResult` is a private type in `fastmcp.tools.function_parsing`
   - What's unclear: Whether this is stable across FastMCP versions
   - Recommendation: Use it for now (it's the only way to exactly match FastMCP's behavior). Alternative: construct the wrapper schema manually (`{"properties": {"result": ...}, "required": ["result"], "type": "object"}`)
