# Phase 32.1: Output Schema Validation Gap - Context

**Gathered:** 2026-03-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Add systemic test safeguards ensuring: (1) serialized tool output validates against the MCP outputSchema that FastMCP advertises, (2) no model can silently break schema via `@model_serializer` or similar patterns, (3) model naming conventions from `docs/architecture.md` are programmatically enforced. Also add CLAUDE.md rules so future agents follow conventions proactively.

**Precondition:** The `@model_serializer` fix on `_FrequencyBase` is already landed (commit `db4bcb0`). A basic `TestSerializationSchema` exists in `test_models.py` — this phase expands it into comprehensive coverage and migrates it to a dedicated file.

**Not in scope:** Fixing serialization bugs (already done), the `cWQ63CD2pjo` impossible-state task (existing fail-fast handles it correctly per D-06 from Phase 32).

</domain>

<decisions>
## Implementation Decisions

### Test Organization
- **D-01:** New test file `test_output_schema.py` for all schema validation and naming convention tests. Migrate existing `TestSerializationSchema` from `test_models.py` into this file for consistency.

### Schema-vs-Data Validation
- **D-02:** For all 6 tools (`get_all`, `get_task`, `get_project`, `get_tag`, `add_tasks`, `edit_tasks`), generate the `outputSchema` (serialization mode via `TypeAdapter.json_schema(mode="serialization")`), serialize realistic fixture data, and validate using the `jsonschema` library — not Pydantic. This is what MCP clients do.
- **D-03:** Test fixtures must include tasks with `repetitionRule` set to actual `Frequency` values — at minimum `DailyFrequency`, `WeeklyFrequency` (with `on_days`), `MonthlyDayOfWeekFrequency` (with `on`), and `MonthlyDayInMonthFrequency` (with `on_dates`). Current fixtures all use `repetitionRule: None`.
- **D-04:** Include at least one task with `EndByDate` and one with `EndByOccurrences`.

### Regression Guard
- **D-05:** Standalone test generating the serialization schema for every union type in tool outputs (`Frequency`, `EndCondition`) and asserting each branch is NOT just `{"type": "object", "additionalProperties": true}`. Catches future `@model_serializer` additions on any model.

### Naming Convention Enforcement
- **D-06:** Programmatic test scanning all Pydantic model classes:
  - `models/` package: no write-side suffixes (`Command`, `Result`, `RepoPayload`, `RepoResult`, `Action`, `Spec`)
  - `contracts/` package: all public models use a recognized write-side suffix (with explicit exemption list for base classes like `CommandModel`)
  - Error messages reference `docs/architecture.md` naming taxonomy

### Documentation
- **D-07:** Add rules to `CLAUDE.md` only (not architecture.md — taxonomy is already there):
  - Read `docs/architecture.md` naming taxonomy before creating any new Pydantic model
  - Run schema validation tests after modifying any model that appears in tool output
  - Reference the output schema test file as the enforcing test

### Dependency
- **D-08:** Add `jsonschema` as explicit dev dependency in `pyproject.toml`. Currently available transitively via fastmcp/mcp chain, but transitive deps can vanish on upgrades.

### Claude's Discretion
- Fixture construction strategy (InMemoryBridge population vs inline Pydantic models)
- Exact structure of schema-vs-data test (parametrized per tool vs per model)
- How to extract outputSchema from FastMCP tool registration (may need to introspect the server)
- `EXEMPT` list contents for naming convention test
- Exact wording of CLAUDE.md rules

</decisions>

<specifics>
## Specific Ideas

- The root cause was `@model_serializer` returning `dict[str, Any]` which erased all property/constraint info from the serialization JSON Schema. `@field_serializer` returning `None` also erases. The fix was to remove custom serialization entirely.
- D-08 (interval=1 omission) was dropped — interval=1 now always appears in output. This is simpler and avoids any schema-erasing serializers.
- The naming violation that motivated the convention test: agent created `FrequencySpec` (write-side suffix) for a read-side model, and had parser return `dict[str, Any]` instead of Pydantic models. Both caught manually, fixed in commit `4104a73`.
- User wants the programmatic naming check despite potential fragility — the architecture doc's naming taxonomy is mechanical enough to encode.

</specifics>
