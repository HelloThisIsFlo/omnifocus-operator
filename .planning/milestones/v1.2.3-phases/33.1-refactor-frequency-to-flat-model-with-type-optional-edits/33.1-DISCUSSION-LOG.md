## Validation Location (philosophical discussion)

| Option | Description | Selected |
|--------|-------------|----------|
| Model validates own invariants, service handles cross-model | @model_validator, @field_validator, Field() on models | ✓ |
| All in validate.py | Current pattern, single location | |
| Split by nature | Some model, some service | |

**User's choice:** Model validates own invariants
**Notes:** Extended philosophical discussion. Key insight: the discriminated union WAS model-level validation (structural). Flattening removed it. The cleanest replacement is explicit enforcement at the same level (@model_validator), not moving it to a different layer. "Parse, don't validate" — once you have a Frequency instance, it's guaranteed valid. Audit showed the scope is small: Frequency validation (being rewritten anyway), task name (trivial), EndByOccurrences (one Field change). Included in Phase 33.1 scope.

## Type Collapse: 9 → 6 Types

| Option | Description | Selected |
|--------|-------------|----------|
| 6 types (collapse sub-types) | weekly_on_days → weekly + optional on_days, etc. | ✓ |
| 9 types (keep sub-types in flat model) | Original todo proposal | |

**User's choice:** 6 types
**Notes:** Breakthrough discovery during discussion. The sub-types existed because the discriminated union needed separate classes for different field shapes. In the flat model, they're just base types with optional specialization fields. Collapsing eliminates: (1) the normalization problem entirely, (2) false type boundaries (weekly ↔ weekly_on_days), (3) three unnecessary type strings. Edit ergonomics improve dramatically — adding/removing days on weekly is a field edit, not a type change.
