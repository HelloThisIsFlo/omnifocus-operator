# Phase 33.1: Refactor Frequency to Flat Model with Type-Optional Edits - Research

**Researched:** 2026-03-29
**Domain:** Pydantic model refactoring, internal codebase restructuring
**Confidence:** HIGH

## Summary

This is a cross-cutting internal refactor replacing a 9-subtype Pydantic discriminated union with a single flat `Frequency` model (6 types), adding proper `CommandModel`-based write specs (`FrequencyAddSpec`, `FrequencyEditSpec`), migrating validation from `validate.py` to Pydantic model validators, and enabling type-optional edits. The architecture doc has already been updated for the 6-type flat model -- the code needs to catch up.

All decisions are locked in CONTEXT.md (D-01 through D-18). The codebase is well-understood -- this is a Python-only refactor with no new dependencies, no external services, and no runtime state concerns. The 1025-test suite passes at 98% coverage, providing a strong safety net.

**Primary recommendation:** Wave the refactor through in layers: (1) flat models + validation migration, (2) parser/builder + service merge logic, (3) server descriptions + output schema tests. Each layer is independently testable.

## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Collapse 9 sub-types to 6 types. `weekly_on_days` -> `weekly` + optional `on_days`. `monthly_day_of_week` -> `monthly` + optional `on`. `monthly_day_in_month` -> `monthly` + optional `on_dates`.
- **D-02:** `type` field uses `Literal["minutely", "hourly", "daily", "weekly", "monthly", "yearly"]` on all three models.
- **D-03:** Field names: `on_days`, `on`, `on_dates`.
- **D-04:** `Frequency` (read, `OmniFocusBaseModel`): `type` required (no default), `interval: int = 1`, optional `on_days`/`on`/`on_dates` with `None` defaults.
- **D-05:** `FrequencyAddSpec` (write-add, `CommandModel`): Same field shapes as `Frequency`. `extra="forbid"`.
- **D-06:** `FrequencyEditSpec` (write-edit, `CommandModel`): Uses `Patch`/`PatchOrClear` pattern. `type: Patch[Literal[...]] = UNSET`, etc.
- **D-07:** Hard error for cross-type fields (e.g., `on_days` sent with type `daily`).
- **D-07a:** Cross-type validators on `Frequency` and `FrequencyAddSpec` only -- NOT on `FrequencyEditSpec`.
- **D-08:** `on` and `on_dates` mutually exclusive on `monthly`. Edit auto-clears the other. Add path = hard error if both set.
- **D-09:** Type omitted on edit = same-type merge. Type change = explicit `type` + full replacement. Edit merge flow: (1) create `FrequencyEditSpec`, (2) infer/use type + merge fields, (3) construct `Frequency` -> `@model_validator` fires.
- **D-10:** Adding/removing specialization fields on same base type is NOT a type change.
- **D-11:** Merge uses `is_set()` TypeGuard, not `model_fields_set`.
- **D-12:** `@field_serializer('frequency')` on `RepetitionRule` calls `freq.model_dump(exclude_defaults=True, by_alias=True)`. Omits `interval=1`; `type` (no default) always included.
- **D-13:** All pure-input validation moves to Pydantic model validators. `validate.py` shrinks or is removed.
- **D-14/D-15:** Verbatim tool description templates for add_tasks and edit_tasks (in CONTEXT.md).
- **D-16:** Architecture doc already updated in-session.
- **D-17:** Empty specialization fields normalize to `None` + warn. Domain layer.
- **D-18:** `build_rrule()` accepts both `Frequency` and `FrequencyAddSpec`. Claude's discretion on mechanism.

### Key Pattern: Edit Merge Flow (D-09)

```
Agent input -> FrequencyEditSpec (patch container, no validation)
           -> Service infers type (existing or explicit)
           -> Service merges fields using is_set()
           -> Construct Frequency from merged result
           -> @model_validator fires on Frequency -> catches cross-type violations
```

### Anti-Patterns to Avoid

- **Don't put cross-type validators on FrequencyEditSpec** (D-07a). It's a patch container -- validation fires when constructing the merged Frequency.
- **Don't use model_fields_set for merge** (D-11). Use `is_set()` TypeGuard -- consistent with rest of codebase.
- **Don't use @model_serializer on Frequency** (Phase 32.1 lesson). Use `@field_serializer` on RepetitionRule.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Day code normalization | Custom normalize loop | `@field_validator` on `Frequency` | Pydantic runs it automatically, removes service-layer code |
| Interval >= 1 check | Custom `if interval < 1` | `Field(ge=1)` | Pydantic handles constraint + error message |
| End occurrences >= 1 | Custom `if occurrences < 1` | `Field(ge=1)` on `EndByOccurrences` | Same pattern |
| Cross-type field rejection | Service-layer if/elif tree | `@model_validator(mode="after")` on `Frequency` | Fires on both add and edit-merge paths |

## Common Pitfalls

### Pitfall 1: union_tag_invalid Error Handling Goes Stale

**What goes wrong:** `_format_validation_errors` in `server.py` has special handling for `union_tag_invalid` errors from the discriminated union. After replacing the union with a flat `Literal` type, invalid types produce `literal_error` instead.
**Why it happens:** The error type changed but the handler code wasn't updated.
**How to avoid:** Update `_format_validation_errors` to handle `literal_error` on frequency type fields. Remove the `union_tag_invalid` handling for frequency (it no longer applies).
**Warning signs:** Invalid frequency types produce raw Pydantic error messages instead of the educational `REPETITION_INVALID_FREQUENCY_TYPE` message.

### Pitfall 2: @field_serializer and exclude_defaults Interaction

**What goes wrong:** `type` must NOT have a default on `Frequency` (read model) -- if it has a default, `exclude_defaults=True` would omit it from serialization output.
**Why it happens:** Easy to copy-paste from `FrequencyAddSpec` or `FrequencyEditSpec` where type might have defaults.
**How to avoid:** D-04 is explicit: `type` is required with no default on the read model.
**Warning signs:** `test_output_schema.py` validations fail because `type` is missing from serialized output.

### Pitfall 3: Merge Logic Must Use is_set() Not model_fields_set

**What goes wrong:** `model_fields_set` doesn't distinguish UNSET from explicit None. On `FrequencyEditSpec`, `on_days: PatchOrClear[list[str]] = UNSET` -- setting it to `None` means "clear", but `model_fields_set` treats UNSET the same as explicit None.
**Why it happens:** `model_fields_set` is the obvious Pydantic API, but it doesn't understand the `UNSET` sentinel.
**How to avoid:** D-11 requires `is_set()` TypeGuard. The existing `_merge_same_type_frequency` already uses `model_fields_set` -- it must be rewritten.
**Warning signs:** Clearing a specialization field (`{onDays: null}`) either doesn't clear or throws.

### Pitfall 4: Monthly Auto-Clear Must Happen in Domain Layer

**What goes wrong:** D-08 auto-clear logic (setting `on` auto-clears `on_dates` and vice versa) must happen AFTER the merge, in the domain layer, not in the model validator.
**Why it happens:** On the edit path, the merge combines edit input with existing state. If auto-clear is in the validator, it fires too early (before merge) or doesn't have existing state context.
**How to avoid:** Auto-clear logic in `domain.py`, same level as `normalize_empty_on_dates`. The model validator on `Frequency` only raises errors (D-07) -- it doesn't auto-clear.
**Warning signs:** Edit that sets `on` on a task with existing `on_dates` either errors or doesn't clear `on_dates`.

### Pitfall 5: Output Schema Test Assertions Assume Discriminated Union Structure

**What goes wrong:** `test_frequency_branches_have_properties_and_const` asserts 9 `$defs` branches with `const` type discriminators. The flat model has no `$defs` branches -- it's a single object with `type` as an `enum` (Literal).
**Why it happens:** The test was written for the discriminated union structure.
**How to avoid:** Rewrite the test to validate the flat model schema: single object with `type` as `enum`, optional fields present.
**Warning signs:** Test fails immediately with "Expected 9 $defs branches, got 0".

### Pitfall 6: REPETITION_EMPTY_ON_DATES Warning Message References Old Type Names

**What goes wrong:** Warning string says "monthly_day_in_month" -- but that type no longer exists.
**Why it happens:** The warning was written for the 9-type system.
**How to avoid:** Update warning to use new terminology. It should reference "monthly with onDates" style.
**Warning signs:** Agent sees a warning referencing a type that doesn't exist in the tool schema.

## Code Examples

### Flat Frequency Model (D-04)

```python
from typing import Literal, Self
from pydantic import Field, field_validator, model_validator
from omnifocus_operator.models.base import OmniFocusBaseModel

FrequencyType = Literal["minutely", "hourly", "daily", "weekly", "monthly", "yearly"]

class Frequency(OmniFocusBaseModel):
    type: FrequencyType                    # required, no default
    interval: int = Field(default=1, ge=1)
    on_days: list[str] | None = None
    on: dict[str, str] | None = None
    on_dates: list[int] | None = None

    @model_validator(mode="after")
    def _validate_cross_type(self) -> Self:
        # D-07: cross-type field rejection
        if self.on_days is not None and self.type != "weekly":
            raise ValueError(f"on_days is not valid for type '{self.type}'.")
        if self.on is not None and self.type != "monthly":
            raise ValueError(f"on is not valid for type '{self.type}'.")
        if self.on_dates is not None and self.type != "monthly":
            raise ValueError(f"on_dates is not valid for type '{self.type}'.")
        # D-08: mutual exclusion (add path -- edit path auto-clears in domain)
        if self.on is not None and self.on_dates is not None:
            raise ValueError("on and on_dates are mutually exclusive on monthly.")
        return self
```

### Merge Logic (D-09, D-11)

```python
def _merge_frequency(
    self,
    edit_spec: FrequencyEditSpec,
    existing: Frequency,
) -> Frequency:
    """Merge edit spec with existing frequency using is_set() TypeGuard."""
    merged: dict[str, Any] = {}

    # Type: use edit type if set, otherwise existing
    merged["type"] = edit_spec.type if is_set(edit_spec.type) else existing.type

    # Interval: use edit if set, otherwise existing
    merged["interval"] = edit_spec.interval if is_set(edit_spec.interval) else existing.interval

    # Specialization fields: UNSET=preserve, None=clear, value=set
    for field_name in ("on_days", "on", "on_dates"):
        edit_val = getattr(edit_spec, field_name)
        if is_set(edit_val):
            merged[field_name] = edit_val  # None (clear) or value (set)
        else:
            merged[field_name] = getattr(existing, field_name)  # preserve

    # Construct Frequency -> @model_validator fires -> catches cross-type violations
    return Frequency.model_validate(merged)
```

   - Options: (a) Keep `validate.py` with just task name functions, (b) Move task name validators to `@field_validator` on `AddTaskCommand`/`EditTaskCommand`, (c) Delete validate.py entirely
   - Recommendation: Claude's discretion -- keep `validate.py` lean (option a) to avoid a sprawling refactor scope.
