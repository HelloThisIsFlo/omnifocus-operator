  <behavior>
    ## MtimeSource Protocol and Implementations

    - MtimeSource: Protocol with single method `async get_mtime_ns() -> int`
    - FileMtimeSource: Production impl, constructor takes `path: str`, calls `asyncio.to_thread(os.stat, path)` and returns `stat_result.st_mtime_ns`
    - FakeMtimeSource: Test impl in test file (not production code), constructor takes `mtime_ns: int = 0`, has `set_mtime_ns(value: int)` to control values

    ## OmniFocusRepository

    Constructor: `__init__(self, bridge: Bridge, mtime_source: MtimeSource) -> None`
    - Stores bridge, mtime_source
    - Creates `asyncio.Lock()`
    - Initializes `_snapshot: DatabaseSnapshot | None = None`
    - Initializes `_last_mtime_ns: int = 0`

    ### get_snapshot() -> DatabaseSnapshot
    SNAP-01: First call (snapshot is None) -> calls bridge.send_command("dump_all") -> parses via DatabaseSnapshot.model_validate() -> caches and returns snapshot
    SNAP-02: Second call with same mtime -> returns cached snapshot, bridge.call_count stays at 1
    SNAP-03: Call with unchanged mtime -> returns cached snapshot (verified: same object identity via `is`)
    SNAP-04: Call after mtime changes -> calls bridge again -> replaces cached snapshot with new one -> returns new snapshot
    SNAP-05: 10 concurrent asyncio.gather calls -> only 1 bridge.call_count (lock prevents duplicate dumps)

    ### initialize() -> None
    SNAP-06: Calls get_snapshot() to pre-warm cache -> after initialize(), get_snapshot() returns cached data without another bridge call

    ### Error propagation (per user decision: fail-fast, no stale fallback)
    - BridgeError from bridge.send_command -> propagates raw to caller (NOT wrapped)
    - ValidationError from DatabaseSnapshot.model_validate -> propagates raw to caller
    - Exception from mtime_source.get_mtime_ns -> propagates raw to caller
    - After failed refresh, cached snapshot is NOT updated (stays None or old value)
    - After initialize() failure, subsequent get_snapshot() tries again (no broken state)

    ### Concurrency edge cases
    - Concurrent reads during refresh: all wait on lock, first refreshes, others see fresh data and return it (mtime matches after refresh)
    - Concurrent reads when cache is warm and mtime unchanged: lock acquired briefly, mtime check fast, all return quickly

    ## FileMtimeSource integration (lightweight)
    - Create a temp directory, get its mtime_ns, verify it returns a positive integer
    - Modify the directory (add a file), verify mtime_ns changes
