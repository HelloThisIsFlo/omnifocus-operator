## Summary

This phase authors the OmniFocus-side JavaScript bridge script, wires it into RealBridge's IPC flow via the `omnijs-run` URL scheme, updates the IPC protocol from the `::::` dispatch format to clean JSON envelopes with file-prefix arguments, renames `dump_all` to `snapshot` across the codebase, and fixes the UAT script for end-to-end validation.

The research script at `.research/operatorBridgeScript.js` is the proven reference implementation for OmniFocus API access (5.6KB, ~170 lines). The canonical `bridge.js` will be a restructured version using the new IPC protocol: reading `{IPC_DIR}/{argument}.request.json`, dispatching on `operation`, and writing `{IPC_DIR}/{argument}.response.json`. Python injects the IPC directory path via `__IPC_DIR__` template replacement before URL-encoding the script into the `script=` parameter.

Vitest (single devDependency) handles JS-side unit testing of pure logic. The `bridge/` npm project at project root references the canonical file at `src/omnifocus_operator/bridge/bridge.js` via relative path. Hatchling automatically includes `.js` files within the Python package directory in wheel builds -- no `pyproject.toml` changes needed. Python loads the script via `importlib.resources.files()`.

**Primary recommendation:** Build the JS bridge script as a testable module with dual-format (exports for Vitest, IIFE wrapper for production), wire `_trigger_omnifocus()` to use `script=` + `arg=` URL parameters, update request envelopes from `{"dispatch": "uuid::::op"}` to `{"operation": "...", "params": {}}`, rename `dump_all` to `snapshot` globally, and fix UAT.

## User Constraints (from CONTEXT.md)

### Locked Decisions

- **Bridge folder structure**: npm project in `bridge/` at project root -- devDependencies only (test runner), zero production deps. Canonical `bridge.js` lives at `src/omnifocus_operator/bridge/bridge.js` (inside the Python package). `bridge/` npm project references the canonical file for testing via relative path (`../src/omnifocus_operator/bridge/bridge.js`). Python loads `bridge.js` via `importlib.resources` at runtime -- PyPI-compatible. RealBridge reads and caches the script once at `__init__()` time (IPC dir doesn't change during a session).

- **JS testing strategy**: Test runner: **Vitest** (one devDependency in `bridge/package.json`). Scope: Pure logic only -- argument parsing, dispatch routing, response envelope shaping, error handling. OmniFocus API calls (`flattenedTasks`, `FileWrapper`, etc.) are mocked at the boundary, not deeply simulated. Script uses **dual-format**: exports internal functions for testability via `module.exports`, production entry point wraps in IIFE. Unified test command: a top-level `Makefile` runs both `uv run pytest` and `cd bridge && npm test`.

- **Argument protocol design**: **Template injection**: Python reads `bridge.js`, replaces `__IPC_DIR__` placeholder with the actual IPC directory path, URL-encodes the result as `script=` parameter. **Argument is a file prefix only**: `arg={pid}_{uuid}` -- the script uses this to construct request/response filenames. URL format: `omnifocus:///omnijs-run?script={encoded_customized_script}&arg={pid}_{uuid}`. The script reads `{IPC_DIR}/{argument}.request.json`, processes the operation, writes `{IPC_DIR}/{argument}.response.json`. The `::::` delimiter is removed from the dispatch flow. **OmniFocus runtime contract**: The `arg=` URL parameter is exposed as the global variable `argument` inside the executed script (Omni Automation convention). `_trigger_omnifocus()` method signature changes -- no longer receives `dispatch: str`, instead takes the file prefix and encoded script. `SimulatorBridge._trigger_omnifocus()` override must be updated to match.

- **Request/response envelope**: **Request file** (clean JSON): `{"operation": "snapshot", "params": {}}` -- no more `{"dispatch": "uuid::::op"}` format. **Response file** (unchanged): `{"success": true, "data": {...}}` or `{"success": false, "error": "message"}`. `send_command()`'s existing `params` argument (currently unused) will be forwarded into the request envelope. This requires updating `_write_request()` in `_real.py` and the simulator's `_handle_request()` to use the new envelope.

- **Operation naming**: Rename `dump_all` to `snapshot` across the codebase (~60+ references in: repository, service, bridge, simulator, tests, UAT). Planning docs retain the historical name. Only one operation for now: `snapshot`. Design for extensibility but only implement snapshot. Aligns with the existing `DatabaseSnapshot` model name. `InMemoryBridge` ignores the operation string -- no changes needed there.

- **Error handling in JS**: Always write a response file on error: `{"success": false, "error": "message"}` -- Python never times out for catchable failures. Also log errors to OmniFocus developer console (`console.error`) for debugging. Covers: can't read request file, unknown operation, OmniFocus API exceptions.

- **File I/O in OmniFocus**: Use `FileWrapper.withContents()` for writing (proven by research script). Use `URL.fromPath()` for path construction. Atomic writes via FileWrapper's built-in atomic option.

### Pattern 1: Template Injection for IPC Directory
**What:** Python reads `bridge.js`, does string replacement of `__IPC_DIR__` with the actual IPC path, then URL-encodes the result.
**When to use:** At `RealBridge.__init__()` time -- script is read once and cached for the session.
**Example:**
```python
# Source: CONTEXT.md decisions + importlib.resources docs
from importlib.resources import files

raw_script = (
    files("omnifocus_operator.bridge")
    .joinpath("bridge.js")
    .read_text(encoding="utf-8")
)
self._script = raw_script.replace("__IPC_DIR__", str(self._ipc_dir))
```

### Pattern 2: URL Scheme Trigger with script= and arg=
**What:** The `_trigger_omnifocus()` method constructs a URL with the full script inline and the file prefix as argument.
**When to use:** Every `send_command()` call on RealBridge.
**Example:**
```python
# Source: Omni Automation docs (https://omni-automation.com/script-url/)
encoded_script = urllib.parse.quote(self._script, safe="")
file_prefix = f"{self._pid}_{request_id}"
encoded_arg = urllib.parse.quote(file_prefix, safe="")
url = f"omnifocus:///omnijs-run?script={encoded_script}&arg={encoded_arg}"
subprocess.run(["open", "-g", url], check=True, capture_output=True)
```

### Pattern 3: Dual-Format JS Module (IIFE + exports)
**What:** The bridge script exports internal functions via `module.exports` for testing. In OmniFocus runtime (where `module` is undefined), the script runs as an IIFE.
**When to use:** Always -- this is the structure of `bridge.js`.
**Example:**
```javascript
// bridge.js
var __IPC_DIR__ = "__IPC_DIR__";  // Replaced by Python at load time

function readRequest(ipcDir, filePrefix) {
    // ... read {ipcDir}/{filePrefix}.request.json
}

function writeResponse(ipcDir, filePrefix, responseObj) {
    // ... write {ipcDir}/{filePrefix}.response.json
}

function handleSnapshot() {
    // ... collect OmniFocus data
}

function dispatch(ipcDir, filePrefix) {
    // ... read request, route operation, write response
}

// Production entry: IIFE that runs in OmniFocus
if (typeof module === "undefined") {
    (function() {
        try {
            dispatch(__IPC_DIR__, argument);
        } catch (e) {
            // Write error response
            writeResponse(__IPC_DIR__, argument, {
                success: false,
                error: e.message
            });
            console.error("Bridge error: " + e.message);
        }
    })();
}

// Test entry: export functions for Vitest
if (typeof module !== "undefined") {
    module.exports = {
        readRequest: readRequest,
        writeResponse: writeResponse,
        handleSnapshot: handleSnapshot,
        dispatch: dispatch
    };
}
```

### Pattern 4: Clean Request Envelope
**What:** Request files use `{"operation": "snapshot", "params": {}}` instead of `{"dispatch": "uuid::::op"}`.
**When to use:** All request file writes in `_write_request()`.
**Example:**
```python
# _real.py: _write_request()
content = json.dumps({"operation": operation, "params": params or {}})
```

### Anti-Patterns to Avoid
- **Passing IPC dir in arg=**: Makes the arg parameter complex and requires parsing in JS. Template injection is simpler.
- **Deep OmniFocus API mocking in tests**: Don't simulate `flattenedTasks` behavior -- mock at the I/O boundary (file read/write) and test dispatch logic only.
- **Using `module.exports = { ... }` at top level without guard**: OmniFocus runtime doesn't have `module` -- must guard behind `typeof module !== "undefined"`.
- **Using ES module syntax (`export`, `import`)**: OmniFocus Omni Automation runtime uses a legacy JS engine without ES module support. Use `var`, `function`, and CommonJS `module.exports`.
- **Reading/writing files via Node.js `fs` in tests**: Vitest tests should mock the OmniFocus-specific `FileWrapper`/`URL` APIs. Use `vi.fn()` for mocks.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| URL encoding | Custom percent-encoding | `urllib.parse.quote(script, safe="")` | Edge cases with Unicode, reserved chars, etc. |
| Package resource loading | `Path(__file__).parent / "bridge.js"` | `importlib.resources.files()` | Works with zip imports, namespace packages, PyPI wheels |
| Atomic file writes in OmniFocus | Manual temp-file-rename in JS | `FileWrapper.write(..., [FileWrapper.WritingOptions.Atomic], null)` | OmniFocus provides this built-in |
| JS test framework | Custom test harness | Vitest with `vi.fn()` / `vi.mock()` | Full-featured mocking, assertion library, coverage |

**Key insight:** The OmniFocus JavaScript runtime is NOT Node.js. It lacks `require()`, `fs`, `path`, `process`, etc. All file I/O must use OmniFocus's `FileWrapper`/`URL`/`Data` classes. This makes testing pure logic (separated from I/O) essential.

## Common Pitfalls

### Pitfall 1: OmniFocus JS Runtime Is Not Node.js
**What goes wrong:** Writing JS code that uses `require()`, `fs.readFileSync()`, `console.log()`, `process.env`, or ES module syntax.
**Why it happens:** Developer muscle memory from Node.js. OmniFocus uses Omni Automation's JavaScriptCore-based runtime.
**How to avoid:** Use only OmniFocus API globals (`flattenedTasks`, `FileWrapper`, `URL`, `Data`, `argument`, `console.error`, `console.log`). No `require`, no `import/export`, no Node.js builtins.
**Warning signs:** `ReferenceError` in OmniFocus console for `require`, `module`, `process`.

### Pitfall 2: `__IPC_DIR__` Placeholder Collision
**What goes wrong:** The placeholder string appears in normal JS code or comments, causing unintended replacements.
**Why it happens:** Using a placeholder that's too generic.
**How to avoid:** Use double-underscore prefix/suffix (`__IPC_DIR__`) which won't appear in normal JS. Define it as a `var` assignment so the replacement is clean: `var __IPC_DIR__ = "__IPC_DIR__"` becomes `var __IPC_DIR__ = "/actual/path"`.
**Warning signs:** Script behavior differs between test and production.

### Pitfall 3: URL Length with Encoded Script
**What goes wrong:** The URL-encoded script is too large for the macOS URL scheme handler.
**Why it happens:** The bridge script is ~5.6KB raw, ~17KB URL-encoded. macOS command line limit is ~262,144 characters.
**How to avoid:** Keep bridge.js lean. The current research script is well within limits. Monitor script size if adding operations.
**Warning signs:** `subprocess.CalledProcessError` from `open -g`, or OmniFocus silently not executing.

### Pitfall 4: SimulatorBridge Signature Mismatch
**What goes wrong:** `SimulatorBridge._trigger_omnifocus()` has a different signature than `RealBridge._trigger_omnifocus()` after the change.
**Why it happens:** The override currently takes `dispatch: str` but the parent will change to take `file_prefix: str`.
**How to avoid:** Update both signatures simultaneously. The simulator's override is a no-op, so the parameter name doesn't matter for behavior -- but the signature must match for type safety.
**Warning signs:** mypy errors on override signature.

### Pitfall 5: Forgetting to Update Simulator's Request Parsing
**What goes wrong:** The mock simulator (`simulator/__main__.py`) still expects `{"dispatch": "uuid::::op"}` format after the envelope change.
**Why it happens:** The simulator is a separate process that reads request files independently.
**How to avoid:** Update `_handle_request()` in `simulator/__main__.py` to parse `{"operation": "...", "params": {}}` instead of `{"dispatch": "..."}`.
**Warning signs:** Simulator integration tests fail with KeyError on `"dispatch"`.

### Pitfall 6: Missing `params` Forwarding
**What goes wrong:** `send_command("snapshot", params={"key": "val"})` doesn't forward params to the request file.
**Why it happens:** Current `_write_request()` only writes `dispatch` -- params forwarding is new.
**How to avoid:** Update `_write_request()` signature to accept `operation` and `params`, write `{"operation": operation, "params": params or {}}`.
**Warning signs:** Tests that verify request file content find missing `params` key.

### Pitfall 7: `dump_all` Rename Incomplete
**What goes wrong:** Some code paths still use `"dump_all"` after the rename to `"snapshot"`.
**Why it happens:** 60+ references across many files. Easy to miss one.
**How to avoid:** Use project-wide search for `dump_all` after rename. Automated tests should catch most missed spots since operation strings are asserted in tests.
**Warning signs:** `BridgeProtocolError: Unknown operation: dump_all` at runtime.

### Updated send_command() Flow
```python
# Source: CONTEXT.md decisions + existing _real.py pattern
async def send_command(self, operation: str, params: dict | None = None) -> dict:
    request_id = uuid.uuid4()
    file_prefix = f"{self._pid}_{request_id}"

    await self._write_request(request_id, operation, params)
    self._trigger_omnifocus(file_prefix)

    try:
        raw = await asyncio.wait_for(
            self._wait_response(request_id),
            timeout=self._timeout,
        )
    except TimeoutError:
        await self._cleanup_request(request_id)
        raise BridgeTimeoutError(...)

    result = self._validate_response(raw, operation)
    await self._cleanup_files(request_id)
    return result
```

### Updated _write_request() Envelope
```python
# Source: CONTEXT.md decisions
async def _write_request(
    self, request_id: uuid.UUID, operation: str, params: dict | None = None
) -> None:
    filename = f"{self._pid}_{request_id}.request.json"
    final_path = self._ipc_dir / filename
    tmp_path = self._ipc_dir / f"{filename}.tmp"
    content = json.dumps({"operation": operation, "params": params or {}})

    def _write() -> None:
        tmp_path.write_text(content, encoding="utf-8")
        os.replace(tmp_path, final_path)

    await asyncio.to_thread(_write)
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `{"dispatch": "uuid::::op"}` envelope | `{"operation": "...", "params": {}}` envelope | Phase 08.1 | Cleaner protocol, extensible for write operations, no `::::` delimiter parsing |
| `arg=<dispatch>` (dispatch string as URL arg) | `script=<encoded_script>&arg=<file_prefix>` | Phase 08.1 | Script is inline, arg is just the file prefix. No need for OmniFocus to have the script pre-installed. |
| `dump_all` operation name | `snapshot` operation name | Phase 08.1 | Aligns with `DatabaseSnapshot` model name, clearer semantics |
| No JS bridge script | Canonical `bridge.js` in Python package | Phase 08.1 | Complete OmniFocus integration -- the missing piece for end-to-end functionality |
| `pkg_resources` / `__file__` path hacks | `importlib.resources.files()` | Python 3.9+ (standard since 3.12) | Works in zip imports, namespace packages, wheels |

**Deprecated/outdated:**
- `importlib.resources.read_text(package, resource)` single-arg form: Superseded by `files().joinpath().read_text()` since Python 3.9. Both work but `files()` is the modern API.
- `::::` dispatch delimiter: Removed in this phase. No longer needed since UUID is embedded in the filename.
