_adapt_task(raw): skips if "status" not in raw. When status IS present:
  - Pops "status", maps to urgency+availability via _TASK_STATUS_MAP

  - Calls _adapt_parent_ref(raw)

_adapt_parent_ref(raw): transforms parent/project strings to unified parent dict:
  - If raw["parent"] (string) is not None: type="task", id=parent, name=parentName
  - Elif raw["project"] (string) is not None: type="project", id=project, name=projectName
  - Else: parent=None
  - Pops: project, parentName, projectName

_adapt_project(raw): skips if "status" not in raw. When present:
  - Pops "status" -> availability via _PROJECT_STATUS_MAP
  - Pops "taskStatus" -> urgency via _TASK_STATUS_MAP[0]
  - Removes dead fields

_adapt_tag(raw): skips if availability already in valid set. When not:
  - Pops "status" -> availability via _TAG_AVAILABILITY_MAP
  - Removes dead fields (allowsNextAction, active, effectiveActive)

_adapt_folder(raw): similar to tag

From src/omnifocus_operator/bridge/bridge.js (raw task format, lines 120-150):
  status: ts(t.taskStatus)  -- "Available", "Blocked", "DueSoon", etc.
  project: pk(t.containingProject)  -- string ID or null
  parent: pk(t.parent)  -- string ID or null
  (NO urgency, NO availability -- these are adapter outputs)

CRITICAL: In OmniFocus, `t.containingProject` returns the project that contains a task,
REGARDLESS of nesting depth. A sub-task 3 levels deep still has containingProject = the
project at the top. The `parent` and `project` fields are INDEPENDENT:
  - Inbox task:           parent=None,          project=None
  - Direct project child: parent=project_root,  project=project_id  (often same ID)
  - Sub-task in project:  parent=parent_task_id, project=containing_project_id
  - Sub-task in inbox:    parent=parent_task_id, project=None
