## Architecture Patterns

### Resolution Cascade (D-05)

The resolution cascade follows the existing write-side pattern but adds substring matching:

```
Agent filter value
    |
    v
[Step 1: ID match] -- does value match any entity's .id?
    |-- YES -> return [that_id]
    |-- NO  -> continue
    v
[Step 2: Substring match] -- case-insensitive `in` check against entity .name
    |-- matches found -> return [matched_ids]  (can be multiple)
    |-- no matches -> continue
    v
[Step 3: Fuzzy "did you mean?"] -- difflib.get_close_matches(value, all_names)
    |-- close matches found -> return ([], suggestions)
    |-- nothing close -> return ([], [])
    |
Pipeline: if resolved_ids empty, skip that filter field + attach warning
```

**Key: Resolver handles steps 1-2, DomainLogic handles step 3, Pipeline orchestrates.**

### Existing Write-Side Pattern to Mirror

Current `Resolver._match_tag()` (resolve.py lines 96-111):
- Exact name match (case-insensitive) -> returns single ID
- Fallback to ID match -> returns ID
- No match -> raises ValueError

Read-side difference: substring match replaces exact-name-only, multiple results allowed, no ValueError on miss (skip filter instead).

### Pipeline Structure (Method Object Pattern)

Per CLAUDE.md convention: `_ListTasksPipeline(_Pipeline)` and `_ListProjectsPipeline(_Pipeline)`.

Pipeline steps for tasks:
1. Resolve project filter (if present) -> project_ids
2. Resolve tag filters (if present) -> tag_ids
3. Build RepoQuery with resolved IDs + pass-through fields
4. Delegate to repo
5. Convert RepoResult -> ListResult (add warnings if any)

Pipeline steps for projects:
1. Resolve folder filter (if present) -> folder_ids
2. Build RepoQuery with resolved IDs + pass-through fields
3. Delegate to repo
4. Convert RepoResult -> ListResult (add warnings if any)

## Common Pitfalls

### Pitfall 1: Forgetting to Update All Test Files
**What goes wrong:** Contract field renames (project -> project_ids) break every test that constructs a RepoQuery
**Why it happens:** Tests in `test_query_builder.py` and `test_hybrid_repository.py` currently pass name strings
**How to avoid:** Search for all usages of old field names before marking the contract change as done
**Warning signs:** `mypy` or `pytest` failures on field names

### Pitfall 2: Substring Resolution Returning Too Many Results
**What goes wrong:** Agent passes "a" as project filter, matches 50 projects, produces a massive WHERE IN clause
**Why it happens:** Substring match is inherently broad
**How to avoid:** This is actually fine -- it mirrors current LIKE behavior. The agent already gets all partial matches today. No artificial limit needed.

### Pitfall 3: Tag Filter Already Uses IDs at Repo Layer
**What goes wrong:** Implementing unnecessary changes to the tag filter SQL
**Why it happens:** `tags` field on RepoQuery currently holds IDs (D-04c confirms "unchanged")
**How to avoid:** Only rename the field (`tags` -> `tag_ids`) for clarity. The SQL `WHERE IN` logic stays the same.

### Pitfall 4: Breaking the `get_all()` Cache with Multiple Calls
**What goes wrong:** Each filter resolution calls `get_all()` separately (project, tags, folders)
**Why it happens:** Following the write-side pattern of calling `get_all()` per resolution
**How to avoid:** The repo layer caches `get_all()` results (HybridRepository reads from SQLite). Multiple calls are fine perf-wise. But for cleanliness, the pipeline could call `get_all()` once and pass the snapshot to each resolution step.

### Pitfall 5: Missing `__init__.py` Re-exports After Renames
**What goes wrong:** Import paths break because `__init__.py` still exports old names
**Why it happens:** Contract field renames don't change class names, but if any re-export references the old field name in a factory or default, it breaks
**How to avoid:** The class names stay the same (ListTasksRepoQuery, etc.) -- only field names inside them change. No `__init__.py` changes needed.

## Code Examples

### difflib.get_close_matches Usage

```python
import difflib

# Returns up to 3 matches with >= 0.6 similarity
suggestions = difflib.get_close_matches(
    "Personl",
    ["Personal", "Work", "Errands", "Home"],
    n=3,
    cutoff=0.6,
)
# suggestions == ["Personal"]
```

### Resolver Read-Side Method (Discretion Area)

Recommended: single generic method that works for any entity type.

```python
def resolve_filter(
    self,
    value: str,
    entities: list[T],  # any entity with .id and .name
    *,
    id_attr: str = "id",
    name_attr: str = "name",
) -> list[str]:
    """Resolve a filter value to entity IDs. Returns [] on no match."""
    # Step 1: ID match
    if any(getattr(e, id_attr) == value for e in entities):
        return [value]
    # Step 2: Substring match (case-insensitive)
    matches = [
        getattr(e, id_attr)
        for e in entities
        if value.lower() in getattr(e, name_attr).lower()
    ]
    return matches
```

### DomainLogic Fuzzy Suggestion Method

```python
def suggest_close_matches(
    self,
    value: str,
    entity_names: list[str],
    n: int = 3,
    cutoff: float = 0.6,
) -> list[str]:
    """Return close name matches for a failed resolution."""
    return difflib.get_close_matches(value, entity_names, n=n, cutoff=cutoff)
```

### Query Builder Rewrite: Project Filter

**Before (LIKE):**
```python
if query.project is not None:
    conditions.append(
        "t.containingProjectInfo IN ("
        "SELECT pi2.pk FROM ProjectInfo pi2 "
        "JOIN Task t2 ON pi2.task = t2.persistentIdentifier "
        "WHERE t2.name LIKE ? COLLATE NOCASE)"
    )
    params.append(f"%{query.project}%")
```

**After (WHERE IN):**
```python
if query.project_ids is not None and len(query.project_ids) > 0:
    placeholders = ",".join("?" * len(query.project_ids))
    conditions.append(f"t.containingProjectInfo IN ({placeholders})")
    params.extend(query.project_ids)
```

Note: project_ids here are ProjectInfo PKs, not project persistent IDs. The service resolution must map project IDs to their ProjectInfo PKs, OR the WHERE IN should match against a subquery. Let me verify...

### Critical: Project ID vs ProjectInfo PK Mapping

Looking at the current SQL, `t.containingProjectInfo` references the `ProjectInfo.pk` column, NOT the project's `persistentIdentifier`. The current LIKE subquery handles this by joining through ProjectInfo to get the name.

For WHERE IN with pre-resolved IDs, two approaches:
1. **Resolve to ProjectInfo PKs** -- requires knowing the PK, which is an internal DB detail not exposed in the Project model
2. **Keep a subquery but use WHERE IN on persistentIdentifier** -- cleaner, keeps resolution in domain terms

Recommended approach 2:
```python
if query.project_ids is not None and len(query.project_ids) > 0:
    placeholders = ",".join("?" * len(query.project_ids))
    conditions.append(
        f"t.containingProjectInfo IN ("
        f"SELECT pi2.pk FROM ProjectInfo pi2 "
        f"WHERE pi2.task IN ({placeholders}))"
    )
    params.extend(query.project_ids)
```

This resolves project persistent IDs (which the service layer knows) through ProjectInfo to get PKs. The subquery is simple and indexed.

### Critical: Folder ID Mapping

Same pattern for folders. `pi.folder` stores the folder's `persistentIdentifier` directly, so WHERE IN is straightforward:
```python
if query.folder_ids is not None and len(query.folder_ids) > 0:
    placeholders = ",".join("?" * len(query.folder_ids))
    conditions.append(f"pi.folder IN ({placeholders})")
    params.extend(query.folder_ids)
```

## Open Questions

1. **`get_all()` call strategy for resolution**
   - What we know: Write-side calls `get_all()` per resolution. Repo layer caches it.
   - What's unclear: Should the pipeline call `get_all()` once and pass the snapshot, or let each resolution call it independently?
   - Recommendation: Let each call independently -- the caching handles it, and it keeps the Resolver API clean. Discretion area.

2. **Project ID to containingProjectInfo PK mapping**
   - What we know: `t.containingProjectInfo` stores ProjectInfo PKs, not project persistent IDs
   - What's unclear: Whether the query builder should do the PK lookup (subquery) or the service should resolve to PKs
   - Recommendation: Keep a minimal subquery in the query builder (`WHERE pi2.task IN (?)`) -- the service resolves to project persistent IDs, the SQL maps to PKs. This keeps the service in domain terms.

3. **Multiple tag resolution -- should each tag be resolved independently?**
   - What we know: Agent can pass `tags: ["Errand", "kFj9xL5"]` mixing names and IDs
   - What's unclear: Should the cascade run per-tag (matching write-side `resolve_tags`) or batch?
   - Recommendation: Per-tag resolution (matching write-side pattern). Each tag independently goes through ID match -> name substring -> fuzzy. Warnings accumulated per-tag.
