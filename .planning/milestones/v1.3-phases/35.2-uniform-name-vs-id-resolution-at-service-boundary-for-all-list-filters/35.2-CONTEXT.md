# Phase 35.2: Uniform Name-vs-ID Resolution at Service Boundary — Context

**Gathered:** 2026-03-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Service layer resolves ALL entity references (project, folder, tags) to IDs before the repository layer. RepoQuery becomes IDs-only. Query builder uses `WHERE IN (?)` uniformly. Full resolution cascade: ID match → substring match → fuzzy "did you mean?" with warnings.

**Requirements:** INFRA-07 (pulled forward from ~~Phase 37~~ Phase 36), INFRA-12, INFRA-13, INFRA-14, INFRA-15, INFRA-16

### What this phase delivers

1. **Resolution cascade** in the service layer — for each name-based filter:
   - Step 1: ID match (string matches a known entity ID → resolve to that ID)
   - Step 2: Substring match (case-insensitive, replicates current LIKE behavior in Python → resolve to one or more IDs)
   - Step 3: Fuzzy match via `difflib.get_close_matches` → attach "did you mean?" warning, skip the filter
2. **RepoQuery field redesign** — name fields replaced with ID-list fields
3. **Query builder rewrite** — project/folder LIKE-based SQL → `WHERE IN (?)` for all filters
4. **Minimal list pipelines** for tasks and projects (resolution + bare delegation)
5. **Inline pass-throughs** for tags, folders, perspectives (no pipeline overhead)
6. **`ListResult.warnings` field** — pulled forward from ~~Phase 37~~ Phase 36 D-02c

### NOT in scope

- Default completed/dropped exclusion logic (~~Phase 37~~ Phase 36 SC#1)
- Status shorthand expansion — `remaining` → `[available, blocked]` (~~Phase 37~~ Phase 36 SC#2)
- `review_due_within` duration parsing (~~Phase 37~~ Phase 36 SC#2)
- Validation (offset-requires-limit, invalid enum values) (~~Phase 37~~ Phase 36 SC#1, SC#4 / INFRA-06)
- ~~Phase 36~~ delivered in Phase 35.2 (BridgeRepository / in-memory fallback)

</domain>

## Implementation Decisions

### Scope: Minimal Pipelines (D-01)

- **D-01a:** Phase 35.2 builds `_ListTasksPipeline` and `_ListProjectsPipeline` with resolution + Query→RepoQuery conversion + repo delegation + RepoResult→Result conversion
- **D-01b:** Lifts `NotImplementedError` stubs — service list methods become callable after 35.2
- **D-01c:** ~~Phase 37~~ Phase 36 enriches existing pipeline classes with validation, defaults, shorthand expansion — it does NOT create them
- **D-01d:** Discipline rule: 35.2 pipelines do resolution + delegation only. Anything touching validation, shorthand expansion, or duration parsing → ~~Phase 37~~ Phase 36

### Did-you-mean Included (D-02)

- **D-02a:** Full resolution cascade (steps 1-2-3) ships in 35.2, including fuzzy matching via `difflib.get_close_matches` (stdlib, zero new deps)
- **D-02b:** INFRA-07 is delivered in Phase 35.2, pulled forward from ~~Phase 37~~ Phase 36
- **D-02c:** `ListResult.warnings: list[str] | None = None` added in 35.2 (pulled forward from Phase 35.1 D-02c / ~~Phase 37~~ Phase 36)
- **D-02d:** `warnings` field on `ListResult` only (agent-facing), NOT on `ListRepoResult` (repo-facing) — warnings are a service-layer concern
- **D-02e:** No-match policy: skip the filter (don't pass to repo), attach "did you mean?" warning if close matches found
  - **SUPERSEDED by Phase 57-04** (2026-04-24): the "skip the filter" fallback was a packaging artifact of bundling two UX choices into one option. Post-57-04: name-resolver no-match returns an empty result + a reworded "no match" warning; did-you-mean suggestions are preserved, but the "skipped" wording is dropped. See `.planning/phases/57-parent-filter-filter-unification/57-04-SUMMARY.md` for the full rationale.

### RepoQuery: Clean Replacement (D-03)

- **D-03a:** Name fields replaced with ID-list fields — clean cut, no dual fields
- **D-03b:** `ListTasksRepoQuery`: `project: str | None` → `project_ids: list[str] | None`, `tags: list[str] | None` → `tag_ids: list[str] | None`
- **D-03c:** `ListProjectsRepoQuery`: `folder: str | None` → `folder_ids: list[str] | None`
- **D-03d:** Type change from single `str` to `list[str]` for project/folder because substring resolution can match multiple entities
- **D-03e:** Mirrors write-side precedent: `AddTaskCommand.tags: list[str]` (names) → `AddTaskRepoPayload.tag_ids: list[str]` (IDs)

### Query Builder: Full Rewrite (D-04)

- **D-04a:** Project filter: correlated subquery `JOIN ProjectInfo→Task WHERE name LIKE ?` → `WHERE t.containingProjectInfo IN (?, ?, ?)`
- **D-04b:** Folder filter: subquery `JOIN Folder WHERE name LIKE ?` → `WHERE pi.folder IN (?, ?, ?)`
- **D-04c:** Tag filter: unchanged — already uses `WHERE IN` with IDs
- **D-04d:** Existing repo-level tests rewritten to pass IDs instead of names. Substring matching behavior moves to service-layer tests

### Resolution Code Location (D-05)

- **D-05a:** `Resolver` (service/resolve.py) — lookup role: `resolve_filter(value, entities) → list[str]` — ID match or substring match, returns IDs. Empty list on no match
- **D-05b:** `DomainLogic` (service/domain.py) — policy role: fuzzy matching via `difflib.get_close_matches`, returns candidate names for warning construction
- **D-05c:** Pipeline orchestrates: calls Resolver → if empty, calls DomainLogic for suggestions → builds warning string → skips filter
- **D-05d:** Pattern is consistent with write-side: Resolver = "what is this?", DomainLogic = "what do we tell the agent?"

### Simple Pass-throughs (D-06)

- **D-06a:** Tags, folders, perspectives get inline one-liner pass-throughs in the service — no pipeline classes
- **D-06b:** Service methods: convert Query→RepoQuery (identical fields today), call repo, convert RepoResult→Result. 3-4 lines each
- **D-06c:** Per ~~Phase 37~~ Phase 36 SC#3: "forward query models without unnecessary pipeline overhead"

- **"Add did-you-mean suggestions for unresolved name filters"** — folded into scope (D-02). Move to done on completion.

- `DomainLogic`: Write-side business rules class — add `suggest_close_matches()` for read-side fuzzy warnings
- `_AddTaskPipeline` / `_EditTaskPipeline`: Method Object pattern — template for `_ListTasksPipeline` / `_ListProjectsPipeline`

- `_build_tag_name_lookup()` in `hybrid.py`: Builds `{tag_id: tag_name}` dict — entities already in memory for resolution

### Established Patterns
- **Resolver = lookup, DomainLogic = policy**: Write-side already separates "find the entity" from "decide what to tell the agent"
- **Method Object for pipelines**: `_VerbNounPipeline` created, executed, discarded in one call. Mutable `self` is fine
- **Write-side boundary split**: `AddTaskCommand.tags: list[str]` (names) → `AddTaskRepoPayload.tag_ids: list[str]` (IDs). Read side mirrors this pattern exactly
- **Inline pass-throughs for simple reads**: `get_task`, `get_project`, `get_tag` are one-liner delegations in the service. Tags/folders/perspectives list methods follow the same pattern

### Resolution Cascade Example (from todo)

```
Agent Query              Service                          RepoQuery
────────────            ──────────────────               ──────────────
project: "Work"     →   not an ID                     →  project_ids: ["p1", "p2", "p3"]
                         substring match: "Work",
                         "Work Projects", "Homework"

project: "pJK.."    →   matches ID                    →  project_ids: ["pJK.."]

project: "Personl"  →   not an ID, no substring match →  (skip filter)
                         fuzzy: "Did you mean Personal?"
                         → warning attached to ListResult

tags: ["Errand",    →   resolve each:                 →  tag_ids: ["abc", "xyz"]
       "kFj9xL5"]       "Errand" → name → "abc"
                         "kFj9xL5" → ID → "xyz"
```
