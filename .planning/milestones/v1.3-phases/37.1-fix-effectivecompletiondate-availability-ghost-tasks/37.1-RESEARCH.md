# Phase 37.1: Fix effectiveCompletionDate availability ghost tasks - Research

**Researched:** 2026-04-04
**Domain:** SQLite query + Python mapper bug fix (OmniFocus availability mapping)
**Confidence:** HIGH

## Summary

Tasks with `effectiveDateCompleted` set (inherited from completed parent) but no own `dateCompleted` appear as `availability: "available"` — ghost tasks that pollute default queries. The fix is surgical: swap `dateCompleted` → `effectiveDateCompleted` and `dateHidden` → `effectiveDateHidden` in two locations (Python mapper + SQL clauses). Same for projects.

**Primary recommendation:** Fix `_map_task_availability()` and `_TASK_AVAILABILITY_CLAUSES` to use effective columns. Review and fix project equivalents. Add test cases for effective-only completion/drop scenarios in both get_all and list paths.

## Common Pitfalls

### Pitfall 1: Forgetting to update both Python mapper AND SQL clauses
**What goes wrong:** `get_all` (Python mapper) and `list_tasks` (SQL) produce different availability for same task
**Why it happens:** Two parallel codepaths compute the same thing
**How to avoid:** Fix both in the same plan; test both paths
**Warning signs:** Cross-path equivalence test failures

### Pitfall 2: Missing the call site update
**What goes wrong:** Changing the function signature without updating what columns are passed at the call site
**Why it happens:** Param rename alone doesn't change which row column is read
**How to avoid:** Update both the function params AND the call site row lookups

### Pitfall 3: Project availability has extra complexity
**What goes wrong:** Projects use `effectiveStatus` from ProjectInfo in addition to date columns
**Why it happens:** Project dropped status comes from either `effectiveStatus='dropped'` OR `dateHidden`
**How to avoid:** For projects, both `effectiveDateHidden` AND `effectiveStatus='dropped'` must still map to dropped. The date column swap applies to completion and the dateHidden check only.
