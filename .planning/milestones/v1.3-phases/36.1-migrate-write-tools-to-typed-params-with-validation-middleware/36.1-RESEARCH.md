# Phase 36.1: Migrate Write Tools to Typed Params with Validation Middleware - Research

**Researched:** 2026-03-31
**Domain:** FastMCP middleware, Pydantic validation, MCP inputSchema generation
**Confidence:** HIGH

## Summary

This phase replaces `list[dict[str, Any]]` handler signatures on `add_tasks` and `edit_tasks` with typed `list[AddTaskCommand]` / `list[EditTaskCommand]`, adds `ValidationReformatterMiddleware` to catch `ValidationError` and reformat into clean `ToolError`, and removes manual `model_validate()` + try/except from handlers. The approach is fully validated by prior spike work in `.research/deep-dives/fastmcp-middleware-validation/FINDINGS.md` (37/39 integration tests pass, 10/12 exact error parity).

The Command models, UNSET sentinel, and Patch types already exist and work correctly. The `PydanticJsonSchemaWarning` is already suppressed in `pyproject.toml`. The primary engineering challenge is test fixture updates -- the `server` fixture in `conftest.py` does not register middleware, so validation error tests currently exercise handler-level try/except, not middleware.

**Primary recommendation:** Execute the migration in this order: (1) create middleware + move formatting helpers, (2) update test fixture to register middleware, (3) change handler signatures + remove try/except, (4) update/verify error path tests, (5) add canary test.

## Architecture Patterns

### Middleware Registration Order (LIFO)
```python
# In create_server():
mcp.add_middleware(ValidationReformatterMiddleware())  # added FIRST = innermost
mcp.add_middleware(ToolLoggingMiddleware(logger))       # added SECOND = outermost
```
- Outermost (logging) wraps innermost (validation)
- Request flow: Logging -> Validation -> tool handler
- Error flow: tool raises ValidationError -> Validation catches + raises ToolError -> Logging sees ToolError, logs it

### Middleware Implementation Pattern
```python
class ValidationReformatterMiddleware(Middleware):
    async def on_call_tool(self, context: MiddlewareContext, call_next: Any) -> Any:
        try:
            return await call_next(context)
        except ValidationError as exc:
            messages = _format_validation_errors(exc)
            raise ToolError("; ".join(messages) or INVALID_INPUT) from None
```

### Handler Before/After
```python
# BEFORE (current):
async def add_tasks(items: list[dict[str, Any]], ctx: Context) -> list[AddTaskResult]:
    if len(items) != 1:
        raise ValueError(...)
    try:
        command = AddTaskCommand.model_validate(items[0])
    except ValidationError as exc:
        messages = _format_validation_errors(exc)
        raise ValueError("; ".join(messages) or INVALID_INPUT) from None
    # ... use command

# AFTER (target):
async def add_tasks(items: list[AddTaskCommand], ctx: Context) -> list[AddTaskResult]:
    if len(items) != 1:
        raise ValueError(...)
    command = items[0]
    # ... use command
```

### Error Location Rewriting (D-08)
With `list[Model]` params, Pydantic error locs include the list wrapper: `("items", 0, "fieldName")`. The formatter must:
1. Strip the `items.N.` prefix (first two elements when `loc[0] == "items"` and `loc[1]` is int)
2. Prepend `"Task {idx+1}: "` for agent readability
3. Handle nested fields: `("items", 0, "repetitionRule", "bogusField")` -> `"Task 1: repetitionRule.bogusField"`


### Anti-Patterns to Avoid
- **Catching ToolError in middleware:** Only catch `ValidationError`. `ToolError` from `ValueError` (batch limit, service errors) must pass through untouched.
- **Filtering by tool name:** D-04 says global scope. `ValidationError` won't fire on read tools (they take `str` params), so no cost.
- **Using `max_length=1` on list param:** D-07 says keep batch limit as `ValueError` in handler. Schema shouldn't advertise a temporary limit.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON Schema generation | Manual schema dict | Pydantic `TypeAdapter` + FastMCP auto-generation | Automatically stays in sync with models |
| Error type detection | String matching on error messages | `e.get("ctx", {})` structured data | Pydantic message wording may change across versions |
| Middleware dispatch | Manual method/path routing | FastMCP `Middleware` base class with `on_call_tool` | Handles all dispatch, type safety |

## Common Pitfalls

### Pitfall 1: Test fixture missing middleware
**What goes wrong:** Tests pass but validation errors bypass middleware, testing the old code path
**Why it happens:** `conftest.py` `server` fixture currently calls `_register_tools()` but NOT `add_middleware()`
**How to avoid:** Update `server` fixture to register both middlewares in correct order
**Warning signs:** Validation error tests pass but match `ValueError` instead of `ToolError` -- though currently tests already match `ToolError` because FastMCP wraps `ValueError` to `ToolError` at the protocol boundary

### Pitfall 2: ValueError vs ToolError confusion
**What goes wrong:** Tests break because error type changed
**Why it happens:** Currently handlers raise `ValueError`, FastMCP wraps to `ToolError` for the client. After migration, middleware raises `ToolError` directly. From the client's perspective, both appear as `ToolError`. BUT: batch limit errors still flow as `ValueError` -> FastMCP wraps to `ToolError`.
**How to avoid:** Verify that all `pytest.raises(ToolError, ...)` tests still pass -- the client-facing type doesn't change.
**Warning signs:** Tests that assert on internal error types rather than client-facing behavior

### Pitfall 3: items.0. prefix in unknown field paths
**What goes wrong:** `Unknown field 'items.0.bogusField'` instead of `Unknown field 'bogusField'`
**Why it happens:** With `list[Model]` params, Pydantic includes `items` and index in the loc tuple
**How to avoid:** Strip first two loc elements when they match the `items.N` pattern before formatting
**Warning signs:** Test `test_add_tasks_unknown_field_names_field` fails with `items.0.` in match string

### Pitfall 4: Nested unknown field paths need prefix stripping too
**What goes wrong:** `Unknown field 'items.0.repetitionRule.bogusField'` instead of `Unknown field 'repetitionRule.bogusField'`
**Why it happens:** Same `items.N.` prefix issue, but with nested models
**How to avoid:** Prefix stripping must happen before path joining in the `extra_forbidden` branch
**Warning signs:** Test `test_add_tasks_repetition_rule_unknown_field` fails

### Pitfall 5: Middleware catches too broadly
**What goes wrong:** Middleware catches exceptions it shouldn't (e.g., `ToolError` from service layer)
**Why it happens:** Overly broad except clause
**How to avoid:** Only catch `pydantic.ValidationError`, nothing else
**Warning signs:** Service-layer errors (task not found, parent not found) get reformatted

### Pitfall 6: _Unset.msg format assumption
**What goes wrong:** UNSET noise leaks into agent-facing error messages
**Why it happens:** Using string matching `"_Unset" in e["msg"]` which depends on Pydantic message format
**How to avoid:** D-08a: use `e.get("ctx", {}).get("class") == "_Unset"` (structural, version-stable)
**Warning signs:** Error messages containing `_Unset` or `is-instance[_Unset]`

## Code Examples

### ValidationReformatterMiddleware structure
```python
# Source: D-01, D-03, FINDINGS.md Approach 1
from pydantic import ValidationError
from fastmcp.exceptions import ToolError
from fastmcp.server.middleware import Middleware, MiddlewareContext

class ValidationReformatterMiddleware(Middleware):
    """Catch Pydantic ValidationError from typed tool params, reformat to clean ToolError."""

    async def on_call_tool(self, context: MiddlewareContext, call_next: Any) -> Any:
        try:
            return await call_next(context)
        except ValidationError as exc:
            messages = _format_validation_errors(exc)
            raise ToolError("; ".join(messages) or INVALID_INPUT) from None
```

### Updated _format_validation_errors with D-08 + D-08a
```python
# Source: D-08, D-08a, FINDINGS.md
def _format_validation_errors(exc: ValidationError) -> list[str]:
    messages: list[str] = []
    for e in exc.errors():
        # D-08a: ctx-based UNSET filtering (structural, not string-based)
        if e.get("ctx", {}).get("class") == "_Unset":
            continue
        if e["type"] == "missing":
            continue
        loc = _clean_loc(e.get("loc", ()))
        # D-08: strip items.N. prefix from list[Model] params
        loc = _strip_items_prefix(loc)
        # D-08: add "Task N:" prefix
        idx = _extract_item_index(e.get("loc", ()))  # before cleaning
        if e["type"] == "extra_forbidden":
            field = ".".join(str(part) for part in loc)
            msg = UNKNOWN_FIELD.format(field=field)
        else:
            msg = e["msg"]
        if idx is not None:
            msg = f"Task {idx + 1}: {msg}"
        messages.append(msg)
    return messages
```

### Error Type Flow Comparison

**Before:**
```
Invalid input -> handler model_validate() -> ValidationError -> handler catches -> ValueError -> FastMCP wraps -> ToolError to client
Batch limit  -> handler ValueError -> FastMCP wraps -> ToolError to client
Service error -> service ValueError -> FastMCP wraps -> ToolError to client
```

**After:**
```
Invalid input -> FastMCP type_adapter.validate_python() -> ValidationError -> middleware catches -> ToolError to client
Batch limit  -> handler ValueError -> FastMCP wraps -> ToolError to client (UNCHANGED)
Service error -> service ValueError -> FastMCP wraps -> ToolError to client (UNCHANGED)
```
