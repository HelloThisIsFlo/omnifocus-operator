# Phase 36.3: Centralize field descriptions into constants like warnings and errors - Context

**Gathered:** 2026-04-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Extract all agent-visible `Field(description=...)` strings and class docstrings from `models/` and `contracts/` into a centralized `agent_messages/descriptions.py` file. Follow the exact pattern established by `errors.py` and `warnings.py`. Add tests mirroring the existing error/warning constant tests.

**Requirements:** DESC-01, DESC-02, DESC-03, DESC-04, DESC-05, DESC-06

**NOT in scope:**
- Changing the wording of descriptions (those were approved in Phase 36.2)
- Docstrings on internal-only models (RepoPayload, RepoResult, RepoQuery, base classes)
- Module-level docstrings
- Docstrings on methods/functions
- Tool-level docstrings in server.py (those are separate from model/field descriptions)

</domain>

<decisions>
## Implementation Decisions

### Shared vs Split Constants
- **D-01:** Split constants where write-side and read-side descriptions genuinely differ. Write-side descriptions include instructional guidance ("Not for intentions -- use plannedDate instead") and format requirements ("Requires timezone..."). Read-side descriptions are factual labels ("Deadline with real consequences if missed."). Use naming like `DUE_DATE_WRITE` / `DUE_DATE` — they sit side-by-side in descriptions.py for easy drift detection.
- **D-02:** Where the text is genuinely identical across models/contracts (e.g., `on_days`: "Days of the week for weekly recurrence."), use a single shared constant.

### Enum Docstring Handling
- **D-03:** Centralize all enum docstrings. Confirmed by schema inspection: Pydantic pulls `StrEnum` class docstrings into JSON Schema `$defs` entries. They ARE agent-visible. Use `__doc__ = CONSTANT` pattern — this serves as a signal that the docstring is agent-facing, not just developer docs.

### File Organization
- **D-04:** Domain-based sections in `descriptions.py`, interleaving class docstrings and field descriptions within each domain. Matches the established pattern in `errors.py` and `warnings.py`. The primary audit question is "are all date descriptions consistent?" — domain cohesion serves that.

### Docstring Cleanup During Centralization
- **D-05:** Several class docstrings contain implementation details that leak into agent-visible schema. Clean these during centralization:
  - `OrdinalWeekdaySpec`: "Write-side ordinal-weekday model... inherits CommandModel (extra='forbid') for agent input validation" → remove implementation detail
  - `Perspective`: "Uses OmniFocusBaseModel (not OmniFocusEntity) because..." → trim to "A single OmniFocus perspective."
  - `Folder`: "Inherits id, name, url, added, modified from OmniFocusEntity." → trim to "A single OmniFocus folder with all fields."
  - `AllEntities`: "Aggregates all five entity collection lists..." → evaluate whether useful or implementation-flavored
  - `ListResult`: "Non-paginated: total=len(items), has_more=False" → trim implementation detail
  - `ReviewDueFilter`: "Value object: parsed duration..." → remove "Value object:" prefix
  - `OrdinalWeekday` (core model): "Exactly one of the 6 ordinal fields should be set (at-most-one validator)." → trim validator detail
  - `ParentRef`: "represented as None at the Task level" → evaluate usefulness

### What Gets Centralized vs Not
- **D-06:** Only strings that appear in agent-visible JSON Schema get centralized. Verified by generating schemas and inspecting `$defs`:
  - **NOT centralized** (not in schema): `OmniFocusEntity`, `ActionableEntity` (base classes — Pydantic flattens inherited fields into leaf schemas, no `$defs` entry), `OmniFocusBaseModel`, `StrictModel`, `CommandModel`, `QueryModel`, all `RepoPayload`/`RepoResult`/`RepoQuery`, `_Unset`
  - **Centralized even though not yet wired**: `DurationUnit`, `ReviewDueFilter` — appear in `ListProjectsQuery` schema `$defs`, will be agent-visible when list tools are wired in Phase 37

### `__doc__` Assignment Pattern
- **D-07:** Use `__doc__ = CONSTANT` as first line of class body for all agent-facing class docstrings. Confirmed by experiment: produces identical JSON Schema `description` as a regular docstring. This pattern is the convention signal for "this docstring is agent-facing."

### Tests
- **D-08:** Mirror existing error/warning constant tests:
  - All constants are non-empty strings
  - No unused constants (every constant is imported somewhere)
  - No inline description strings remain in agent-facing models (grep check)
  - The researcher/planner may find additional test angles
- **D-09:** Enforcement test approach: scan ALL classes in `models/` and `contracts/` directories, with an **exception list** for known internal classes (base classes, RepoPayload/RepoResult/RepoQuery, sentinels, etc.). New classes default to "must use centralized descriptions" unless explicitly added to the exception list. This is safer than maintaining a list of agent-facing classes — the assumption is agent-facing unless proven otherwise. For `Field(description=)`: AST-walk consumer files, find `Field()` calls with `description` keyword, flag if the value is a string literal (should be a constant). For class docstrings: check that agent-facing classes use `__doc__ = CONSTANT` assignment, not inline triple-quoted docstrings.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `agent_messages/errors.py` (~30 constants) and `warnings.py` (~20 constants): exact pattern to replicate
- `agent_messages/__init__.py`: star-import re-export — add `from omnifocus_operator.agent_messages.descriptions import *`

### Established Patterns
- Domain-grouped sections with `# --- Section Name ---` comment headers
- Module docstring explaining the file's purpose and usage
- Parameterized strings use `{placeholder}` syntax with `.format()` at call site
- Constants are UPPER_SNAKE_CASE

### Key Technical Facts
- `__doc__ = CONSTANT` produces identical JSON Schema as regular docstrings (confirmed by experiment)
- Pydantic does NOT create `$defs` entries for parent classes in inheritance chains — only models referenced as field types get entries. This is why `OmniFocusEntity`/`ActionableEntity` are invisible in schema but `Urgency` (enum used as field type) is visible.
- `StrEnum` class docstrings DO surface in JSON Schema `$defs` entries
- `Annotated[..., Field(description=...)]` type aliases (like `OnDate`) also surface in schema

</code_context>
