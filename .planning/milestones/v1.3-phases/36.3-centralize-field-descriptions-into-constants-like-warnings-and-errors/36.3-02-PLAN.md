**Class 1: `TestDescriptionConsolidation`** (mirrors TestWarningConsolidation / TestErrorConsolidation from test_warnings.py)

2. **`test_all_description_constants_referenced_in_consumers`** (DESC-04) -- Every constant in descriptions.py must appear in at least one consumer module's source. If the value is a string literal (`ast.Constant` with `isinstance(value, str)`) or f-string (`ast.JoinedStr`), fail with the file and line number. Constants (ast.Name) are OK. This catches inline `description="..."` sneaking back in.

4. **`test_no_inline_class_docstrings_on_agent_classes`** (DESC-03 enforcement) -- For each agent-facing class (collected via `inspect.getmembers` on consumer modules, filtering for classes defined in that module), check that the class body uses `__doc__ = ...` assignment pattern, NOT an inline triple-quoted docstring. Implementation approach: use AST to parse each consumer module, find ClassDef nodes for agent-facing classes, check if the first body statement is an `Expr(Constant(str))` (inline docstring = bad) vs `Assign` to `__doc__` (good). Skip classes in the exception list (see below).

**Important implementation notes:**
- For test 3 (inline Field descriptions): the AST check should look for `ast.keyword` nodes where `arg == "description"` and `value` is `ast.Constant` (bad) or `ast.JoinedStr` (bad). `ast.Name` (constant reference) is OK.
- For test 4 (inline docstrings): an inline docstring in AST is `Expr(value=Constant(value=<str>))` as the first statement of a ClassDef body. The `__doc__ = CONSTANT` pattern shows as `Assign(targets=[Name(id='__doc__')], value=Name(...))`.
- For test 5 (enforcement scan): use `pathlib.Path.rglob("*.py")` on models/ and contracts/ directories. Skip `__init__.py`, `base.py`, `_validators.py` files.
