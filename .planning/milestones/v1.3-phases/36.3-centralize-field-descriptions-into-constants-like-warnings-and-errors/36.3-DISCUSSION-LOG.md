## Shared vs Split Constants

| Option | Description | Selected |
|--------|-------------|----------|
| Split constants (e.g. `DUE_DATE_WRITE` / `DUE_DATE`) | Preserves semantic difference; write-side stays instructional, read-side stays clean | ✓ |
| Unified constant everywhere | Single source of truth per concept; read-side gets noisy instructional text | |

**User's choice:** Split constants — write and read descriptions do different jobs
**Notes:** User emphasized that centralization is precisely what prevents drift: "the whole point of concept drift is that's the whole reason I want to centralise things." Having both constants side by side in the same file makes divergence visible.

---

## Enum Docstring Handling

| Option | Description | Selected |
|--------|-------------|----------|
| Centralize (consistent — everything agent-facing in one place) | Complete audit trail; `__doc__ = CONSTANT` signals agent-facing | ✓ |
| Leave inline (short one-liners, low drift risk) | Zero overhead, IDE hover works naturally | |

**User's choice:** Centralize — anything in JSON Schema should be extracted
**Notes:** User pointed out that even Claude got confused about whether enum docstrings appear in schema. The `__doc__ = CONSTANT` pattern makes it "very clear that these are things we need to care about versus the other docstrings, which are more for developer experience." User also suggested inspecting the live MCP tool schema to see what agents actually see — this confirmed enum descriptions surface in `$defs`.
