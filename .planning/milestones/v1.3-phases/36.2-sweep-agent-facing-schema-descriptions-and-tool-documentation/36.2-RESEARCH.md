**Primary recommendation:** Execute the CONTEXT.md decisions mechanically -- this is a well-specified editing task, not a design task. The planner should organize by model layer (contracts first, read models second, tool docstrings last) to minimize context-switching.

## Architecture Patterns

### How Descriptions Flow to Agents

Two distinct layers produce agent-visible text:

1. **Tool docstrings** (`server.py` function docstrings) -- become `tool.description` in MCP `tools/list`
2. **Model docstrings** (class-level) -- become `description` on JSON Schema `$defs` entries in `inputSchema`
3. **Field descriptions** (`Field(description=...)`) -- become `description` on individual JSON Schema properties

All three are visible to agents. The tool docstring is the first thing seen. Schema descriptions are visible when agents inspect the inputSchema.

### Field Description Mechanics (Verified)

```python
# Adding Field(description=...) to a Pydantic field
from pydantic import Field, AwareDatetime

# On a direct field:
due_date: AwareDatetime | None = Field(default=None, description="...")

# On an inherited field (re-declare in child or parent):
# Works in both ActionableEntity and AddTaskCommand
# Alias (camelCase) is preserved from alias_generator
```

**Verified behavior:**
- `Field(description=...)` adds `"description"` to the property in JSON Schema
- Works with `AwareDatetime | None` union types
- Works with inherited fields (override in parent class is fine)
- camelCase alias from `alias_generator` is preserved
- No impact on serialization behavior -- schema-only change

## Common Pitfalls

### Pitfall 1: Breaking test_output_schema.py
**What goes wrong:** Changing a model's `Field()` declaration accidentally alters the default value or type, causing schema validation failure.
**Why it happens:** When adding `Field(description=...)` to a field that previously had a plain default, you must also preserve the default.
**How to avoid:** Always include `default=` when adding Field to an existing optional field. E.g., `due_date: AwareDatetime | None = Field(default=None, description="...")`.
**Warning signs:** `test_output_schema.py` fails after changes.

### Pitfall 2: Forgetting to re-declare Field on inherited fields
**What goes wrong:** Adding `Field(description=...)` to `ActionableEntity.due_date` works for Task and Project read models. But write-side `AddTaskCommand.due_date` is a separate declaration and needs its own `Field(description=...)`.
**Why it happens:** Read models inherit from ActionableEntity; write models are separate classes.
**How to avoid:** Treat write-side and read-side as two independent sets of fields to annotate.

### Pitfall 3: Forgetting default_factory on list fields
**What goes wrong:** `tags: list[TagRef] = Field(default_factory=list)` -- if you add `description=...` to this field, you must also preserve `default_factory=list`.
**How to avoid:** Check current Field() kwargs before adding description.

### Pitfall 4: Module docstrings vs class docstrings
**What goes wrong:** Removing bridge references from the module docstring but leaving the class docstring untouched (or vice versa).
**Why it happens:** D-02 lists both module and class docstrings to clean.
**How to avoid:** Each file has potentially TWO things to edit: the module-level `"""..."""` and the class-level `"""..."""`.

### Pitfall 5: Python conventions leaking into JSON-facing text
**What goes wrong:** Writing `None` instead of `null`, or `True/False` instead of `true/false` in Field descriptions.
**How to avoid:** All Field descriptions and docstrings visible in schema should use JSON conventions (null, true, false).

## Code Examples

### Adding Field(description=...) to existing optional field

```python
# BEFORE
due_date: AwareDatetime | None = None

# AFTER -- preserve the default

    description="Real-world deadline with negative consequences if missed. "
    "Requires timezone (ISO 8601 with offset or Z)."
)
```
