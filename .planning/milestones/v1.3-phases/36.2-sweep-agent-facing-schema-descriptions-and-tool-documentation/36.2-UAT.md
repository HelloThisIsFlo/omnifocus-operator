### Gap 8: on field should be a typed model, not dict[str, str]

- truth: "The 'on' field should use a typed Pydantic model so the schema is self-describing"
  status: open
  severity: major
  category: modeling
  locations:
    - "src/omnifocus_operator/models/repetition_rule.py:145 (read-side)"
    - "src/omnifocus_operator/contracts/shared/repetition_rule.py:59 (FrequencyAddSpec)"
    - "src/omnifocus_operator/contracts/shared/repetition_rule.py:99 (FrequencyEditSpec)"
  current_definition: "on: dict[str, str] | None = None"
  problem: |
    `dict[str, str]` is completely opaque in JSON Schema — an agent sees
    `{"type": "object", "additionalProperties": {"type": "string"}}` with
    no hint about the expected shape.

    The actual contract is: exactly one key from {first, second, third, fourth, last}
    mapped to a day name {sunday, monday, ..., saturday, day}. This is a fixed,
    well-known shape — similar to how MoveAction replaced a generic dict.

    A typed model (e.g., OrdinalWeekday with `ordinal: Literal[...]` and
    `day: Literal[...]`) would make the schema fully self-describing and
    enable Pydantic validation of both key and value.
  fix: |
    Create an OrdinalWeekday model (or similar) with:
    - ordinal: Literal["first", "second", "third", "fourth", "last"]
    - day: Literal["sunday", "monday", ..., "saturday", "day"]
    Replace dict[str, str] with OrdinalWeekday in all three locations.
    Update serialization to preserve backward compatibility in JSON output
    (if the current output is {"last": "friday"}, the new model should
    serialize the same way, or the outputSchema change is documented).
