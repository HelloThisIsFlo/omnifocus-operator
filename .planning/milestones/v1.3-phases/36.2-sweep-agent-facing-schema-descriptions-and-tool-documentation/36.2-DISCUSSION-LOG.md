## Docstring Cleanup Strategy

| Option | Description | Selected |
|--------|-------------|----------|
| Two-tier (Recommended) | Strip implementation notes, keep brief agent-useful summaries on nested specs agents construct | ✓ |
| Remove all | Delete all model docstrings. Schema field names/types speak for themselves | |
| Full rewrite | Rewrite every model docstring to be agent-useful behavioral description | |

**User's choice:** Two-tier — strip implementation notes, keep agent-useful summaries where nested specs are actively constructed by agents.
**Notes:** Confirmed by advisor research. The two-tier approach eliminates harmful leakage (validator delegation, extra='forbid', bridge internals) while preserving behavioral hints on specs like FrequencyEditSpec.

## Tool Description Density

| Option | Description | Selected |
|--------|-------------|----------|
| Behavioral rules only (Recommended) | Cut all field listings from docstrings. Keep patch semantics, mode constraints, repetition examples. ~15 lines per tool | |
| Minimal (5 lines) | Cut field listings AND repetition examples. Trust schema enums for everything | |
| Field descriptions + slim docstrings | Add Field(description=...) to key Pydantic fields AND slim tool docstrings. Architecturally clean but expands scope | ✓ |

**User's choice:** Field descriptions + slim docstrings — the architecturally clean option.
**Notes:** User explicitly stated "the effort of the changes is irrelevant. What is the number one thing to optimise is how fluent it is for the agent to read in the end."

### Follow-up: Field description scope

| Option | Description | Selected |
|--------|-------------|----------|
| Key behavioral fields only (Recommended) | Add descriptions to fields where behavior is non-obvious (~15-20 fields) | |
| All write-side fields | Every field gets a description (~40-50 fields) | |
| All fields including read-side | Write AND read models get Field(description=...) | |

**User's choice:** Other (free text) — "What we optimise for is agent fluency. Adding description to every field doesn't make much sense. If the field is called 'flagged', adding a string that says whether the task is flagged or not is useless, because it's already in the field name."
**Notes:** The fluency filter: add descriptions only where the field name + type leave ambiguity about behavior or valid values. Not a count target — a quality filter.

## Read Tool Description Scope

| Option | Description | Selected |
|--------|-------------|----------|
| Yes, descriptions only (Recommended) | Improve read tool descriptions. No read-model field annotations | |
| No, write tools only | Keep focused on write tools | |
| Yes, with read-model Field descriptions | Same fluency filter as write side: add Field(description=...) to read-model fields where field name is ambiguous | ✓ |

**User's choice:** Other (free text) — "Again, what do we optimise for? Agent fluency, nothing else."
**Notes:** Same fluency filter applies to read models. Fields like `effective_due_date`, `children_are_mutually_exclusive`, `next_task` fail the "can the agent infer this from the name?" test. `flagged`, `name`, `id` pass it.

### Follow-up: Availability/urgency descriptions

Discussion about whether `availability` and `urgency` enums need descriptions. User noted these are codebase concepts, not OmniFocus native terms. After reviewing the enum values (`available`, `blocked`, `completed`, `dropped` / `overdue`, `due_soon`, `none`), consensus was that the values are self-explanatory — no Field(description=...) needed. The `effective_*` date fields are the real candidates for descriptions.

## Three-Model Agent Review

After the four discussion areas were resolved, three sub-agents (Haiku, Sonnet, Opus) were launched to review all 6 tools purely from an agent consumer perspective — no codebase access, only tool descriptions and input schemas.

### Consensus findings (flagged by all three models):
1. Implementation details leaking through schema descriptions (FrequencyAddSpec, FrequencyEditSpec, RepetitionRuleEditSpec, Schedule, TagAction)
2. Tag resolution completely undocumented (name vs ID, ambiguity, non-existent names)
3. No response/output shape documentation (read tools say "Returns the full Task object" with no field info)
4. "Limited to 1 item" vs unconstrained array schema contradiction
5. Tool description field listings redundant with schema
6. Partial repetitionRule on task with no existing rule — undefined behavior

### Unique insights:
- **Haiku:** "interval defaults to 1, omit or set explicitly" is self-contradictory; `lifecycle: "drop"` is OmniFocus jargon
- **Sonnet:** `schedule`/`basedOn` required asymmetry between add/edit; camelCase note inconsistent across tools; `replace: null` vs `replace: []` equivalence unclear
- **Opus:** Schema descriptions vs tool descriptions feel written by different audiences; no guidance on operation ordering (complete + edit in same call)

### Key discovery: outputSchema not advertised
FastMCP v3 does NOT include `outputSchema` in `tools/list`. Agents have zero visibility into response shape. This elevates the importance of read tool descriptions.
