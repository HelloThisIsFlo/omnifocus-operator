## Type alias location

| Option | Description | Selected |
|--------|-------------|----------|
| Keep in models/ (status quo) | Single source of truth, no dependency inversion | |
| Move to contracts/ | Aliases live where consumed for JSON Schema. models/ uses plain sets for validation. | ✓ |
| Move to models/type_aliases.py | Separate file for pure type aliases inside models/ | |

**User's choice:** Move to contracts/
**Notes:** Initial confusion about dependency direction (contracts/ depends on models/, not the reverse). After clarifying that `_VALID_DAY_NAMES` derivation from `DayName.__args__` can be replaced with a plain set literal, the move to contracts/ was straightforward. User confirmed dependency direction stays clean.

### Follow-up: Inline vs named aliases

| Option | Description | Selected |
|--------|-------------|----------|
| Inline Literal types at point of use | Remove named aliases, put Literal directly on fields | |
| Keep named aliases | DayCode, FrequencyType, OnDate, DayName as named type aliases | ✓ |

**User's choice:** Keep named aliases
**Notes:** User initially suggested inlining for less indirection, but reconsidered when learning all aliases are used on 2+ fields (DayName on 6 fields in OrdinalWeekdaySpec). Agreed named aliases are worth it when reused.

## Core model field types

| Option | Description | Selected |

**Notes:** Also identified two contract fields that should gain Annotated constraints: `FrequencyAddSpec.interval` and `EndByOccurrences.occurrences` (both validated at runtime but schema doesn't advertise `minimum: 1`).
