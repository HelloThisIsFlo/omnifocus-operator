# Phase 53.1: Code Review Report

**Reviewed:** 2026-04-14
**Depth:** standard
**Files Reviewed:** 16
**Status:** issues_found

## Summary

This phase moves inherited fields from `ActionableEntity` (shared base) to `Task`-only, adds a `compute_true_inheritance` walk in `DomainLogic`, and wires the walk into all read paths. The structural change is clean — `Project` correctly has zero inherited fields, the adapter removes them, and the service layer applies the walk. The test coverage for the new walk is thorough.

Three warnings and two info items found. No critical issues.

The most significant finding is a potential logical gap in `_walk_one`: it checks `hasattr(project, direct)` before reading a project field, but `Project` inherits `drop_date` and `completion_date` from `ActionableEntity` — those fields always exist. The guard is therefore vacuously true and never short-circuits, which is fine for correctness, but conceals that `_INHERITED_FIELD_PAIRS` includes `("drop_date", "inherited_drop_date")` and `("completion_date", "inherited_completion_date")`. A project's `drop_date` and `completion_date` are real values (a dropped/completed project has them set), so the walk will correctly preserve `inherited_drop_date` / `inherited_completion_date` when the project is in those states — which is intentional. However, the code comment says "D-01: check containing project" with no indication that `drop_date` and `completion_date` on the project side are relevant. This deserves a comment clarification to prevent future misreads.

## Warnings

### WR-01: `_walk_one` does not guard against infinite loops in cyclic parent chains

**File:** `src/omnifocus_operator/service/domain.py:232-243`
**Issue:** The `while current_id is not None` loop in `_walk_one` walks the parent chain via `task.parent.task.id`. If the OmniFocus database ever contains a cycle (task A's parent is task B, task B's parent is task A), this loop runs forever. The data comes from SQLite/bridge so a cycle is not expected in production, but there is no cycle-detection guard (visited set) and no depth limit. The existing cycle-detection in `DomainLogic.process_move` shows the project is cycle-aware — the inheritance walk has no equivalent protection.

**Fix:**
```python
visited: set[str] = {task.id}  # seed with self to detect self-cycles too
current_id: str | None = task.parent.task.id if task.parent.task else None
while current_id is not None and current_id not in visited:
    visited.add(current_id)
    ancestor = task_map.get(current_id)
    if ancestor is None:
        break
    # ... existing loop body ...
    current_id = ancestor.parent.task.id if ancestor.parent.task else None
```

### WR-02: `compute_true_inheritance` in `list_tasks` applies the walk to already-paginated results, not the full task set

**File:** `src/omnifocus_operator/service/service.py:456-458`
**Issue:** In `_ListTasksPipeline._delegate`, the walk is applied after pagination:
```python
repo_result = await self._repository.list_tasks(self._repo_query)
walked_items = await self._domain.compute_true_inheritance(repo_result.items)
```
`compute_true_inheritance` internally calls `self._repo.get_all()` to build `task_map` and `project_map`, then walks only the items passed in. This is correct for the walk itself (ancestors are looked up in the full `task_map`). However, there is a subtle issue: if the repo query returns a filtered page (e.g., only flagged tasks), the items in `walked_items` may include tasks whose ancestors are in `task_map` but were NOT in the page. The walk will still resolve them correctly via `task_map`. **This is actually fine** — the comment in `compute_true_inheritance` explains `get_all()` is called internally.

The real concern is the opposite: for the `get_all_data()` path (line 150-151), the walk is applied to `raw.tasks` only. If `raw.tasks` is a subset (which it always is — it's all tasks), ancestors are available. No bug here either. However, the two call sites are asymmetric:
- `get_all_data`: walk applied AFTER repo returns all tasks
- `list_tasks._delegate`: walk applied on the paginated page, but ancestors fetched from full snapshot

This asymmetry is harmless but could confuse a future maintainer. The `get_task` path (line 157-158) passes a single-element list, which is correct.

The actual latent issue: `compute_true_inheritance` calls `self._repo.get_all()` for every invocation of `_ListTasksPipeline._delegate`. For `BridgeOnlyRepository`, this is cached. For `HybridRepository`, this opens a **fresh SQLite connection** every time (via `asyncio.to_thread(self._read_all)`). A list_tasks call therefore triggers two full SQLite reads: one for the filtered query, one for the full snapshot inside `compute_true_inheritance`. This is a performance concern but is out of v1 review scope. The correctness concern is that the two reads are not atomic — data could change between them in theory. Under normal usage (single server, OmniFocus updates are user-driven) this race is negligible.

No code fix required for correctness; document the call-site asymmetry for clarity.

### WR-03: `_adapt_task` applies `_rename_inherited_fields` separately from `_adapt_task`, creating a two-pass risk

**File:** `src/omnifocus_operator/repository/bridge_only/adapter.py:413-415`
**Issue:** In `adapt_snapshot`, the loop over tasks first calls `_adapt_task(task)` then `_rename_inherited_fields(task)` in separate passes:
```python
for task in raw.get("tasks", []):
    _adapt_task(task)
    _rename_inherited_fields(task)
```
`_adapt_task` itself is a no-op if `"status"` is not present (idempotency guard at line 232). But `_rename_inherited_fields` is always called, even on tasks that were no-ops. If a task dict somehow arrives with the adapter already applied (i.e., `inherited*` keys already present) but with `effective*` keys also still present, both old and new keys would temporarily coexist during the rename pass. This is the documented "safe to call on already-renamed data: missing old keys are skipped" behavior, which is correct.

However, there is a functional mismatch: `_adapt_task` is gated on `"status"` presence, but `_rename_inherited_fields` has no such gate. A task with `effectiveFlagged` but no `"status"` (hypothetical partial update) would get its `effective*` keys renamed to `inherited*` without the rest of the task being adapted. In the current codebase this scenario cannot arise (all bridge output includes `status`), but the split guard logic is fragile.

**Fix:** Move `_rename_inherited_fields` call inside `_adapt_task` so they share the same idempotency gate:
```python
def _adapt_task(raw: dict[str, Any]) -> None:
    if "status" not in raw:
        return
    # ... existing body ...
    _rename_inherited_fields(raw)  # move here, remove from adapt_snapshot loop
```
And in `adapt_snapshot`:
```python
for task in raw.get("tasks", []):
    _adapt_task(task)  # _rename now called inside
```

## Info

### IN-01: `hasattr(project, direct)` guard in `_walk_one` is always True for all current field pairs

**File:** `src/omnifocus_operator/service/domain.py:250`
**Issue:** The guard `hasattr(project, direct)` before checking the project's field value will always return `True` for every field in `_INHERITED_FIELD_PAIRS`. All six direct fields (`flagged`, `due_date`, `defer_date`, `planned_date`, `drop_date`, `completion_date`) are defined on `ActionableEntity`, which `Project` inherits. The `hasattr` check was likely added to be safe against future refactors removing project fields, but as written it is a no-op guard.

The comment "# Check containing project as final ancestor (D-01)" doesn't explain why `drop_date` and `completion_date` are meaningful on a project ancestor (a dropped or completed project does propagate to children in OmniFocus). A clarifying comment would help.

**Fix:** Either remove the `hasattr` guard with a comment explaining all fields are always present on projects, or replace it with an explicit allowlist of which project fields are checked:
```python
# All 6 fields exist on Project (inherited from ActionableEntity),
# including drop_date/completion_date which are set when a project is
# dropped/completed and do propagate to children.
for direct, inherited in _INHERITED_FIELD_PAIRS:
    if not ancestor_sets[inherited]:
        val = getattr(project, direct)
        ...
```

### IN-02: `test_all_six_field_pairs_self_echo_stripped` passes `dropDate` and `completionDate` as task-level overrides but not `make_model_project_dict`-compatible keys

**File:** `tests/test_service_domain.py:1803-1811`
**Issue:** In `test_all_six_field_pairs_self_echo_stripped`, the `_make_project` call passes `dropDate=None` and `completionDate=None` as overrides. `make_model_project_dict` uses snake_case for some fields and camelCase for others — specifically, `drop_date` is stored as `dropDate` in the dict (camelCase). Let's verify: `make_model_project_dict` does use `"dropDate": None` (camelCase, line 270 of conftest). So passing `dropDate=None` in the override is correct here.

However, the test does NOT pass `dropDate` or `completionDate` to `_make_project`. The test only clears `flagged`, `dueDate`, `deferDate`, `plannedDate`. The project has `dropDate=None` and `completionDate=None` by default from the factory. This means the test implicitly relies on the factory defaults being `None` for those two fields — which they are. The test is correct, but it doesn't explicitly verify that `drop_date` / `completion_date` on the project side are also checked. Adding explicit non-None project `completionDate` to one test variant would improve completeness.

This is a test coverage gap, not a bug.

**Fix:** Add a variant test confirming that a completed project causes `inherited_completion_date` to be preserved on the child task. (No immediate code change needed — existing tests cover the main paths.)
