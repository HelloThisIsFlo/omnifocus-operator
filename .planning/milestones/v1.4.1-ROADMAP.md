# Milestone v1.4.1: Task Property Surface & Subtree Retrieval

**Status:** ✅ SHIPPED 2026-04-24
**Phases:** 56-57
**Total Plans:** 14 (7 original + 2 gap-closure on Phase 56; 3 original + 2 gap-closure on Phase 57)

## Overview

Strictly additive point release that exposes OmniFocus task properties the server previously hid (auto-complete, parallel/sequential, presence of notes/repetition/attachments) and adds a `parent` filter on `list_tasks` so agents can fetch a task's descendants in a single call. Design locked 2026-04-19 after four-session interview + two pre-implementation spikes. No fields scoped out. Projects stay read-only — project writes deferred to v1.5.

## Milestone Summary

**Decimal Phases:** None (gap-closure plans 56-08/09 and 57-04/05 are plan-level additions within their existing phases, not inserted phases)

**Key Decisions:**

- **Cache-backed task property surface** — no per-row bridge fallback on either repo; HybridRepository uses `Task.completeWhenChildrenComplete`/`Task.sequential`/`ProjectInfo.containsSingletonActions` + batched `SELECT task FROM Attachment`; BridgeOnlyRepository uses inline `completedByChildren`/`sequential`/`attachments.length` during snapshot-build
- **Write never relies on OmniFocus implicit defaulting** — when agent omits `completesWithChildren`/`type` on `add_tasks`, server writes resolved preference value explicitly (factory default `true`/`"parallel"` when OF Setting store empty)
- **`ProjectType` assembled at service layer** from `(sequential, containsSingletonActions)` — `"singleActions"` takes precedence (HIER-05 truth table, consolidated to `ProjectType.from_flags` classmethod during milestone audit — single source of truth across both repos + `DomainLogic.assemble_project_type`)
- **No-suppression invariant** — default + hierarchy include pipelines are independent; redundant emission between `dependsOnChildren` (default) and `completesWithChildren` (hierarchy) is intentional (low-cost redundancy, high-value predictability); proven by contract test
- **`candidate_task_ids` as unified wire-level primitive** — retired `ListTasksRepoQuery.project_ids`, renamed from `task_id_scope` mid-phase (57-04); both `project` and `parent` filters pass through this single primitive
- **Anchor preservation via repo-layer `pinned_task_ids` OR-clause (Option A)** — chosen over service-layer alternatives after G1 UAT gap; honors `FILTERED_SUBTREE_WARNING` text ("resolved parent tasks are always included"); xfail test promoted to pass
- **Scope expansion at service, set-membership at repo** — single code path across both repos, no divergent subtree implementations; benchmark-validated (p95 ≤ 1.30 ms @ 10K, 77× under viable threshold)
- **Warnings live in domain layer** — filter-semantics advice, not projection concern; `DomainLogic.check_filtered_subtree` + `check_parent_project_combined` emit from pipeline after all resolutions
- **`OmniFocusPreferences` lazy-load-once pattern extended** — two new keys (`OFMCompleteWhenLastItemComplete`, `OFMTaskDefaultSequential`) slot into existing bridge-based pattern; absence-as-factory-default semantic (missing key OR null value resolves to OF factory default)
- **`extra="forbid"` rejects all 6 derived read-only flags** — `hasNote`, `hasRepetition`, `hasAttachments`, `hasChildren`, `dependsOnChildren`, `isSequential` blocked on write tools via generic Pydantic schema error (no custom educational messaging needed; JSON Schema already tells agents which fields are writable)
