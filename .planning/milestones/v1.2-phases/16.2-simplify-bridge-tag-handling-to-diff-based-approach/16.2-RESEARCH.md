## Architecture Patterns

### Current Flow (being replaced)
```
TagActionSpec → service builds tagMode+tagIds/addTagIds/removeTagIds → bridge dispatches 4 branches
```

### New Flow
```
TagActionSpec → service._compute_tag_diff() → (add_ids, remove_ids, warnings)
  → if both empty: no-op (skip bridge)
  → else: payload gets addTagIds/removeTagIds only → bridge applies ~4 lines
```

### Diff Algorithm (locked)
```python
# Input: TagActionSpec + current task tags
# Output: (add_ids: list[str], remove_ids: list[str], warnings: list[str])

current = {t.id for t in task.tags}

# Compute final_set based on action type:
if replace:  final = set(resolved_replace_ids)  # replace: null/[] → empty set
elif add+remove: final = (current | set(add_ids)) - set(remove_ids)
elif add:    final = current | set(add_ids)
elif remove: final = current - set(remove_ids)

to_add = final - current
to_remove = current - final
```

### No-Op Detection Simplification

Current no-op flow has a separate tag check (lines 328-338) that inspects `tagMode` in the payload. With diff-based approach:
- If tag diff is empty, **no tag keys appear in payload at all** -- no-op detection for tags is implicit
- The `tagMode` check block (lines 328-338) is deleted entirely
- Only field comparisons + moveTo check remain in the no-op section
- Fix: `len(payload) > 1` → `len(payload) >= 1` catches the "only id, no changes" case

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Tag set operations | Manual index-based iteration | Python set operations | Current code uses index-based loops with `enumerate` -- sets are clearer and correct |
| Warning deduplication | Custom tracking | Set-based diff naturally deduplicates | Diff produces exact add/remove sets; warnings derive from comparing input intent vs diff |

## Common Pitfalls

### Pitfall 1: Tag Resolution Ordering
**What goes wrong:** `_resolve_tags()` takes names-or-IDs and returns IDs. The diff must use resolved IDs, not raw input strings.
**How to avoid:** Resolve first, then compute sets. The resolved IDs are what get compared to `current_tags[].id`.

### Pitfall 2: Warning Generation for Replace Mode
**What goes wrong:** Replace mode currently has no per-tag warnings. New behavior adds a replace no-op warning when diff is empty.
**How to avoid:** After computing diff for replace, if `to_add` and `to_remove` are both empty, generate "tags already match" warning. Different from add/remove per-tag warnings.

### Pitfall 3: InMemoryRepository Skip Keys
**What goes wrong:** `skip_keys` in `in_memory.py` includes `tagMode`, `tagIds`, `addTagIds`, `removeTagIds`. After refactor, `tagMode` and `tagIds` are gone.
**How to avoid:** Update skip_keys to only `{"id", "addTagIds", "removeTagIds", "moveTo"}` and rewrite tag handling block.

### Pitfall 4: Empty Arrays vs Missing Keys
**What goes wrong:** Sending `addTagIds: []` to bridge when there's nothing to add -- unnecessary work.
**How to avoid:** Per CONTEXT.md: if tag diff is empty, no tag payload sent at all (skip, not empty arrays). Only add keys when non-empty.

### Pitfall 5: add+remove with Same Tag
**What goes wrong:** `add=["A"], remove=["A"]` where A is already on the task. Current test (`test_warning_addtags_duplicate_in_add_remove`) expects warning for "already on this task" but NOT "is not on this task".
**How to avoid:** Diff computation order matters: `final = (current | add) - remove`. So adding A (already there) then removing A means final = current - {A}. Warning should be for the remove part's intent, not the intermediate state. Need to preserve per-tag warning semantics from the original input, not from the diff.

### Pitfall 6: Warning Source -- Input Intent vs Diff Result
**What goes wrong:** Diff-based approach loses information about user intent. User said `add: ["A"]` but A is already there -- that's a warning. Pure diff just sees "no change to A" and might skip it.
**How to avoid:** Generate warnings by comparing user's input actions against current state BEFORE computing the final diff. Then compute diff for the bridge payload. Two concerns: warnings (from input) and payload (from diff).

function resolveTagIds(ids) {
    return ids.map(function(id) {
        var tag = Tag.byIdentifier(id);
        if (!tag) throw new Error("Tag not found: " + id);
        return tag;
    });
}
// ...
if (params.hasOwnProperty("removeTagIds"))
    task.removeTags(resolveTagIds(params.removeTagIds));

### InMemoryRepository New Tag Handling
```python
# Handle tag operations (diff-based: addTagIds/removeTagIds)
remove_ids = set(payload.get("removeTagIds", []))
if remove_ids:
    task.tags = [t for t in task.tags if t.id not in remove_ids]
add_ids = payload.get("addTagIds", [])
if add_ids:
    existing_ids = {t.id for t in task.tags}
    for tid in add_ids:
        if tid not in existing_ids:
            task.tags.append(TagRef(id=tid, name=tid))
```
