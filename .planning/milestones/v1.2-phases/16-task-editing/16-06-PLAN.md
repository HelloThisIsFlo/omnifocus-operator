**GAP 2 — service.py (lines ~273-275): UTC-normalize date comparison.**
In the `field_comparisons` dict, instead of comparing ISO strings, compare UTC timestamps. Change the date entries from:
  `"dueDate": task.due_date.isoformat() if task.due_date else None,`
to a comparison approach that normalizes both sides. The payload dates are ISO strings from Pydantic serialization. The task dates are datetime objects. Compare by converting both to UTC datetime:
```python
from datetime import datetime, timezone
```
For date fields in field_comparisons, store the task's current date normalized to UTC. When checking payload values against current values, parse the payload ISO string back to a UTC datetime for comparison. The simplest approach: store current values as UTC timestamps (float) and convert payload strings to UTC timestamps too.

Recommended implementation for the date fields in field_comparisons:
```python
def _to_utc_ts(val: object) -> object:
    """Normalize a date value to UTC timestamp for comparison, or return as-is."""
    if val is None:
        return None
    if isinstance(val, datetime):
        return val.astimezone(timezone.utc).timestamp()
    if isinstance(val, str):
        return datetime.fromisoformat(val).astimezone(timezone.utc).timestamp()
    return val
```
Then in the no-op comparison loop, use `_to_utc_ts(payload[key]) != _to_utc_ts(current_value)` instead of direct `!=` for the three date fields. Or simpler: store current dates as UTC timestamps in field_comparisons and convert payload dates at comparison time.

**GAP 5 — service.py (lines ~294-296): Same-container move detection.**
Replace the unconditional `is_noop = False` for moveTo with a check:
- For `beginning`/`ending` positions: compare the `containerId` against the task's current parent ID (`task.parent.id if task.parent else None`). If they match, the task is already in this container — append warning `"Task is already in this location"` and do NOT set `is_noop = False`. Still send the move to the bridge (position within container may differ), but add the warning.
- For `before`/`after` positions: we cannot determine sibling order from our data model, so always treat as a real change (`is_noop = False`).
- Important: Even when same-container is detected, still include the moveTo in the payload (the position within the container may change). The warning is advisory, not a skip.

Actually, re-reading UAT test 70 more carefully: the expectation is "Task is already in this position" warning. Since we can't fully detect same-position, let's detect same-container for beginning/ending only and add an advisory warning. For the is_noop flag: if moveTo is the ONLY change and it's same-container, keep is_noop=True so we return the no-op early return with the position warning appended. If there are other field changes alongside the same-container move, don't skip — just add the warning.

Revised approach for GAP 5:
```python
if is_noop and "moveTo" in payload:
    move_data = payload["moveTo"]
    position = move_data.get("position")
    if position in ("beginning", "ending"):
        container_id = move_data.get("containerId")
        current_parent_id = task.parent.id if task.parent else None
        if container_id == current_parent_id:
            warnings.append("Task is already in this location")
            # is_noop stays True — same container
        else:
            is_noop = False
    else:
        # before/after — can't detect same position
        is_noop = False
```
