<objective>
Close gap G1 from `56-HUMAN-UAT.md`: FLAG-04's "next-in-line child is available" semantic applies equally to projects with `type == 'sequential'`, but Phase 56 shipped `is_sequential` as tasks-only. Hoist `is_sequential` from `Task` to `ActionableEntity` so both entities expose the derived flag, add a project-side enrichment step that mirrors the existing task enrichment, surface the field on the project default response + tool docs, and update negative assertions and REQUIREMENTS.md to reflect the corrected scope.

Purpose: the gap was surfaced during Human UAT when reviewing the FLAG-07 behavioral copy — agents reasoning about project actionability face the same "don't over-count available children" trap on a sequential project that they do on a sequential task. Tasks-only was a design omission, not an intentional boundary. `dependsOnChildren` (FLAG-05) stays tasks-only because projects are always containers; the "real unit of work waiting on children" semantic does not apply to projects.

Output:
- `is_sequential` declared once on `ActionableEntity`, removed from `Task`, inherited by `Project` with no explicit declaration.
- New `DomainLogic.enrich_project_presence_flags` computing `is_sequential = (project.type == ProjectType.SEQUENTIAL)`.
- Project read pipelines (`get_all_data`, `get_project`, `_ListProjectsPipeline._delegate`) wire the new enrichment in the same shape as the existing task enrichment.
- `PROJECT_DEFAULT_FIELDS` gains `"isSequential"`; stripped-when-false behavior inherited automatically from existing projection logic.
- `IS_SEQUENTIAL_DESC` reworded to drop the tasks-only prefix; `LIST_PROJECTS_TOOL_DOC` updated with a project-scoped behavioral note covering `isSequential` only.
- Four existing negative assertions flipped to positive (model fields test, two projection tests, tool-doc test).
- REQUIREMENTS.md FLAG-04 wording revised.
- Full test suite + mypy + output-schema test green.

Note on scope discipline: WR-01 (duplicate preferences warning surfaced in 56-REVIEW) is NOT in scope for this gap-closure plan; that's a separate bug documented in 56-REVIEW and will be addressed independently.
</objective>
