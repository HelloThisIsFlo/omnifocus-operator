<objective>
Add the two task-only derived presence flags `is_sequential` and `depends_on_children` as service-layer computed fields, and lock the `ProjectType` assembly placement (HIER-05) at the service layer. This is the "domain logic" half of Wave 2 — projection/strip rules for these flags happen in 56-04.

Purpose: FLAG-04 and FLAG-05 define these as DERIVED flags, not storage columns. They must be computed somewhere. CONTEXT.md locks the `ProjectType` assembly at the service layer per HIER-05. This plan extends `DomainLogic` with a single enrichment method invoked by all three read pipelines (`get_all`, `get_task`, `list_tasks`), following the existing `compute_true_inheritance` pattern.

Output:
- `Task` model gains `is_sequential: bool` and `depends_on_children: bool` fields (both required — the enrichment step populates them).
- `DomainLogic.enrich_task_presence_flags(task: Task) -> Task` returns a copy with the two derived booleans set.
- `DomainLogic` also owns `ProjectType` re-derivation or validation, depending on the placement decision — see Task 2 below.
- Service read pipelines call the enrichment on every task they return.
- Projects do NOT get the task-only flags.

No strip rules, no projection changes, no tool description changes — those land in 56-04 and 56-05.
</objective>

Placement decision point — `Task.is_sequential` and `Task.depends_on_children`:
- Option A: Required on the model; repo returns placeholder values (e.g., False/False), domain overrides.
- Option B: Optional on the model (default=False); domain fills in.
- Option C: Required on the model; repo mapper computes them directly from `sequential` + `(hasChildren AND NOT completeWhenChildrenComplete)`, domain re-validates.

Choose **Option B**. Rationale: the fields are DERIVED — domain owns that computation. Repos MUST NOT implement product decisions. Option B keeps repos ignorant of the derivation rule while making the model ergonomic (defaults fall back to sensible values if the enrichment is somehow bypassed). Option A requires repos to know about a "placeholder" contract which is fragile.

If 56-02 added is_sequential / depends_on_children emission at the repo layer as an interim, remove it in Task 1 of this plan — the service layer owns this.
