<objective>
Close UAT-57 G4: non-default `availability` combined with scope (project/parent) genuinely narrows the result set by excluding lifecycle states (e.g. `['available']` excludes blocked tasks from the default `remaining` bucket), but `FILTERED_SUBTREE_WARNING` does NOT fire — agents get no pedagogical hint that they may be missing intermediate parents from the subtree.

Root cause (from UAT-57 G4 root_cause section): `availability` is a regular `list[AvailabilityFilter] = Field(default=[REMAINING])` on `ListTasksQuery`, NOT a `Patch[T]` field. It is therefore:
- Absent from `_PATCH_FIELDS`
- Absent from `_SUBTREE_PRUNING_FIELDS`
- Absent from `_NON_SUBTREE_PRUNING_FIELDS`
- Silently invisible to `check_filtered_subtree` which iterates `_SUBTREE_PRUNING_FIELDS` using `is_set()` (Patch-only semantic).

D-13's original "exclude availability" intent was implemented by omission rather than by explicit exclusion, which silences the non-default case along with the default case.

**Fix shape (agreed with Flo in UAT discussion, from G4 design section):**
1. Add `availability` to `_SUBTREE_PRUNING_FIELDS` — same classification axis as other pruning fields.
2. Introduce `is_non_default(model, field_name) -> bool` helper that dispatches on field type:
   - Patch[T] field → equivalent to `is_set(value)` (value is not UNSET).
   - Regular field with concrete default → `value != field_info.default` (read via Pydantic `model_fields`).
3. `check_filtered_subtree` iterates `_SUBTREE_PRUNING_FIELDS` using `is_non_default` instead of `is_set(getattr(...))`.
4. Scope check (`is_set(query.project) or is_set(query.parent)`) stays unchanged — both are Patch fields; `is_set` is sufficient and more specific there.
5. `TestSubtreePruningFieldsDrift` broadens: the three existing drift tests still enforce the classification contract, but the "every classified field must be in `_PATCH_FIELDS`" constraint relaxes — it now checks "every classified field must be a real field on `ListTasksQuery`" (Patch OR regular).

Purpose: Close WARN-01's coverage gap for the `availability` lifecycle-narrowing case. Restore D-13's "don't spam the warning on the default value" intent without silencing the non-default case along with it.

Output: Agents combining `project=X` with `availability=['available']` get FILTERED_SUBTREE_WARNING — the same guidance they get for `project=X + flagged=True`. Agents using only the default `availability=['remaining']` (explicitly or implicitly) still don't see the warning.

    **is_non_default helper — value-aware predicate:**

    Signature: `is_non_default(model: BaseModel, field_name: str) -> bool`.

    Dispatch rules:
    - If `getattr(model, field_name)` is an instance of `_Unset` → return False (Patch field defaulting to UNSET = "not set" = default).
    - If `getattr(model, field_name)` differs from the default stored at `type(model).model_fields[field_name].default` (via `!=` equality) → return True.
    - Otherwise return False.

    **Equality semantics for mutable defaults:** Pydantic v2 stores the declared default on `FieldInfo.default` even for list/dict. `Field(default=[AvailabilityFilter.REMAINING])` produces a FieldInfo whose `.default` attribute is the literal list `[AvailabilityFilter.REMAINING]`. Python's `list == list` compares element-wise, so `[REMAINING] != [REMAINING]` is False (they're equal). `[REMAINING] != [AVAILABLE]` is True. Both needed cases work via stdlib equality.

    **List equality is order-sensitive** — this is the stdlib `==` contract and is intentional. Document in the helper's docstring (see Step 5b). For the current `availability` default, which is a single-element list, ordering is moot. If a future default is a multi-element list, reordering it at the contract would be an API-breaking change (schema version bump), not a runtime concern — the drift test's CI gate would surface any classification/default mismatch.

    **Edge case — `Field(default_factory=list)`:** when a field uses `default_factory` instead of a literal `default`, Pydantic stores `FieldInfo.default_factory` and `.default` is the `PydanticUndefined` sentinel. For the MVP, `availability` uses a literal `default` (verified in line 80 read), so `is_non_default` can rely on `.default` being a real value. Add a clear fallback: if `FieldInfo.default` is `PydanticUndefined`, call the factory to get a fresh instance and compare. Document this in the docstring so future regular-field additions are handled.

    **Patch-field compatibility:** For `Patch[T]` fields, `FieldInfo.default == UNSET`. When the field holds a real value, `getattr(model, field_name) != UNSET` is True. When the field is UNSET, equality is True, so `is_non_default` returns False. Result is equivalent to `is_set` for Patch fields. Confirmed compatible — check with tests below.

    ```python
    def is_non_default(model: BaseModel, field_name: str) -> bool:
        """True if the named field has a value differing from its declared default.

        Value-aware predicate:
        - Patch[T] field → equivalent to is_set (value is not UNSET).
        - Regular field with concrete default → value != model_fields[field_name].default.
        - Regular field with default_factory → value != factory().

        Reads the default from Pydantic's model_fields. Handles mutable defaults
        correctly because Pydantic stores the literal default on FieldInfo.default
        for Field(default=[...]); for Field(default_factory=...) cases, we call
        the factory once to produce a fresh baseline instance for comparison.

        List equality via stdlib `==` is order-sensitive — intentional contract.
        Reordering a multi-element default at the contract level is an API-breaking
        change (schema version bump), not a runtime concern.
        """
        value = getattr(model, field_name)
        if isinstance(value, _Unset):
            return False
        field_info = type(model).model_fields[field_name]
        if field_info.default is PydanticUndefined:
            # default_factory path
            if field_info.default_factory is None:
                # Neither default nor default_factory — field is required and therefore set.
                return True
            baseline = field_info.default_factory()
        else:
            baseline = field_info.default
        return value != baseline
    ```
