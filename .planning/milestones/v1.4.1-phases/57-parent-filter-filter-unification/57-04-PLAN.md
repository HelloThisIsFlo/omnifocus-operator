2. **Repo WHERE clause (Option A):** Both hybrid SQL and bridge_only Python compose as `WHERE (t.id IN pinned_task_ids) OR (t.id IN candidate_task_ids AND <all other predicates>)`. When `pinned_task_ids` is None/empty, the shape collapses to the existing candidate-only behavior.
3. **Service `_resolve_parent`:** After resolution, filter the resolved IDs to task IDs only (projects are excluded — they don't inject anchors). Store on `self._parent_anchor_ids`.
4. **Service `_build_repo_query`:** `candidate_task_ids` = same intersection logic as today's `task_id_scope`. `pinned_task_ids` = `self._parent_anchor_ids` filtered through `self._project_scope` (when set). Sorted for deterministic placeholder order.
5. **Service no-match semantic flip:** `_resolve_scope_filter` returns `(set(), resolved)` on no-match (not None). `_resolve_tags` unconditionally assigns `self._tag_ids = all_resolved` (so empty list flows through).

    """G2/G3: detect queries whose resolved scope is empty-by-construction.

    - candidate_task_ids is [] (project/parent intersection or single scope resolved empty) AND
      pinned_task_ids is None/empty (no anchors to preserve).
    - tag_ids is [] (tags resolved to zero matches).

        return True
